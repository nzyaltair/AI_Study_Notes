# 向量数据库

## 1. 概述

向量数据库（Vector Database）是专为存储、索引和检索高维向量数据而设计的数据库系统。在 `RAG` 系统中，向量数据库扮演"外部知识库"的核心角色，负责文档向量表示的存储和相似度检索。

随着ANN（近似最近邻）搜索算法的成熟和LLM应用的普及，向量数据库已成为AI基础设施的关键组件。它的核心价值在于：将"语义匹配"问题转化为"高维向量相似度计算"问题，支持大规模语义检索。

## 3. 核心概念

### 相似度度量

| 度量 | 公式 | 适用场景 |
|------|------|---------|
| 余弦相似度 | $\cos(\theta) = \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \|\mathbf{B}\|}$ | 文本嵌入，方向相似性 |
| L2距离（欧氏） | $d = \sqrt{\sum_i (A_i - B_i)^2}$ | 图像嵌入，绝对距离 |
| 内积（点积） | $\mathbf{A} \cdot \mathbf{B} = \sum_i A_i B_i$ | 已归一化向量的快速计算 |
| 曼哈顿距离 | $d = \sum_i |A_i - B_i|$ | 高维数据，抗异常值 |

### 关键术语

- **Embedding**：将非结构化数据（文本、图像）映射为高维向量
- **ANN（Approximate Nearest Neighbor）**：近似最近邻搜索，牺牲少量精度换取大幅速度提升
- **Recall@K**：前K个结果中包含真实最近邻的比例
- **QPS（Queries Per Second）**：每秒查询数，衡量检索吞吐量
- **HNSW**：分层可导航小世界图，主流ANN索引算法
- **IVF**：倒排文件索引，基于聚类的检索方法
- **PQ**：乘积量化，向量压缩技术

## 4. 技术原理

### ANN索引算法

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| HNSW | 分层可导航小世界图 | 高召回率，查询速度快 | 内存占用高 | 高精度，中小规模 |
| IVF | 倒排文件 + 聚类 | 内存效率高 | 需要训练数据 | 大规模数据 |
| PQ | 乘积量化压缩 | 内存占用极低 | 精度损失 | 超大规模，内存受限 |
| IVF+PQ | 聚类 + 量化 | 内存与精度平衡 | 实现复杂 | 大规模生产环境 |
| LSH | 局部敏感哈希 | 速度快，可扩展 | 精度较低 | 近似去重 |
| ScaNN | 各向异性量化 | 速度与精度平衡 | Google生态 | 高性能需求 |
| Flat | 暴力扫描 | 100%召回率 | 速度慢 | 小规模，基准测试 |

### HNSW算法详解

HNSW（Hierarchical Navigable Small World）是目前最常用的ANN索引算法：

**核心思想**：构建多层图结构，上层稀疏（长距离连接），下层密集（短距离连接）。查询时从最上层开始快速定位到目标区域，逐层下降进行精细搜索。

**核心参数**：

| 参数 | 含义 | 典型值 | 影响 |
|------|------|--------|------|
| $M$ | 每个节点最大连接数 | 16-48 | 越大→精度↑、内存↑ |
| $efConstruction$ | 建索引时候选列表大小 | 200-800 | 越大→建索引质量↑、速度↓ |
| $efSearch$ | 查询时候选列表大小 | 100-400 | 越大→召回率↑、查询速度↓ |

### IVF算法

**核心思想**：使用K-Means将向量空间划分为$nlist$个聚类（倒排桶），查询时只搜索最近的$nprobe$个桶。

| 参数 | 含义 | 典型值 |
|------|------|--------|
| $nlist$ | 聚类中心数 | $\sqrt{N}$ ~ $4\sqrt{N}$（N为数据量） |
| $nprobe$ | 查询时搜索的桶数 | 1-10% of $nlist$ |

### 乘积量化（PQ）

**核心思想**：将高维向量切分为多个子向量，每个子向量独立量化为编码，大幅压缩存储空间。

```
1024维向量 → 切分为8个128维子向量 → 每个子向量量化为8bit编码
存储：1024 × 4字节 = 4096字节 → 8 × 1字节 = 8字节（压缩512倍）
```

### 元数据过滤

在向量检索基础上叠加结构化过滤条件，实现混合查询：

```python
# 预过滤：先按元数据过滤，再向量检索
filtered_docs = [d for d in docs if d.metadata["category"] == "tech"]
results = vector_index.search(query_vector, candidates=filtered_docs, k=5)

# 后过滤：先向量检索获取候选，再按元数据过滤
candidates = vector_index.search(query_vector, k=top_k * 5)
filtered = [r for r in candidates if r.metadata["category"] == "tech"]
```

### 分布式架构

- **分片（Sharding）**：按哈希取模将向量分散到多个节点
- **副本（Replication）**：多副本保证高可用
- **一致性级别**：quorum / strong / eventual

```
数据量 < 100万 → Chroma / FAISS
数据量 100万-1亿 → Qdrant / Weaviate
数据量 > 1亿 → Milvus (分布式) / Pinecone
已有PostgreSQL → pgvector
已有Elasticsearch → ES向量检索
需要全托管 → Pinecone
```

## 6. 优势与局限

### 优势

- 语义检索能力强，支持"意思相近"的匹配
- 查询速度快，ANN算法可在毫秒级完成百万级向量检索
- 可扩展性好，支持分布式部署
- 多模态支持，可存储文本、图像、音频的向量

### 局限

- **内存消耗大**：HNSW等图索引需要将向量载入内存
- **精度与速度权衡**：ANN是近似算法，无法保证100%召回
- **更新成本**：大规模索引的增量更新可能影响性能
- **缺乏精确匹配**：向量检索可能遗漏精确关键词匹配的文档
- **维度诅咒**：极高维度下，向量相似度的区分度下降

## 7. 应用场景

- `RAG` 系统的知识库存储
- 语义搜索引擎
- 推荐系统的相似度匹配
- 图像/视频检索
- 去重与聚类
- `Agent记忆` 系统的长期记忆存储

## 8. 与其他技术关系

- 向量数据库是 `RAG` 系统的存储基础
- 嵌入模型为向量数据库提供数据
- `多模态RAG` 需要多模态向量存储
- `Agent记忆` 系统可基于向量数据库实现
- HNSW/IVF等 `ANN算法` 是向量检索的核心
- `混合检索` 需要向量数据库支持向量+关键词联合查询

## 9. 前沿发展

- **混合检索原生支持**：向量数据库内置BM25等关键词检索，无需外部组件
- **学习索引（Learned Indexes）**：用机器学习替代传统索引结构，自适应数据分布
- **多模态向量存储**：原生支持图像、视频等多模态向量的存储与检索
- **GPU加速索引**：利用GPU加速向量索引构建和查询
- **Serverless向量数据库**：按需弹性扩展，降低使用成本
- **向量数据库 + LLM原生集成**：向量检索成为LLM应用框架的标准组件
- **隐私向量检索**：支持加密向量检索，保护数据隐私
- **增量索引优化**：支持高效的实时向量插入与删除，无需重建索引
