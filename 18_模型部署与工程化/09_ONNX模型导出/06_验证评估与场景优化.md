# 验证评估与场景优化

## 背景与发展

[[ONNX]] 模型转换后必须经过严格验证，确保与原始模型输出一致。验证涵盖三个层面：正确性（数值一致性）、精度（量化后指标保持）、性能（延迟和吞吐量）。验证通过后，还需根据部署场景（云端、边缘、大规模）进行针对性优化。

## 核心思想

验证的核心是**逐层对比原始框架输出与 ONNX Runtime 输出的数值差异**，使用相对容差（rtol）和绝对容差（atol）判断是否在可接受范围内。性能优化遵循分层策略：先通过 ONNX Runtime 图优化获得基础加速，再通过 TensorRT FP16/INT8 获得极致性能，最后通过量化感知训练（QAT）恢复精度。

## 技术原理

### 正确性验证

```python
import numpy as np
import onnxruntime as ort

def validate_outputs(ref_output, onnx_output, rtol=1e-3, atol=1e-5):
    """验证 ONNX 输出与参考输出的一致性"""
    abs_diff = np.abs(np.asarray(ref_output) - np.asarray(onnx_output))
    max_abs_diff = np.max(abs_diff)
    is_valid = np.allclose(ref_output, onnx_output, rtol=rtol, atol=atol)
    return is_valid, max_abs_diff

# 对比 PyTorch 与 ONNX Runtime 输出
model.eval()
with torch.no_grad():
    torch_output = model(dummy_input)

sess = ort.InferenceSession("model.onnx")
onnx_output = sess.run(None, {"input": dummy_input.numpy()})[0]

is_valid, max_diff = validate_outputs(torch_output.numpy(), onnx_output)
print(f"最大误差: {max_diff:.2e}, 验证{'通过' if is_valid else '失败'}")
```

**容差阈值建议**：

| 数据类型 | rtol | atol | 说明 |
|---------|------|------|------|
| FP32 | 1e-3 | 1e-5 | 标准配置 |
| FP16 | 1e-2 | 1e-3 | 适当放宽 |
| INT8 量化 | 1e-1 | 1e-2 | 显著精度损失可接受 |

### 精度验证与调优

量化后精度下降的常见原因：算子实现差异累积、量化范围不准确、常量折叠引入偏差。调优优先级：

1. **使用真实校准数据**（预期收益 +2-5%）
2. **增加校准样本数至 500-1000**（+1-3%）
3. **尝试不同校准方法**：MinMax / Entropy / Percentile（+0.5-2%）
4. **量化感知训练（QAT）**（+3-8%）
5. **调整 Opset 版本**（+0.5-1%）

```python
from onnxruntime.quantization import quantize_static, QuantFormat, QuantType

class CalibrationDataReader:
    def __init__(self, data, input_name):
        self.data = data
        self.input_name = input_name
        self.idx = 0
    def get_next(self):
        if self.idx >= len(self.data):
            return None
        batch = {self.input_name: self.data[self.idx:self.idx+1]}
        self.idx += 1
        return batch

quantize_static(

    quant_format=QuantFormat.QDQ,
    activation_type=QuantType.QUInt8,
    weight_type=QuantType.QInt8,
    per_channel=True
)
```

### 性能测试

延迟测量须遵循：预热 10-20 次消除初始化开销、正式测试至少 100 次迭代、记录 P50/P95/P99 分位数。

```python
import time, numpy as np

def benchmark(model_path, batch_size=1, num_iters=100, warmup=10):
    sess = ort.InferenceSession(model_path)
    input_name = sess.get_inputs()[0].name
    dummy = np.random.randn(batch_size, 3, 224, 224).astype(np.float32)

    for _ in range(warmup):
        sess.run(None, {input_name: dummy})

    latencies = []
    for _ in range(num_iters):
        start = time.perf_counter()
        sess.run(None, {input_name: dummy})
        latencies.append((time.perf_counter() - start) * 1000)

    print(f"Mean: {np.mean(latencies):.2f}ms, P95: {np.percentile(latencies, 95):.2f}ms")
    print(f"Throughput: {batch_size / (np.mean(latencies)/1000):.1f} samples/s")
```

**优化路径选择矩阵**：

| 优化层级 | 加速比 | 精度损失 | 适用场景 |
|---------|--------|---------|---------|
| ONNX Runtime | 1.1-1.3x | <0.5% | 多平台部署 |
| TensorRT FP16 | 1.4-1.8x | <1% | NVIDIA GPU |
| TensorRT INT8 | 2.0-3.5x | 1-2% | 高吞吐场景 |
| 模型剪枝+量化 | 3-5x | 2-5% | 边缘设备 |

### 云端推理优化

云端部署核心是高并发、低延迟和成本控制：

- **REST API**：FastAPI + ONNX Runtime，适合快速原型
- **gRPC**：高性能二进制协议，适合微服务间通信
- **动态批处理**：在超时窗口内合并多个请求为 batch，平衡吞吐和延迟
- **Kubernetes HPA**：基于 CPU/GPU 利用率自动扩缩容

```python
# 动态批处理核心逻辑
class BatchingInference:
    def __init__(self, session, max_batch=32, timeout_ms=50):
        self.session = session
        self.max_batch = max_batch
        self.timeout = timeout_ms / 1000

    async def process_batch(self, requests):
        batch = np.stack([req['input'] for req in requests])
        outputs = self.session.run(None, {'input': batch})
        return outputs[0]  # 分散结果给各请求
```

### 边缘设备优化

边缘设备优化优先级：INT8 量化 > 模型剪枝 > 知识蒸馏。

| 优化方法 | 模型大小减少 | 精度影响 | 推理加速 |
|---------|-------------|---------|---------|
| FP32 → FP16 | 50% | <0.5% | 1.5-2x |
| INT8 动态量化 | 75% | 1-3% | 2-3x |
| INT8 静态量化 | 75% | 2-5% | 3-4x |
| 通道剪枝 30% | 30% | 1-2% | 1.3-1.8x |

**Android 部署**：通过 NNAPI 加速，iOS 通过 CoreML EP 利用 Apple Neural Engine。

### 硬件平台适配

ONNX Runtime 通过 Execution Provider 机制适配多硬件：

| 硬件平台 | 推荐 EP | 关键配置 |
|---------|--------|---------|
| NVIDIA GPU | TensorRT > CUDA | `trt_fp16_enable=True` |
| Intel CPU | OpenVINO / CPU | `OMP_NUM_THREADS=核心数` |
| 华为 NPU | Ascend | 可能需 ATC 工具转 OM 格式 |
| Apple Silicon | CoreML | 自动利用 ANE |
| ARM 边缘 | CPU | NEON 自动优化 |

### 大规模部署实践

- **模型版本管理**：使用 MLflow 或 DVC 管理模型生命周期
- **金丝雀发布**：10% 流量验证新模型，逐步扩大到 100%
- **监控指标**：P50/P95/P99 延迟、QPS、错误率、GPU 内存、数据漂移
- **CI/CD 流水线**：模型验证 → 量化 → Docker 构建 → Staging 部署 → 金丝雀 → 全量

## 发展演进

ONNX 模型验证从简单的数值对比发展为系统化测试框架，pytest-benchmark 等工具支持性能回归检测。云端部署从单机 REST API 演进为 Kubernetes 原生的微服务架构，配合 Istio/Argo Rollouts 实现精细化的流量管理。边缘部署随着 ONNX Runtime Mobile 和 NNAPI/CoreML EP 的成熟，移动端推理性能持续提升。

## 关键算法·模型

- **np.allclose 验证**：同时检查相对误差和绝对误差，避免浮点精度问题导致的误判
- **EntropyCalib2 校准**：通过 KL 散度最小化量化前后分布差异，是 TensorRT INT8 的默认校准方法
- **动态批处理调度**：在固定时间窗口内收集请求，合并为 batch 后统一推理，提高 GPU 利用率
- **知识蒸馏**：用大模型（teacher）指导小模型（student）训练，实现 90% 参数减少且精度损失 <2%

## 应用场景

| 场景 | 验证重点 | 优化策略 |
|------|---------|---------|
| 云端 GPU 服务 | 延迟 P99 < 100ms | TensorRT FP16 + 动态批处理 |
| 边缘设备 | 内存 < 500MB | INT8 量化 + 通道剪枝 |
| 大规模 LLM | 首字延迟 < 500ms | 模型拆分 + KV Cache 量化 |

| 医疗诊断 | 精度损失 < 0.1% | 保持 FP32，仅做图优化 |
