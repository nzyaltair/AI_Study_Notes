# 问答与对话系统

## 1. 概述

问答系统（Question Answering, QA）和对话系统（Dialogue System）是 NLP 最具应用价值的任务——让计算机能够回答人类问题并进行自然语言交互。问答系统专注于对特定查询给出准确答案，对话系统则处理多轮交互，维持上下文和对话状态。

从 1960 年代的 ELIZA 到 2022 年的 ChatGPT，问答与对话系统经历了从"模式匹配"到"检索式"再到"生成式"的演进。大语言模型的出现使对话系统实现了质的飞跃——从"任务型"和"检索型"走向"通用开放域对话"。

## 3. 核心概念

### 3.1 问答系统分类

| 类型 | 知识来源 | 方法 | 示例 |
|:---|:---|:---|:---|
| 知识库问答（KBQA） | 结构化知识库 | SPARQL/查询转换 | "姚明身高？" → 查询 KG |
| 机器阅读理解（MRC） | 给定文档 | 抽取/生成答案 | SQuAD 任务 |
| 开放域问答（OpenQA） | 全网/大语料 | 检索+阅读理解 | "谁发明了电话？" |
| 表格问答（TableQA） | 结构化表格 | NL2SQL | "销售额最高的产品？" |
| 视觉问答（VQA） | 图像+文本 | 多模态理解 | 图片中有什么？ |

### 3.2 对话系统分类

| 类型 | 目标 | 特点 | 示例 |
|:---|:---|:---|:---|
| 任务型对话 | 完成特定任务 | 槽位填充、对话状态追踪 | 订餐、订票 |
| 问答型对话 | 回答用户问题 | 检索/生成答案 | 客服、智能助手 |
| 闲聊对话 | 社交互动 | 开放域生成 | 聊天机器人 |
| 通用对话 | 多功能 | 综合能力 | ChatGPT |

### 3.3 任务型对话架构

传统任务型对话系统采用模块化架构：

```
用户输入 → ASR → NLU（意图识别+槽位抽取）→ DM（对话管理）→ NLG → TTS → 输出
                         ↑                       ↑
                    对话状态追踪           对话策略
```

| 模块 | 功能 | 方法 |
|:---|:---|:---|
| NLU | 理解用户意图和槽位 | 意图分类 + 槽位标注 |
| DST | 追踪对话状态（已填槽位） | 规则/神经网络 |
| POLICY | 决策下一步动作 | 规则/RL |
| NLG | 生成自然语言回复 | 模板/Seq2Seq |

## 4. 技术原理

### 4.1 机器阅读理解（MRC）

给定问题和文档，从文档中抽取答案：

**抽取式 MRC**：

$$P(\text{start}=i) = \text{softmax}(W_s \cdot h_i)$$

$$P(\text{end}=j) = \text{softmax}(W_e \cdot h_j)$$

BERT 在 `[CLS] 问题 [SEP] 文档 [SEP]` 上预测答案的起止位置。

**SQuAD 基准**：给定 Wikipedia 段落和问题，从段落中抽取答案 span。BERT 在 SQuAD 2.0 上 F1 超过人类水平。

### 4.2 开放域问答（OpenQA）

**Retrieve-and-Read 框架**：

```
问题 → 检索器（BM25/DPR）→ Top-K 文档 → 阅读器（BERT）→ 答案
```

- 检索器从大规模语料中召回相关文档
- 阅读器从检索到的文档中抽取或生成答案

### 4.3 知识库问答（KBQA）

将自然语言问题转换为知识库查询：

```
"姚明的妻子是谁？" → SPARQL: SELECT ?x WHERE { :姚明 :配偶 ?x }
→ 答案：叶莉
```

方法：
- **语义解析**：将问题解析为逻辑表达式/查询语言
- **基于嵌入的方法**：在知识图谱嵌入空间中匹配答案

### 4.4 生成式对话

#### Seq2Seq 对话

将对话建模为序列到序列生成：

$$P(\text{response} | \text{context}) = \prod_t P(r_t | r_{<t}, \text{context})$$

**问题**：生成"安全回复"（"我不知道"、"好的"）——缺乏多样性和信息量。

#### 检索式对话

从预构建的对话库中检索最合适的回复：

1. 对话上下文编码为向量
2. 在对话库中检索最相似的上下文-回复对
3. 返回对应回复

### 4.5 LLM 对话

大语言模型通过指令微调（SFT）+ RLHF 实现高质量对话：

- **指令微调**：教会模型遵循指令进行对话
- **RLHF**：用人类偏好优化对话质量
- **上下文窗口**：支持长对话历史

ChatGPT 的出现标志着对话系统从"专门构建"走向"通用大模型"。

## 5. 关键方法/模型

| 方法 | 类型 | 优势 | 局限 |
|:---|:---|:---|:---|
| 规则/模板对话 | 任务型 | 精确可控 | 不灵活、开发成本高 |
| 模块化对话 | 任务型 | 可解释 | 级联误差 |
| Seq2Seq 对话 | 闲聊 | 端到端 | 通用回复、无个性 |
| 检索式对话 | 闲聊 | 回复质量有保障 | 受限于对话库 |
| BERT MRC | QA | 精确抽取 | 仅限给定文档 |
| RAG QA | 开放域 | 结合检索+生成 | 检索质量影响答案 |
| LLM 对话 | 通用 | 高质量、多能力 | 幻觉、成本高 |

## 6. 优势与局限

### 优势

1. **自然交互**：最自然的人机交互方式
2. **信息获取**：问答系统高效获取精确信息
3. **任务自动化**：对话系统驱动任务完成
4. **个性化**：可根据用户画像定制对话

### 局限

1. **幻觉**：LLM 对话可能生成看似合理但错误的信息
2. **多轮一致性**：长对话中模型可能自相矛盾
3. **领域知识**：通用模型在专业领域可能缺乏深度
4. **安全与伦理**：对话内容的安全过滤和对齐
5. **评估困难**：对话质量评估高度主观

## 7. 应用场景

- **智能助手**：Siri、Alexa、Google Assistant
- **客服机器人**：自动回答客户问题
- **教育**：智能辅导、答疑系统
- **医疗**：症状问诊、健康咨询
- **企业内部**：知识库问答、HR 助手
- **编程助手**：Copilot、代码问答
- **通用对话**：ChatGPT、Claude、Gemini

## 9. 前沿发展

- **LLM Agent**：对话系统从"回答问题"走向"执行任务"——LLM 作为 Agent 自主调用工具、规划行动
- **RAG 增强**：检索增强生成减少幻觉，提供最新信息
- **多模态对话**：支持图像、音频、视频的对话交互
- **个性化对话**：长期记忆和用户画像驱动的个性化对话
- **对话安全**：对齐技术确保对话内容安全、无偏见
- **实时语音对话**：GPT-4o 等实现端到端实时语音对话
- **多 Agent 协作**：多个 LLM Agent 组成协作团队解决复杂任务
