# 模型部署与工程化

## 背景与发展

[[大语言模型]]从实验走向生产，需要经历训练完成后的部署、服务化、监控、版本管理、成本控制与合规治理等环节。早期模型部署以 Flask/Django 手动封装为主，缺乏标准化流程。随着模型规模从亿级增长到千亿级，推理引擎、容器编排、API 网关等工程化设施成为保障服务延迟、吞吐与可靠性的关键。MLOps 理念将 DevOps 实践扩展到机器学习系统，形成覆盖模型注册、灰度发布、自动扩缩容、全链路监控的完整体系。

## 核心思想

模型部署与工程化的核心是将模型从离线实验环境迁移到在线生产环境，并在该过程中解决三组矛盾：**延迟与吞吐**的矛盾（批处理提高吞吐但增加单请求延迟）、**成本与质量**的矛盾（大模型精度高但推理成本高）、**效率与安全**的矛盾（快速迭代需求与合规审计要求的张力）。工程化通过标准化容器镜像、自动化 CI/CD 流水线、多级监控告警和配额治理来系统性地管理这些矛盾。

## 技术原理

部署工程化的分层架构包括：

- **网关层**：认证授权、限流熔断、请求路由、API 版本管理
- **服务层**：推理引擎调度、动态批处理、流式响应（SSE/gRPC stream）
- **推理层**：[[模型推理]]引擎（vLLM、TensorRT-LLM、TGI）、[[KV-Cache]]管理、量化计算
- **基础设施层**：GPU 集群、[[Docker]]容器、Kubernetes 编排、存储与网络

关键性能指标包括首 Token 延迟（TTFT）、每 Token 生成延迟（TPOT）、吞吐量（tokens/s）、GPU 利用率和显存占用率。

## 发展演进

手动部署（Flask/FastAPI） → 通用推理服务（TF Serving/TorchServe） → LLM 专用推理引擎（vLLM/TGI/TensorRT-LLM） → 容器化与 K8s 编排 → Serverless LLM → 边缘 LLM 部署

## 关键算法·模型

- **PagedAttention**（vLLM）：将 KV-Cache 分页管理，减少显存碎片，提高并发吞吐
- **连续批处理**（Continuous Batching）：请求完成即释放资源，无需等待整批完成
- **推测解码**（Speculative Decoding）：小模型生成候选 Token，大模型并行验证，减少解码步数
- **模型量化**：INT8/INT4 量化（GPTQ、AWQ）减少显存占用和计算成本

## 应用场景

- **LLM API 服务**：OpenAI 兼容接口的公共或私有 API 平台
- **企业私有部署**：金融、医疗等数据敏感行业在本地部署模型
- **Agent 平台**：[[大语言模型]]作为 Agent 后端推理服务
- **多模型路由**：按任务复杂度将请求路由到不同规模模型，优化成本
