# 模型部署与服务化

## 背景

将训练好的[[大语言模型|LLM]]部署为可用的服务，是AI落地的关键环节。模型部署涉及推理引擎选择、API设计、负载均衡、高可用架构等多个方面，直接影响服务的延迟、吞吐和成本。

## 核心思想

选择合适的推理引擎，设计可扩展的服务架构，通过API网关统一管理请求，实现高可用、低成本、低延迟的模型服务。

## 技术原理

### 推理引擎选择

| 引擎 | 特点 | 适用场景 |
|------|------|---------|
| **vLLM** | PagedAttention，高吞吐 | 通用LLM服务 |
| **TGI**（HuggingFace） | Rust核心，连续批处理 | HF生态 |
| **TensorRT-LLM** | NVIDIA最优性能 | 生产级低延迟 |
| **llama.cpp** | CPU/GPU混合，轻量 | 本地/边缘 |
| **SGLang** | RadixAttention，结构化生成 | Agent应用 |
| **Ollama** | 本地一键部署 | 个人/开发 |

### 部署方式

| 方式 | 特点 | 适用场景 |
|------|------|---------|
| **云端部署** | 弹性扩缩容，按需付费 | 生产环境 |
| **本地部署** | 数据不出域，低延迟 | 企业/敏感数据 |
| **边缘部署** | 端侧推理，离线可用 | 移动端/IoT |
| **混合部署** | 云端+本地结合 | 平衡成本与隐私 |

### API设计

```python
# RESTful API 示例
POST /v1/chat/completions
{
    "model": "llama-3-70b",
    "messages": [{"role": "user", "content": "Hello"}],
    "temperature": 0.7,
    "max_tokens": 1024,
    "stream": true  # 流式响应
}
```

**API治理**：
- 认证授权（API Key / OAuth）
- 限流熔断（Rate Limiting / Circuit Breaker）
- 版本管理（/v1/, /v2/）
- 请求验证与参数校验
- API文档（OpenAPI/Swagger）

### 负载均衡与高可用

- **负载均衡**：Round-Robin / Least-Connections / 基于GPU负载
- **高可用**：多副本部署 + 健康检查 + 自动故障转移
- **弹性伸缩**：基于QPS/GPU利用率自动扩缩容
- **请求路由**：按模型大小/任务类型路由到不同实例

### 流式响应

Server-Sent Events（SSE）实现流式生成：

```
data: {"token": "Hello"}
data: {"token": " world"}
data: {"token": "!"}
data: [DONE]
```

### 灰度发布与A/B测试

- **蓝绿部署**：新旧版本并行，流量切换
- **金丝雀发布**：小流量验证新版本
- **A/B测试**：对比不同模型/参数效果
- **快速回滚**：版本异常时秒级回退

## 发展演进

Flask/FastAPI手动部署 → TF Serving/TorchServe → vLLM/TGI专用推理引擎 → Serverless LLM → 边缘LLM部署

## 应用领域

- **LLM API服务**：OpenAI兼容API
- **企业私有化**：数据安全要求高的场景
- **[[Agent智能体|Agent]]平台**：Agent后端推理服务
- **多模型路由**：根据请求路由到不同大小模型

- [[容器化与Docker|容器化]]是部署的标准方式
- [[监控与运维|监控]]确保服务稳定运行
- [[ONNX模型导出|ONNX]]实现跨框架部署
- [[机器学习流水线|MLOps]]提供部署自动化框架
