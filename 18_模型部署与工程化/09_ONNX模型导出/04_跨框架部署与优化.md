# 跨框架部署与优化

## 1. 概述

将 ONNX 模型部署到生产环境时，需要在多种推理引擎间做出选择。ONNX Runtime 提供跨平台通用推理能力，TensorRT 在 NVIDIA GPU 上提供极致性能，OpenVINO 面向 Intel 硬件优化。

跨框架部署解决的核心问题：
- **推理引擎选型**：在性能、兼容性和维护成本间取得平衡
- **硬件适配**：一次导出，多平台部署
- **性能优化**：通过算子融合、精度校准、Kernel 调优获得极致性能

ONNX 作为中间格式，解耦了训练框架和推理引擎：一次导出，多处部署。推理引擎通过 Execution Provider（EP）机制适配不同硬件后端。

## 3. 核心概念

### 3.1 Execution Provider（EP）

ONNX Runtime 通过 EP 机制适配不同硬件后端。选择推理引擎时遵循"最佳性能优先"原则——按 EP 列表顺序尝试，首个支持的 EP 被使用。

### 3.2 精度模式

| 模式 | 内存占用 | 推理速度 | 精度损失 |
|:---|:---|:---|:---|
| FP32 | 1.0x | 1.0x | 0% |
| FP16 | ~0.6x | 1.5-2x | <1% |
| INT8 | ~0.5x | 2-4x | 1-2% |

### 3.3 算子兼容性

不同推理引擎的算子支持范围各异，需检查目标引擎是否支持模型中使用的所有 ONNX 算子。

## 4. 技术原理

### 4.1 ONNX → TensorRT 优化

TensorRT 加载 ONNX 模型后执行多级优化：

1. **层融合**：将 Conv+BatchNorm+ReLU 等连续算子融合为单个 kernel
2. **精度校准**：FP16 无需校准数据，INT8 需代表性数据集
3. **Kernel 自动调优**：针对目标 GPU 架构选择最优 kernel
4. **动态形状优化**：通过 min/opt/max shapes 预编译多个 kernel

```bash
# trtexec 命令行转换
trtexec --onnx=model.onnx --saveEngine=model.engine \
    --fp16 --workspace=4096 \
    --minShapes=input:1x3x224x224 \
    --optShapes=input:8x3x224x224 \
    --maxShapes=input:32x3x224x224
```

### 4.2 ONNX Runtime 配置

```python
import onnxruntime as ort

sess_options = ort.SessionOptions()
sess_options.graph_optimization_level = ort.GraphOptimizationLevel.ORT_ENABLE_ALL

providers = [
    ('CUDAExecutionProvider', {
        'device_id': 0,
        'cudnn_conv_algo_search': 'DEFAULT',
        'gpu_mem_limit': 2 * 1024 * 1024 * 1024,
    }),
    'CPUExecutionProvider'  # Fallback
]

session = ort.InferenceSession("model.onnx", sess_options=sess_options, providers=providers)
```

### 4.3 算子兼容性处理

按优先级排列的解决方案：

1. **提升 Opset 版本**：新 Opset 可能已支持目标算子
2. **onnxruntime-extensions**：加载扩展算子库
3. **算子替换**：用标准 ONNX 算子组合替代不兼容算子
4. **GraphSurgeon 子图替换**：图级别替换不兼容节点
5. **自定义算子注册**：实现 C++ kernel 并注册

```python
from onnxruntime_extensions import get_library_path
sess_options = ort.SessionOptions()
sess_options.register_custom_ops_library(get_library_path())
session = ort.InferenceSession("model.onnx", sess_options)
```

### 4.4 Conv-BN-ReLU 融合

将三层算子合并为单个卷积 kernel，BatchNorm 参数折叠到 Conv 权重中：

$$W_{fused} = W_{conv} \cdot \frac{\gamma}{\sqrt{\sigma^2 + \epsilon}}$$

$$b_{fused} = \frac{b_{conv} - \mu}{\sqrt{\sigma^2 + \epsilon}} \cdot \gamma + \beta$$

### 4.5 QKV 矩阵融合

将 Attention 中三个独立的 Q/K/V 投影矩阵合并为一次 MatMul，减少 kernel 启动开销。

## 5. 关键方法与模型

### 5.1 性能对比（ResNet50, V100 GPU）

| 运行时 | Batch=1 | Batch=32 | 加速比 |
|:---|:---|:---|:---|
| PyTorch 原生 | 2.1 ms | 15.2 ms | 1.0x |
| ONNX Runtime (CUDA) | 1.8 ms | 11.2 ms | 1.2-1.4x |
| TensorRT FP16 | 1.2 ms | 6.8 ms | 1.8-2.2x |
| TensorRT INT8 | 0.9 ms | 5.2 ms | 2.3-2.9x |

### 5.2 Timing Cache

TensorRT 首次构建时缓存 kernel 计时数据，后续构建复用可加速 50%+。

### 5.3 EntropyCalib2

INT8 量化校准算法，通过最大化输出分布熵（KL 散度最小化）来确定激活值量化范围。

### 5.4 IOBinding 零拷贝

ONNX Runtime 的 IOBinding 机制，避免 GPU↔CPU 间的数据拷贝，减少通信开销。

## 6. 优势与局限

### 6.1 优势

- **极致性能**：TensorRT 在 NVIDIA GPU 上提供最优推理性能
- **跨平台**：ONNX Runtime 一套代码覆盖多硬件
- **灵活选型**：按性能需求选择不同推理引擎

### 6.2 局限

- **引擎锁定**：TensorRT 优化后的 engine 不可跨 GPU 架构
- **动态形状开销**：TensorRT 动态形状优化需预编译多个 profile
- **算子兼容性**：不同引擎的算子支持范围不同
- **调试复杂**：优化后的模型难以调试

## 7. 应用场景

| 场景 | 推荐引擎 | 优化策略 |
|:---|:---|:---|
| NVIDIA GPU 生产部署 | TensorRT | FP16/INT8 + 层融合 |
| 多平台通用部署 | ONNX Runtime | CUDA/CPU EP |
| Intel 硬件优化 | OpenVINO | IR 转换 + 量化 |
| 移动端部署 | ONNX Runtime Mobile | INT8 + NNAPI/CoreML |
| Apple Silicon | ONNX Runtime + CoreML EP | ANE 神经引擎 |
