# 模型服务化

## 1. 概述

### 1.1 目标定位
本文档聚焦大模型从训练完成到生产环境高效、稳定、可扩展服务的全过程，强调工程落地能力，为AI工程师、MLOps从业者和系统架构师提供实践指导。

### 1.2 核心挑战
- **性能**：低延迟、高吞吐量、高效显存利用
- **稳定性**：避免OOM、服务降级、故障恢复
- **可扩展性**：支持自动扩缩容、多副本部署
- **可观测性**：全面监控、快速排障
- **安全性**：鉴权、限流、输入过滤

## 2. 服务架构模式

### 2.1 API设计模式

#### 2.1.1 REST API
- **适用场景**：简单部署、跨语言调用、易于调试
- **技术栈**：FastAPI + Transformers、Flask + LangChain
- **示例配置**：
```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI(title="大模型推理服务")

class InferenceRequest(BaseModel):
    prompt: str
    max_new_tokens: int = 256
    temperature: float = 0.7

@app.post("/v1/completions")
def completions(request: InferenceRequest):
    # 推理逻辑
    return {
        "id": "cmpl-123",
        "object": "text_completion",
        "created": 1706000000,
        "model": "llama-3-8b",
        "choices": [{"text": "推理结果", "index": 0}]
    }
```

#### 2.1.2 gRPC API
- **适用场景**：高并发、低延迟、内部服务调用
- **优势**：二进制传输、强类型定义、连接复用
- **示例配置**：
```proto
service LLMService {
  rpc Generate(GenerateRequest) returns (GenerateResponse);
  rpc StreamGenerate(GenerateRequest) returns (stream GenerateResponse);
}

message GenerateRequest {
  string prompt = 1;
  int32 max_new_tokens = 2;
  float temperature = 3;
}

message GenerateResponse {
  string text = 1;
  int32 index = 2;
}
```

#### 2.1.3 异步推理与流式响应
- **异步推理**：适用于长文本生成、批量请求
- **流式响应**：提升用户体验，支持边生成边返回
- **vLLM流式示例**：
```bash
python -m vllm.entrypoints.api_server \
    --model meta-llama/Llama-3-8b-hf \
    --served-model-name llama-3-8b \
    --host 0.0.0.0 \
    --port 8000 \
    --enable-streaming
```

## 3. 推理引擎选型

### 3.1 主流推理引擎对比

| 引擎 | 优势 | 劣势 | 适用场景 | 2025年实测性能（Llama-3-8B） |
|------|------|------|----------|----------------------------|
| [[vLLM]] | 连续批处理、高吞吐、易部署 | 内存占用较高 | 高并发在线服务 | 吞吐量：1000+ tok/s，延迟：50ms |
| [[TGI]] | HuggingFace生态集成、多模态支持 | 配置复杂 | 生产级部署、多模型管理 | 吞吐量：800+ tok/s，延迟：60ms |
| [[TensorRT-LLM]] | 极致性能、CUDA内核优化 | 编译时间长、灵活性低 | 固定模型的高性能部署 | 吞吐量：1200+ tok/s，延迟：40ms |
| [[ONNX Runtime]] | 跨平台、框架无关 | 模型转换复杂 | 边缘部署、跨硬件支持 | 吞吐量：600+ tok/s，延迟：80ms |
| [[DeepSpeed-Inference]] | 分布式推理、ZeRO优化 | 部署复杂度高 | 超大模型（70B+）部署 | 吞吐量：900+ tok/s，延迟：70ms |
| [[llama.cpp]] | 轻量级、CPU/GPU混合推理 | 性能低于专用引擎 | 本地部署、边缘设备 | 吞吐量：100+ tok/s，延迟：200ms |

### 3.2 选型决策树
1. **模型规模**：
   - < 7B：vLLM / TGI
   - 7B-70B：vLLM / TensorRT-LLM / DeepSpeed-Inference
   - > 70B：DeepSpeed-Inference / vLLM分布式
2. **性能需求**：
   - 低延迟优先：TensorRT-LLM > vLLM > TGI
   - 高吞吐优先：vLLM > TensorRT-LLM > TGI
3. **部署环境**：
   - 云端：vLLM / TGI / TensorRT-LLM
   - 边缘：ONNX Runtime / llama.cpp
   - 本地开发：llama.cpp / vLLM

## 4. 批处理与连续批处理

### 4.1 传统批处理 vs 连续批处理
- **传统批处理**：固定batch大小，等待所有请求完成
- **连续批处理**：动态合并请求，请求完成后立即释放资源
- **连续批处理优势**：
  - 吞吐量提升3-10倍
  - 降低首token延迟
  - 提高GPU利用率

### 4.2 连续批处理实现机制
- **请求调度器**：动态将新请求加入批处理队列
- **KV Cache复用**：共享相同前缀的缓存
- **增量解码**：只计算新增token

### 4.3 vLLM连续批处理配置
```bash
python -m vllm.entrypoints.api_server \
    --model meta-llama/Llama-3-8b-hf \
    --tensor-parallel-size 1 \
    --gpu-memory-utilization 0.9 \
    --max-num-seqs 256  # 最大并发序列数
    --max-model-len 4096
```

## 5. KV Cache管理与显存优化

### 5.1 KV Cache原理
- KV Cache存储注意力机制中的键值对
- 减少重复计算，加速生成过程
- 显存占用随序列长度线性增长

### 5.2 KV Cache优化策略
1. **动态KV Cache**：根据请求长度动态调整缓存大小
2. **KV Cache量化**：INT8/FP8量化，减少显存占用50%
3. **分页KV Cache**：将缓存分割为固定大小的页，提高利用率
4. **KV Cache共享**：相同前缀的请求共享缓存

### 5.3 显存优化配置示例（vLLM）
```bash
python -m vllm.entrypoints.api_server \
    --model meta-llama/Llama-3-8b-hf \
    --gpu-memory-utilization 0.95 \
    --kv-cache-dtype fp8  # 使用FP8量化KV Cache
    --enable-paged-attention  # 启用分页注意力
```

## 6. 模型量化与编译优化

### 6.1 量化技术对比

| 量化类型 | 精度 | 显存节省 | 性能损失 | 工具支持 |
|----------|------|----------|----------|----------|
| FP16 | 半精度 | 50% | 极小 | 所有引擎 |
| INT8 | 8位整数 | 75% | 5-10% | vLLM, TGI, TensorRT-LLM |
| INT4 | 4位整数 | 87.5% | 10-15% | AWQ, GPTQ, GGUF |
| AWQ | 4位自适应 | 87.5% | 5-10% | vLLM, TGI |
| GGUF | 多种精度 | 50-90% | 5-20% | llama.cpp |

### 6.2 编译优化技术
- **Kernel Fusion**：合并多个算子，减少内存访问
- **Tensor Parallelism**：模型并行，分散显存压力
- **Pipeline Parallelism**：流水线并行，提高GPU利用率
- **Auto-Tuning**：自动优化内核参数

### 6.3 量化与编译工作流
```bash
# 1. 使用AWQ量化模型
python -m awq.entrypoints.quantize \
    --model_path meta-llama/Llama-3-8b-hf \
    --w_bit 4 \
    --q_group_size 128 \
    --output_dir ./llama-3-8b-awq

# 2. 使用vLLM加载量化模型
python -m vllm.entrypoints.api_server \
    --model ./llama-3-8b-awq \
    --dtype auto \
    --tensor-parallel-size 1
```

### 7.2 负载均衡策略
- **轮询**：简单易用，适合请求均匀的场景
- **最少连接**：优先分配给连接数最少的实例
- **IP哈希**：同一客户端始终路由到同一实例
- **加权轮询**：根据实例性能分配权重

### 8.1 核心监控指标

#### 8.1.1 性能指标
- **P99延迟**：99%请求的响应时间
- **首token延迟**：第一个token的生成时间
- **吞吐量**：每秒处理的token数

- **请求QPS**：每秒处理的请求数

## 10. 新兴范式与前瞻性探讨

### 10.1 Serverless LLM
- **优势**：按需付费、自动扩缩容、无运维
- **挑战**：冷启动延迟、成本模型复杂
- **平台支持**：AWS Bedrock、Azure OpenAI Service、Google Vertex AI

### 10.2 边缘部署
- **场景**：低延迟要求、离线运行、数据隐私
- **技术**：ONNX Runtime Mobile、llama.cpp、TensorRT Edge
- **挑战**：资源受限、模型精简

### 10.3 多模态服务融合
- **架构**：统一API网关 + 多模态推理引擎
- **挑战**：不同模态的性能优化、统一鉴权
- **工具**：vLLM MultiModal、TGI MultiModal

### 10.4 模型即服务（MaaS）
- **模式**：API-first、标准化接口、计量计费
- **趋势**：模型市场、动态模型切换、版本管理

## 11. 常见陷阱与应对策略

### 11.1 OOM（内存溢出）
- **原因**：模型过大、batch_size设置不当、KV Cache溢出
- **应对**：
  - 启用分页注意力（PagedAttention）
  - 降低max_model_len
  - 减少max_num_seqs
  - 使用模型量化

### 11.2 首token延迟高
- **原因**：模型加载、KV Cache初始化、编译开销
- **应对**：
  - 预热模型（发送测试请求）
  - 启用模型缓存
  - 使用编译优化

### 11.3 长文本截断
- **原因**：max_model_len限制、KV Cache不足
- **应对**：
  - 分段生成
  - 优化KV Cache管理
  - 使用更大显存的GPU

### 11.4 吞吐量波动
- **原因**：请求分布不均、动态批处理调度
- **应对**：
  - 调整连续批处理参数
  - 优化负载均衡策略
  - 增加副本数
