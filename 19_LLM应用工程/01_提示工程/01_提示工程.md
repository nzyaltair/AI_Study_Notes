# 提示工程

## 背景与发展

提示工程（Prompt Engineering）是与[[大语言模型]]交互的核心技术。GPT-3（2020）展示了Few-Shot能力，CoT（Wei et al., 2022）证明逐步推理可显著提升数学与逻辑任务表现，ReAct（Yao et al., 2022）将推理与行动结合，DSPy（2023）实现提示自动优化。提示工程从手工调参发展为系统化的工程学科。

## 核心思想

LLM的行为由输入提示决定。通过结构化的提示设计、示例引导和推理引导，可以显著提升LLM的输出质量和可控性。提示是LLM应用的"软编程"接口。

## 技术原理

### 基础提示技术

| 技术 | 原理 | 适用场景 |
|------|------|---------|
| 零样本（Zero-Shot） | 直接提问，不提供示例 | 简单任务 |
| 少样本（Few-Shot） | 提供3-5个示例引导 | 特定格式/领域任务 |
| 角色提示 | 设定专业角色（"你是XX专家"） | 专业领域 |
| 指令提示 | 明确任务要求与约束 | 格式化输出 |

### 思维链（CoT）

引导模型展示推理过程，适用于数学、逻辑、多步推理任务。详见[[思维链]]。

```
让我们逐步思考：
1. 首先...
2. 然后...
3. 因此...
答案：...
```

### ReAct框架

Thought → Action → Observation循环，将推理与工具调用交替执行。详见[[ReAct]]。

```
Thought: 用户想知道天气，我需要查询
Action: get_weather("北京")
Observation: 晴天 25°C
Thought: 获取到了天气信息，可以回答
Final Answer: 北京今天晴天 25°C
```

### 高级提示技术

| 技术 | 原理 | 应用 |
|------|------|------|
| 自洽性（Self-Consistency） | 多次采样取多数投票 | 提高准确性 |
| 思维树（ToT） | 树形搜索多个推理路径 | 规划类任务 |
| 自我修正 | 生成后审查和修改 | 代码生成 |
| 分解（Decomposition） | 将复杂任务分解为子任务 | 多步推理 |
| Meta-prompting | 用LLM生成提示 | 自动优化 |

### 提示工程化

- **模板化**：使用变量和模板引擎管理提示
- **版本控制**：对提示进行版本管理
- **A/B测试**：对比不同提示效果
- **评估体系**：建立提示质量评估指标
- **工具**：LangChain PromptTemplate、DSPy

## 发展演进

手工提示 → Few-Shot（2020）→ CoT（2022）→ ReAct（2022）→ 自动化提示优化（DSPy, 2023）→ 提示成为"软编程"

## 关键算法·模型

- **CoT**：Chain-of-Thought Prompting（Wei et al., 2022）
- **ReAct**：Synergizing Reasoning and Acting（Yao et al., 2022）
- **ToT**：Tree of Thoughts（Yao et al., 2023）
- **Self-Consistency**：多次采样取多数投票（Wang et al., 2022）
- **DSPy**：声明式提示优化框架（Khattab et al., 2023）

## 应用场景

- LLM应用开发的基础技能
- [[Agent]]系统：Agent行为由提示定义
- [[RAG]]：检索结果整合到提示中
- 数据标注：用LLM自动标注数据
- LLM-as-Judge：用LLM评估输出质量

## 与其他技术关系

- 提示工程是[[Agent]]系统的核心技术
- [[思维链]]推理是Agent规划的基础
- [[结构化输出]]是[[工具调用]]的基础
- [[RAG]]将检索结果注入提示
- 提示质量影响LLM评测结果
