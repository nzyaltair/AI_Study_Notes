# 流式与动态模型转换

## 1. 概述

自回归模型（如 GPT、T5、流式 TTS）在推理时需要逐 token 生成输出，序列长度动态变化。这类模型的 ONNX 转换面临独特挑战：KV Cache 需要在推理步骤间保持、动态序列长度需要正确配置、大模型可能超出单设备内存。

流式与动态模型转换解决的核心问题：
- **动态序列长度**：自回归生成时序列长度逐 token 增长
- **中间状态管理**：KV Cache、RNN hidden state 需在推理步骤间传递
- **模型拆分**：大模型超出单设备内存时的分块部署

核心思想是将中间状态（KV Cache、RNN hidden state）显式化为 ONNX 图的输入/输出，使 ONNX 模型无需"记忆"历史，只需在每次调用时接收和返回缓存状态。

## 3. 核心概念

### 3.1 KV Cache

Transformer 自回归生成中，缓存历史 Token 的 Key/Value 张量，避免重复计算注意力。在 ONNX 中，需将 KV Cache 展平为独立的输入/输出张量。

### 3.2 动态轴（Dynamic Axes）

允许 ONNX 模型接受可变大小的输入维度。对于流式模型，至少需要配置 batch size 和 sequence length 为动态。

### 3.3 模型拆分

大型 Transformer 模型可按层或功能模块拆分为多个独立 ONNX 模型，实现内存优化、Pipeline 并行和灵活部署。

### 3.4 编码器/解码器分离

流式翻译/语音识别的标准策略：
- 编码器：处理完整输入序列，输出 hidden states
- 解码器：逐 token 自回归生成，每步接收编码器输出和 KV Cache

## 4. 技术原理

### 4.1 动态轴高级配置

```python
# NLP 模型：动态 batch + 动态序列长度
dynamic_axes = {
    'input_ids': {0: 'batch_size', 1: 'seq_length'},
    'attention_mask': {0: 'batch_size', 1: 'seq_length'},
    'output': {0: 'batch_size'}
}

# CV 模型：动态 batch + 动态分辨率
dynamic_axes = {
    'image': {0: 'batch', 2: 'height', 3: 'width'},
    'output': {0: 'batch'}
}
```

**最佳实践**：
- 仅标记真正需要动态的维度，保持其他固定以获得更好优化
- 使用有意义的维度名称
- 使用 Opset 13+ 获得完整的动态形状支持
- 用不同 batch size 和序列长度测试

### 4.2 KV Cache 管理策略

将 KV Cache 展平为独立的输入/输出张量：

```python
class GPT2OnnxWrapper(torch.nn.Module):
    def __init__(self, model):
        super().__init__()
        self.model = model

    def forward(self, input_ids, attention_mask, *past_key_values):
        num_layers = self.model.config.num_hidden_layers
        past = tuple(
            (past_key_values[i*2], past_key_values[i*2+1])
            for i in range(num_layers)
        )
        outputs = self.model(
            input_ids=input_ids,
            attention_mask=attention_mask,
            past_key_values=past,
            use_cache=True
        )
        logits = outputs.logits
        new_past = outputs.past_key_values
        flattened_past = tuple(pk for layer in new_past for pk in layer)
        return (logits,) + flattened_past
```

为每层的 key 和 value 定义独立的输入/输出名称和动态轴：

```python
num_layers = model.config.num_hidden_layers
input_names = ['input_ids', 'attention_mask'] + \
    [f'past_key_{i}' for i in range(num_layers)] + \
    [f'past_value_{i}' for i in range(num_layers)]
output_names = ['logits'] + \
    [f'present_key_{i}' for i in range(num_layers)] + \
    [f'present_value_{i}' for i in range(num_layers)]
```

### 4.3 RNN/LSTM 流式转换

LSTM 的 hidden state 需作为显式输入/输出：

```python
torch.onnx.export(
    model,
    (dummy_input, h_0, c_0),  # 输入含初始 hidden state
    "asr_streaming.onnx",
    input_names=['input', 'h_0', 'c_0'],
    output_names=['logits', 'h_n', 'c_n'],
    dynamic_axes={'input': {0: 'batch', 1: 'chunk_length'}},
    opset_version=14
)
```

### 4.4 Chunked Attention

Whisper 等模型将长音频切分为重叠 chunk 逐段处理，避免超出最大序列长度。

## 5. 关键方法与模型

### 5.1 Beam Search 扩展

多个 beam 路径需要复制和选择 KV Cache，通过 `repeat_interleave` 和索引操作实现。

### 5.2 Pipeline 并行

多个 ONNX session 串联，前一个的输出作为后一个的输入，通过异步执行重叠计算与 IO。

### 5.3 模型拆分优化

- **内存优化**：每个子模型单独加载，降低峰值内存
- **Pipeline 并行**：不同子模型分配到不同设备
- **灵活部署**：编码器一次性运行，解码器逐 token 运行

## 6. 优势与局限

### 6.1 优势

- **灵活性**：动态轴支持可变输入
- **状态管理**：KV Cache 显式化，推理引擎无需维护状态
- **可扩展**：模型拆分支持超显存模型的部署

### 6.2 局限

- **复杂度高**：KV Cache 展平导致大量输入/输出张量
- **性能开销**：动态形状可能影响推理优化效果
- **LLM 适用性差**：大语言模型更适合 vLLM/TensorRT-LLM 等专用引擎

## 7. 应用场景

| 场景 | 模型类型 | 关键策略 |
|:---|:---|:---|
| 文本生成 | GPT 系列 | KV Cache + 动态 sequence length |
| 机器翻译 | T5/BART | 编码器/解码器分离 + KV Cache |
| 语音识别 | Whisper/LSTM | 分块处理 + hidden state 传递 |
| 流式 TTS | Tacotron/VITS | 分段推理 + 状态缓存 |
| 对话系统 | LLM | 上下文窗口管理 + cache 修剪 |
