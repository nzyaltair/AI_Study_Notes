# 软件工程基础

## 背景与发展

AI 系统具有高复杂性、大数据规模和实验驱动特征，软件工程原则对构建可靠、可维护的 AI 系统至关重要。传统软件工程关注代码可维护性，AI 工程还需额外关注实验可复现性、模型版本管理和数据一致性。

## 核心思想

软件工程以系统化、规范化方法管理软件生命周期，核心原则包括抽象、模块化、封装、低耦合高内聚。在 AI 开发中，这些原则延伸为：将模型、数据、实验三要素纳入版本管理，构建可复现的端到端流水线。

## 技术原理

### SOLID 设计原则

- **单一职责**（SRP）：每个模块只负责一个功能领域
- **开闭原则**（OCP）：对扩展开放，对修改关闭
- **里氏替换**（LSP）：子类应能无副作用地替换父类
- **接口隔离**（ISP）：不依赖不需要的接口
- **依赖倒置**（DIP）：依赖抽象而非具体实现

### 版本控制

Git 管理 AI 项目的代码与配置。AI 项目特有的版本管理需求：
- `.gitignore` 排除数据文件和模型权重
- DVC 管理大数据版本，与 Git 协同追踪数据快照
- MLflow / Weights & Biases 追踪实验参数、指标和模型版本

### 测试

- **单元测试**（pytest）：验证数据处理函数、模型前向传播逻辑
- **集成测试**：验证 Pipeline 端到端行为
- **数据质量测试**：验证输入数据分布、缺失率、特征范围
- **模型性能测试**：回归测试确保新版本不降低性能指标
- **对抗性测试**：验证模型鲁棒性

### CI/CD for ML

- **CI**：代码提交时自动运行测试、类型检查、代码规范
- **CD**：自动部署模型到预发布/生产环境
- **CT**（Continuous Training）：数据更新触发自动重训练
- 工具链：GitHub Actions、GitLab CI、Kubeflow Pipelines

### 代码质量

- **静态分析**：ruff、mypy（类型检查）
- **代码格式**：black、isort
- **依赖管理**：pyproject.toml + poetry / uv
- **文档**：docstring 规范，Sphinx 自动生成

### 可复现性

AI 实验的可复现性要求：
- 固定随机种子（Python、NumPy、PyTorch、CUDA）
- 记录完整环境（依赖版本、CUDA 版本、硬件）
- 版本化数据快照与代码 commit
- 记录训练超参数与最终指标

## 发展演进

瀑布模型 → 敏捷开发 → DevOps（CI/CD 自动化）→ MLOps（增加模型与数据版本管理）→ LLMOps（大模型时代的提示词版本管理、A/B 评估、在线监控）。

## 关键算法·模型

- **Git 分支策略**：Git Flow（main/develop/feature/hotfix）、Trunk-Based Development
- **DVC**：基于内容哈希的数据版本追踪，`dvc push/pull` 同步远程存储
- **MLflow**：实验追踪（Tracking）、模型注册（Model Registry）、部署服务（Model Serving）

## 应用场景

- MLOps：从实验到生产的自动化流水线
- 大模型部署：模型服务化、API 设计、流量管理
- Agent 开发：模块化设计、测试驱动、迭代验证
- 数据团队协作：代码审查、分支管理、持续集成

## 与其他技术关系

- 版本控制与 [[数据处理与工程化]] 中的 DVC 数据版本管理协同工作
- CI/CD 是 MLOps 的核心实践，连接开发与运维
- 可复现性原则贯穿模型训练与实验管理全过程
- 类型注解与静态分析支撑大型 AI 项目的代码质量
