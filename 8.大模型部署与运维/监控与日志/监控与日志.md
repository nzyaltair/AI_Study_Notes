# 监控与日志

## 1. 概述

### 1.1 目标定位
本文档聚焦大模型在生产环境中运行时的可观测性需求，兼顾理论框架与工程落地，帮助团队构建高效、可靠的监控与日志体系，确保[[推理服务]]的高性能、高可用和安全性。

### 1.2 核心挑战
- **非确定性输出**：LLM输出的随机性导致传统监控指标难以全面评估服务质量
- **超大模型规模**：TB级模型带来的显存、计算资源监控复杂度
- **长尾请求**：长文本生成场景下的请求延迟分布不均
- **多轮对话状态**：会话级监控需要追踪跨请求的状态变化
- **复杂依赖关系**：硬件、框架、模型、数据集的多层依赖监控
- **敏感信息处理**：日志中的用户输入/输出需要脱敏处理

## 2. 监控指标体系

### 2.1 请求级指标

| 指标名称 | 描述 | 计算方式 | 关键阈值 | 监控频率 |
|----------|------|----------|----------|----------|
| **P99延迟** | 99%请求的响应时间 | 按请求时间排序，取第99百分位 | < 1s（在线服务） | 1分钟 |
| **首token延迟** | 第一个token生成时间 | 请求开始到第一个token输出的时间差 | < 500ms | 1分钟 |
| **吞吐量** | 每秒处理的请求数 | 总请求数 / 时间窗口 | 业务相关 | 1分钟 |
| **token吞吐量** | 每秒生成的token数 | 总生成token数 / 时间窗口 | 硬件相关 | 1分钟 |
| **错误率** | 请求失败比例 | 失败请求数 / 总请求数 | < 0.1% | 1分钟 |
| **输入长度分布** | 请求输入的token长度分布 | 统计不同长度区间的请求占比 | 业务相关 | 5分钟 |
| **输出长度分布** | 生成输出的token长度分布 | 统计不同长度区间的输出占比 | 业务相关 | 5分钟 |
| **超时率** | 超时请求比例 | 超时请求数 / 总请求数 | < 0.5% | 1分钟 |

### 2.2 模型级指标

| 指标名称 | 描述 | 计算方式 | 关键阈值 | 监控频率 |
|----------|------|----------|----------|----------|
| **KV Cache命中率** | [[KV Cache]]的复用比例 | 命中的KV Cache条目数 / 总访问数 | > 90% | 1分钟 |
| **token生成速度** | 每秒生成的token数 | 生成token数 / 生成时间 | 硬件相关 | 1分钟 |
| **MoE路由分布** | 稀疏激活模型的专家使用分布 | 各专家被激活的次数占比 | 分布均匀性 | 5分钟 |
| **量化误差** | 量化模型与原始模型的输出差异 | 输出概率分布的KL散度 | < 0.1 | 1小时 |
| **生成多样性** | 输出结果的多样性指标 | n-gram重复率、困惑度 | 业务相关 | 1小时 |
| **幻觉率** | 生成内容的事实错误比例 | 错误回答数 / 总回答数 | < 5% | 1小时 |
| **对齐一致性** | 输出与系统提示的一致性 | 人工或自动评估的一致性评分 | > 95% | 1小时 |

### 2.3 系统级指标

| 指标名称 | 描述 | 计算方式 | 关键阈值 | 监控频率 |
|----------|------|----------|----------|----------|
| **GPU利用率** | GPU计算核心利用率 | nvidia-smi或类似工具采集 | 业务相关 | 10秒 |
| **GPU显存使用率** | GPU显存占用比例 | 已用显存 / 总显存 | < 90% | 10秒 |
| **GPU功耗** | GPU实时功耗 | nvidia-smi采集 | 硬件限制 | 10秒 |
| **CPU利用率** | CPU核心利用率 | 系统监控工具采集 | < 80% | 10秒 |
| **内存使用率** | 系统内存占用比例 | 已用内存 / 总内存 | < 80% | 10秒 |
| **网络I/O** | 网络输入/输出速率 | 系统监控工具采集 | 业务相关 | 1分钟 |
| **磁盘I/O** | 磁盘读写速率 | 系统监控工具采集 | 业务相关 | 1分钟 |
| **OOM发生率** | 内存溢出事件频率 | OOM事件数 / 时间窗口 | 0 | 1分钟 |

### 2.4 业务级指标

| 指标名称 | 描述 | 计算方式 | 关键阈值 | 监控频率 |
|----------|------|----------|----------|----------|
| **用户满意度** | 用户对生成结果的满意度评分 | 显式反馈或隐式行为分析 | > 4.0（5分制） | 1小时 |
| **有害内容率** | 生成有害内容的比例 | 自动检测或人工审核 | < 0.1% | 1小时 |
| **多轮对话成功率** | 成功完成多轮对话的比例 | 完成多轮交互的会话数 / 总会话数 | > 90% | 1小时 |
| **任务完成率** | 成功完成指定任务的比例 | 任务成功数 / 总任务数 | 业务相关 | 1小时 |
| **API调用成本** | 每千次API调用的成本 | 总费用 / (总调用数 / 1000) | 预算相关 | 1小时 |
| **模型调用频率** | 特定模型的调用频率 | 模型调用次数 / 时间窗口 | 业务相关 | 1分钟 |
| **用户留存率** | 重复使用服务的用户比例 | 重复用户数 / 总用户数 | 业务相关 | 1天 |

## 3. 日志体系设计

### 3.1 日志设计原则
- **结构化**：使用JSON格式，便于机器解析和查询
- **可追踪**：包含trace ID和span ID，支持全链路追踪
- **脱敏处理**：用户输入/输出中的敏感信息必须脱敏
- **分级分类**：按日志级别（DEBUG/INFO/WARN/ERROR）和类型分类
- **高效存储**：采用压缩、采样等策略优化存储成本
- **可扩展**：支持自定义字段和动态扩展

### 3.3 日志分级与采样

#### 3.3.1 日志级别
- **DEBUG**：详细的调试信息，仅在开发环境启用
- **INFO**：正常的业务流程日志，包含关键请求和响应信息
- **WARN**：警告信息，如资源使用率接近阈值
- **ERROR**：错误信息，如请求失败、OOM事件
- **FATAL**：致命错误，如服务崩溃

#### 3.3.2 日志采样策略
1. **按比例采样**：如采样10%的INFO级别日志
2. **按请求类型采样**：对特定API或请求类型全量采样
3. **按错误级别采样**：ERROR和FATAL级别全量采样
4. **智能采样**：基于请求特征（如长请求、异常输入）动态调整采样率

## 4. 工具链推荐与适配

### 4.1 监控工具链

#### 4.1.1 传统监控栈
- **Prometheus + Grafana**：
  - 优势：开源、生态成熟、可视化能力强
  - LLM适配：自定义Exporter采集模型级指标，构建LLM专用Dashboard
  - 示例配置：
    ```yaml
    # Prometheus配置片段
    scrape_configs:
      - job_name: 'llm-service'
        scrape_interval: 10s
        static_configs:
          - targets: ['llm-service:8000']
        metrics_path: '/metrics'
    ```

#### 4.1.2 分布式追踪
- **OpenTelemetry**：
  - 优势：厂商中立、统一的可观测性标准
  - LLM适配：追踪多轮对话、模型调用链、外部依赖
  - 示例代码：
    ```python
    from opentelemetry import trace
    from opentelemetry.sdk.trace import TracerProvider
    from opentelemetry.sdk.trace.export import BatchSpanProcessor
    
    # 初始化追踪器
    trace.set_tracer_provider(TracerProvider())
    tracer = trace.get_tracer(__name__)
    
    with tracer.start_as_current_span("llm_inference") as span:
        span.set_attribute("model.name", "Llama-3-70B")
        span.set_attribute("prompt.tokens", 128)
        # 模型推理代码
        span.set_attribute("generated.tokens", 256)
        span.set_attribute("inference.time_ms", 1200)
    ```

#### 4.1.3 LLM专用监控工具
- **LangSmith**：
  - 优势：专为LLM设计，支持提示词管理、评估、监控
  - 适配场景：开发到生产的全生命周期监控
  - 关键功能：对话历史追踪、评估指标、成本分析

- **Phoenix**：
  - 优势：开源LLM监控平台，实时可视化
  - 适配场景：生产环境监控、模型漂移检测
  - 关键功能：异常检测、性能监控、成本优化

- **Gantry**：
  - 优势：专注于LLM生产监控和质量评估
  - 适配场景：模型质量监控、漂移检测
  - 关键功能：实时质量评分、根因分析

## 5. 告警规则设计

### 5.1 告警分类
- **性能告警**：延迟、吞吐量异常
- **资源告警**：GPU/CPU/内存使用率过高
- **错误告警**：错误率、超时率突增
- **质量告警**：幻觉率、对齐一致性下降
- **安全告警**：异常请求、敏感信息泄露

### 5.2 典型告警规则示例

#### 5.2.1 Prometheus告警规则
```yaml
# P99延迟突增告警
alert: LLMP99LatencyHigh
  expr: histogram_quantile(0.99, sum(rate(llm_request_duration_seconds_bucket[5m])) by (le, service)) > 2
  for: 1m
  labels:
    severity: warning
  annotations:
    summary: "LLM P99延迟过高"
    description: "服务 {{ $labels.service }} 的P99延迟超过2秒，当前值: {{ $value }}秒"

# GPU显存使用率过高告警
alert: LLMGPUMemoryHigh
  expr: llm_gpu_memory_usage_ratio > 0.95
  for: 3m
  labels:
    severity: critical
  annotations:
    summary: "GPU显存使用率过高"
    description: "节点 {{ $labels.instance }} 的GPU显存使用率超过95%，当前值: {{ $value | humanizePercentage }}"

# OOM事件频发告警
alert: LLMOOMEventsFrequent
  expr: rate(llm_oom_events_total[5m]) > 1
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "OOM事件频发"
    description: "服务 {{ $labels.service }} 在过去5分钟内发生了超过1次OOM事件"
```

### 5.3 告警处理流程
1. **告警触发**：监控系统检测到指标异常
2. **告警路由**：根据告警级别和类型路由到相应团队
3. **告警抑制**：避免告警风暴，设置合理的告警间隔
4. **根因分析**：结合监控指标和日志进行分析
5. **告警修复**：采取相应的修复措施
6. **告警关闭**：修复后自动或手动关闭告警
7. **告警复盘**：定期复盘告警，优化告警规则

## 6. 监控数据的运维闭环应用

### 6.1 自动扩缩容
- **基于指标的扩缩容**：
  - 触发条件：GPU利用率 > 80% 持续5分钟
  - 实现方式：Kubernetes HPA或自定义控制器
  - 示例配置：
    ```yaml
    apiVersion: autoscaling/v2
    kind: HorizontalPodAutoscaler
    metadata:
      name: llm-service-hpa
    spec:
      scaleTargetRef:
        apiVersion: apps/v1
        kind: Deployment
        name: llm-service
      minReplicas: 2
      maxReplicas: 10
      metrics:
      - type: Pods
        pods:
          metric:
            name: llm_gpu_utilization
          target:
            type: AverageValue
            averageValue: 80
    ```

### 6.2 A/B测试与模型迭代
- **性能对比**：比较不同模型版本的延迟、吞吐量
- **质量评估**：结合业务指标评估模型输出质量
- **决策支持**：基于监控数据决定是否全量发布新模型

### 6.3 自动回滚机制
- **触发条件**：
  - 错误率 > 5% 持续2分钟
  - P99延迟 > 3秒 持续3分钟
  - 幻觉率 > 10% 持续5分钟
- **实现方式**：结合Kubernetes rollout和监控告警
- 示例流程：
  ```
  告警触发 → 自动回滚 → 服务验证 → 告警关闭 → 复盘分析
  ```

### 6.4 成本优化
- **资源利用率分析**：识别低利用率的资源
- **成本-性能权衡**：在成本和性能之间找到最优平衡点
- **按需分配**：根据业务负载动态调整资源分配

## 7. LLM特有挑战与应对

### 7.1 非确定性输出的监控
- **挑战**：相同输入可能产生不同输出，传统监控难以评估
- **应对策略**：
  - 统计多轮输出的分布特征
  - 使用一致性指标评估输出稳定性
  - 结合人工抽样评估

### 7.2 长尾请求处理
- **挑战**：长文本生成请求导致延迟分布不均
- **应对策略**：
  - 按请求长度分层监控
  - 实现请求优先级队列
  - 长请求超时保护机制

### 7.3 多轮对话状态追踪
- **挑战**：需要跨请求追踪对话状态
- **应对策略**：
  - 使用session ID关联多轮请求
  - 监控对话长度、轮次数
  - 检测对话上下文丢失情况

### 7.4 模型漂移检测
- **挑战**：模型性能随时间下降
- **应对策略**：
  - 定期评估模型性能指标
  - 监控输入分布变化
  - 实现自动重新校准机制

大模型监控与日志体系是确保[[推理服务]]高可用、高性能和高安全性的关键基础设施。随着LLM技术的快速发展，传统的监控范式面临着新的挑战，需要结合LLM的特点进行创新和适配。

未来的发展趋势包括：
1. **智能化**：基于LLM的智能监控和日志分析
2. **一体化**：监控、日志、追踪的深度融合
3. **语义化**：从数值指标到语义级监控
4. **自适应**：动态调整监控策略和告警规则
5. **成本敏感**：成本-性能联合优化

对于AI工程师和MLOps从业者而言，建立完善的可观测性体系是大模型生产化的必备条件，需要持续学习和适应新的技术趋势。
