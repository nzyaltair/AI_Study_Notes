# 实验管理与复现

## 背景与发展

[[深度学习]]实验具有随机性强（初始化、数据 shuffle）、超参数多、训练周期长、计算资源消耗大等特点。缺乏系统化的实验管理会导致结果不可复现、实验对比困难、团队协作混乱。实验管理从手动记录发展到 Excel/文档管理，再到 TensorBoard 本地可视化，最终演进为 MLflow/W&B 平台化管理，并融入 MLOps 全流程。

## 核心思想

将实验视为可追踪、可复现、可比较的科学过程，通过版本控制（数据、代码、环境）、系统化记录（参数、指标、日志）和标准化流程管理实验全生命周期。

## 技术原理

### 实验追踪要素

| 要素 | 内容 | 工具 |
|------|------|------|
| **代码版本** | 模型代码、训练脚本、配置文件 | Git |
| **数据版本** | 训练/验证数据集版本、预处理 pipeline | DVC、LakeFS |
| **环境版本** | 依赖包、CUDA 版本、硬件配置 | Docker、conda |
| **实验记录** | 超参数、指标曲线、日志、模型 checkpoint | MLflow、W&B、TensorBoard |

### 可复现性保障

- **随机种子固定**：`torch.manual_seed()`、`np.random.seed()`、`random.seed()`、CUDA deterministic
- **数据加载确定性**：`DataLoader(num_workers=0)` 或 `worker_init_fn` 固定 worker 种子
- **模型 checkpoint**：保存完整训练状态（模型参数、优化器状态、epoch、调度器状态）
- **配置管理**：使用 YAML/Hydra 管理超参数，避免硬编码

### 实验管理平台对比

| 平台 | 特点 | 适用场景 |
|------|------|---------|
| **MLflow** | 开源、自托管、支持多框架 | 企业内部实验管理 |
| **Weights & Biases** | 云端、可视化强、协作友好 | 团队协作研究 |
| **TensorBoard** | 轻量、本地、TensorFlow 原生 | 个人实验快速可视化 |
| **DVC** | 数据版本控制、与 Git 集成 | 数据集版本管理 |
| **Neptune.ai** | 元数据管理、实验对比 | 大规模实验对比 |

### 实验对比与消融研究

- **消融实验（Ablation Study）**：逐个移除组件，验证各模块贡献
- **超参数搜索**：Grid Search、Random Search、Bayesian Optimization（Optuna）
- **实验报告**：记录实验目的、配置、结果、分析、结论，支持可追溯决策

## 发展演进

手动记录 $\to$ Excel/文档管理 $\to$ TensorBoard 本地可视化 $\to$ MLflow/W&B 平台化管理 $\to$ MLOps 全流程集成

## 关键算法·模型

- **数据版本控制**：DVC 基于 Git 扩展，管理大数据集版本
- **实验追踪**：MLflow Tracking 记录参数、指标、产出物
- **模型注册表**：MLflow Model Registry 管理模型版本和阶段（Staging/Production）
- **配置管理**：Hydra 支持层次化配置和命令行覆盖

## 应用场景

- **学术研究**：论文实验复现、消融实验记录
- **工业项目**：团队协作开发、模型迭代管理
- **LLM 训练**：大规模分布式训练的实验追踪与 checkpoint 管理
- **模型选型**：多模型对比、超参数搜索结果分析

## 与其他技术关系

- 实验管理是[[机器学习流水线]]的工程化保障
- 与[[深度学习框架]]的 checkpoint 机制紧密相关
- Docker 容器化保障环境可复现
- 是模型版本管理和部署的基础
