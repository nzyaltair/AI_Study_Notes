# 机器学习流水线

## 1. 概述

机器学习流水线（ML Pipeline）是将 ML 项目的各个阶段——从数据收集到模型监控——标准化和自动化的工程实践。它解决了传统 ML 开发中的核心痛点：手动操作多、实验难复现、部署不规范、生产环境模型退化难以察觉。

MLOps（Machine Learning Operations）是 ML Pipeline 的体系化延伸，借鉴 DevOps 的 CI/CD 思想，将机器学习系统的生命周期管理提升到工程化水平。Google 2015 年的论文《Hidden Technical Debt in Machine Learning Systems》最早系统性地指出了 ML 系统的工程挑战。

## 2. 发展历史

| 年代 | 里程碑 | 意义 |
|------|--------|------|
| 2015 | Google 发表 "Hidden Technical Debt in ML Systems" | MLOps 思想萌芽 |

| 2018 | Kubeflow 发布（Google） | K8s 原生的端到端 ML 平台 |
| 2018 | MLflow 发布（Databricks） | 实验跟踪与模型管理标准 |
| 2019 | TFX 开源（Google） | 生产级 ML Pipeline 框架 |
| 2020 | MLOps 概念爆发 | 从脚本化到体系化工程实践 |
| 2021+ | Feature Store（Feast/Tecton） | 特征生命周期管理 |
| 2023+ | LLMOps 兴起 | 大模型特有的运维挑战 |

## 3. 核心概念

### 3.1 ML Pipeline 标准架构

```
数据收集 → 数据验证 → 特征工程 → 模型训练 → 模型评估 → 模型注册 → 模型部署 → 模型监控
    ↑                                                                                    │
    └────────────────────────── 反馈循环（重新训练/更新）─────────────────────────────────┘
```

### 3.2 MLOps 成熟度模型

| 等级 | 名称 | 特征 |
|------|------|------|
| **L0** | 手动 | 脚本驱动、无版本控制、手动部署 |
| **L1** | 自动化训练 | CI/CD 触发重训练、模型版本化 |
| **L2** | CI/CD Pipeline | 完整的持续集成、持续部署、持续训练（CT） |

### 3.3 关键的"地狱"

ML 系统独有的工程债务：

1. **纠缠（Entanglement）**：改变一个特征的分布可能影响所有依赖它的模型（CACE 原则：Change Anything Changes Everything）
2. **数据依赖**：数据管道的复杂性远超代码依赖
3. **反馈回路**：模型预测影响未来数据，形成自我强化的反馈循环
4. **未声明的消费者**：一个模型的输出可能被其他系统静默使用（级联失败）
5. **配置债务**：ML 系统的配置远比传统软件复杂（特征、模型、数据、超参数混合）

## 4. 技术原理

### 4.1 各阶段详解

#### 阶段 1：数据收集与验证

**职责**：
- 数据源接入（数据库、数据湖、实时流）
- 数据版本控制（DVC、LakeFS、Delta Lake）
- 数据质量检查：Schema 验证、统计分布检查、缺失值/异常值监控

**关键工具**：
- **DVC**（Data Version Control）：类 Git 的数据版本管理
- **Great Expectations**：数据质量断言和文档化
- **TFX Data Validation**：Schema 推断 + 漂移检测

#### 阶段 2：特征工程流水线

**职责**：
- 特征提取、变换、选择
- 特征存储（Feature Store）管理在线/离线一致性
- 训练-服务特征偏差（Training-Serving Skew）检测

**Feature Store 核心能力**：

| 能力 | 说明 |
|------|------|
| **离线存储** | 存储历史特征用于训练（如 Parquet 文件） |
| **在线存储** | 低延迟特征服务用于实时推理（如 Redis） |
| **特征注册** | 特征定义、元数据、血缘追踪 |
| **一致性保证** | 在线和离线使用完全相同的特征计算逻辑 |

#### 阶段 3：模型训练与实验管理

**职责**：
- 实验跟踪：记录每次训练的超参数、指标、代码版本、数据版本
- 模型注册（Model Registry）：管理模型版本、阶段（Staging/Production/Archived）
- 训练流水线编排

**MLflow 四大组件**：
1. **Tracking**：记录参数、指标、artifacts
2. **Projects**：打包 ML 代码（conda/docker 环境）
3. **Models**：统一模型格式和部署接口
4. **Registry**：模型生命周期管理

#### 阶段 4：模型部署

**部署模式**：

| 模式 | 做法 | 适用场景 |
|------|------|---------|
| **批处理预测** | 离线批量推理 | 报表生成、离线评分 |
| **在线同步推理** | REST/gRPC API | 实时推荐、风控 |
| **异步推理** | 消息队列 + 回调 | 大文件处理、耗时推理 |
| **边缘部署** | 模型压缩后部署到设备 | IoT、移动端 |

**部署策略**：

| 策略 | 做法 | 风险 |
|------|------|------|
| **蓝绿部署** | 两套环境，一键切换 | 低 |
| **金丝雀发布** | 小流量 → 逐步扩大 | 最低 |
| **影子模式** | 新旧模型并行运行，新模型不返回结果 | 无（仅观测） |

#### 阶段 5：模型监控

**监控维度**：

| 维度 | 内容 | 工具 |
|------|------|------|
| **性能监控** | 预测延迟、吞吐量、错误率 | Prometheus + Grafana |
| **数据漂移** | $P(X)$ 分布变化 | Evidently AI、Alibi Detect |
| **概念漂移** | $P(Y|X)$ 变化 | NannyML、Evidently AI |
| **业务指标** | 点击率、转化率等上下游指标 | 业务监控系统 |
| **公平性监控** | 不同群体间的性能差异 | Fairlearn |

**漂移检测方法**：
- **KS 检验**：两样本分布是否来自同一总体（非参数）
- **AD 检验**（Anderson-Darling）：对尾部差异更敏感的分布检验
- **PSI**（Population Stability Index）：工业和金融中的标准漂移指标
- **Wasserstein 距离**：概率分布间的 Earth Mover's Distance

### 4.2 流水线编排工具

| 工具 | 类型 | 核心特点 | 最佳场景 |
|------|------|---------|---------|
| **Airflow** | 通用工作流 | DAG 定义、丰富集成 | 数据工程 ETL |
| **Kubeflow** | K8s ML 平台 | 端到端 ML、分布式训练 | 大规模 K8s 部署 |
| **MLflow** | 实验+模型管理 | 轻量、与框架无关 | 实验管理和追踪 |
| **TFX** | Google ML Pipe | 生产验证、Apache Beam | TensorFlow 生态 |
| **Prefect** | 现代工作流 | Pythonic API、动态 DAG | 数据科学与 ML |
| **Metaflow** | Netflix ML Pipe | 原型→生产无缝 | 单机→云扩展 |
| **ZenML** | 可扩展 MLOps | Pipeline as Code | 多云 MLOps |

### 4.3 LLMOps 的特殊挑战

| 挑战 | 传统 MLOps | LLMOps |
|------|-----------|--------|
| **模型规模** | MB–GB | GB–TB |
| **推理延迟** | 毫秒级 | 秒级（自回归生成） |
| **评估方式** | 标准指标（AUC/RMSE） | 人类偏好、自动评测（GPT-as-Judge） |
| **数据管理** | 结构化数据为主 | 提示词 + 非结构化文本 |
| **部署要求** | CPU/小 GPU | 多 GPU/分布式推理 |
| **优化技术** | 常规压缩 | 量化（GPTQ/AWQ）、KV-Cache、FlashAttention |
| **版本管理** | 模型 + 数据 | 模型 + 提示词 + 配置 |
| **成本模型** | 计算成本 | Token 成本 + 推理延迟 |

## 6. 优势与局限

### 6.1 优势

- **可复现性**：代码、数据、模型三合一版本管理
- **自动化**：CI/CD/CT 减少手动操作和人为错误
- **可观测性**：漂移检测和性能监控及时发现模型退化
- **合规性**：模型血缘追踪支持审计和合规要求

### 6.2 局限

- **前期投入大**：搭建 MLOps 基础设施需要显著工程投入
- **过度工程化风险**：小团队/早期项目可能不需要全套 MLOps
- **工具碎片化**：MLOps 生态工具众多，集成成本高
- **人才稀缺**：同时精通 ML 和 DevOps 的工程师稀缺

## 7. 应用场景

- **互联网金融**：实时风控模型需要低延迟在线推理 + 漂移监控
- **推荐系统**：特征工程复杂、模型频繁更新，需要完整 Pipeline
- **大模型应用**：LLM 的 RAG 系统的监控、检索质量评估
- **边缘设备**：手机/IoT 模型需压缩、量化后部署，并监控设备端性能

- CI/CD 理念来自软件工程实践

- LLMOps 是 MLOps 在大模型时代的延伸，涉及量化、KV-Cache 等特有技术

## 9. 前沿发展

- **Feature Platform as a Service**：Feature Store 标准化推动特征资产的平台化管理
- **主动学习闭环**：模型不确定性 > 人工标注 > 重训练的自动化循环
- **负责任的 ML Pipeline**：将公平性、可解释性、安全评估嵌入流水线
- **边缘 MLOps**：ONNX 导出、模型量化、边缘监控的端到端方案
- **FinOps for ML**：ML 训练和推理成本优化（Spot 实例、弹性推理）
- **AIOps + MLOps 融合**：用 AI 监控 ML——自动检测和响应生产问题
- **多模型编排**：一个 Pipeline 管理数十个模型的并行训练/部署（如 A/B 测试、多租户模型）
