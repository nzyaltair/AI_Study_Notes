# API设计与治理

## 背景与发展

大模型 API 是 LLM-as-a-Service 平台的核心接口。OpenAI 在 2023 年发布 Chat Completions API 后，其请求/响应格式逐渐成为行业标准，vLLM、TGI 等推理引擎均提供兼容接口。API 治理涵盖协议选型、认证授权、限流熔断、版本管理和可观测性，直接决定平台的可维护性、安全性和规模化能力。

## 核心思想

API 设计需在易用性、性能和安全性之间取得平衡。对外公开服务优先选择 RESTful 以获得广泛兼容性；内部高性能通信使用 gRPC 获得二进制序列化和双向流式支持。治理体系通过认证（API Key / OAuth2 / JWT）、限流（令牌桶/滑动窗口）、熔断（Circuit Breaker）和版本控制（URI/Header 版本化）保障服务可靠性。

## 技术原理

### 协议选型

| 协议 | 序列化 | 流式支持 | 适用场景 |
|------|--------|----------|----------|
| RESTful | JSON | SSE 单向流 | 公开 API，浏览器兼容 |
| gRPC | Protobuf | 双向流 | 内部服务，高性能通信 |
| GraphQL | JSON | 无 | 灵活数据查询 |

### 请求/响应结构

OpenAI 兼容格式的核心字段：`model`（模型标识）、`messages`（对话历史）、`temperature`（采样温度）、`max_tokens`（生成上限）、`stream`（流式开关）、`tools`（函数调用定义）。

流式响应采用 SSE 格式，每个 chunk 携带增量 Token：

```
data: {"choices":[{"delta":{"content":"API"}}]}
data: {"choices":[{"delta":{"content":" design"}}]}
data: [DONE]
```

### 认证与访问控制

- **API Key**：简单适用，适合低安全要求场景，需配合 HTTPS 和定期轮换
- **OAuth2.0**：标准化协议，支持细粒度权限委托，适合第三方应用集成
- **JWT**：无状态令牌，自包含用户信息，适合微服务间通信和高并发场景
- **RBAC**：基于角色的访问控制，遵循最小权限原则

### 限流与熔断

**令牌桶算法**：以固定速率向桶中添加令牌，请求消耗令牌，允许突发流量。桶容量 $C$，填充速率 $r$，在时间 $t$ 内可处理的最大请求数为 $C + r \cdot t$。

**熔断器模式**：三个状态闭环——闭合（正常请求）、打开（失败率达阈值时拒绝所有请求）、半开（尝试少量请求验证恢复）。防止下游服务故障导致级联失败。

### 版本管理

- **URI 版本控制**：`/v1/chat/completions` → `/v2/chat/completions`，直观但 URL 膨胀
- **Header 版本控制**：`Accept: application/json; version=2.0`，URL 不变但调试复杂
- **向后兼容**：添加可选字段为兼容变更；删除端点或更改字段类型为不兼容变更

### 分布式追踪

OpenTelemetry 提供统一的 Trace/Span 模型，关联日志、指标和追踪数据。一次 LLM 请求的典型追踪链：`http-receive → authentication → model-router → tokenizer → model-inference → detokenizer → http-send`。

## 发展演进

REST API → OpenAI Chat Completions 标准化 → 函数调用与工具使用 → 多模态统一接口 → Agent 协作协议

## 关键算法·模型

- **令牌桶限流**：平滑速率控制，允许突发，公式为可用令牌数 $\min(C, r \cdot \Delta t)$
- **滑动窗口限流**：维护时间窗口内的请求计数，更精确但内存开销高于固定窗口
- **熔断状态机**：基于失败率阈值 $\theta$ 和恢复超时 $T$ 的状态转换

## 应用场景

- **LLM API 平台**：提供 OpenAI 兼容接口的公共服务
- **企业内部模型服务**：多团队共享模型资源，需配额和审计
- **多模型路由**：按请求参数动态路由到不同模型实例
- **A/B 测试**：通过网关层流量分配对比不同模型版本效果
