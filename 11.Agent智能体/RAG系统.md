# RAG系统

## 背景

检索增强生成（RAG）通过从外部知识库检索相关信息来增强[[大语言模型|LLM]]的生成能力，解决LLM知识过时、幻觉和领域知识不足的问题。RAG是企业LLM应用最落地的技术方案。

## 核心思想

**检索→增强→生成**：先从知识库检索相关文档，再将检索结果作为上下文注入提示，最后由LLM基于上下文生成回答。

## 技术原理

### RAG基本流程

```
用户查询 → Embedding → 向量检索 → Top-K文档 → 上下文构建 → LLM生成 → 回答+引用
```

### 文档处理流水线

1. **文档加载**：PDF/Word/HTML/Markdown等格式
2. **文档切分**：
   - 固定长度切分（如512 token，重叠50）
   - 语义切分（按段落/标题）
   - 递归切分（LangChain RecursiveCharacterTextSplitter）
3. **Embedding**：将文本块编码为向量
4. **存储**：向量 + 原文 + 元数据存入向量数据库

### 向量检索

| 方法 | 原理 | 特点 |
|------|------|------|
| **余弦相似度** | $\cos(\theta) = \frac{A \cdot B}{\|A\|\|B\|}$ | 最常用 |
| **欧氏距离** | $\|A - B\|$ | 几何距离 |
| **内积** | $A \cdot B$ | 归一化后等价余弦 |

**近似最近邻（ANN）索引**：
- HNSW（Hierarchical Navigable Small World）：高召回率
- IVF（Inverted File Index）：适合大规模
- PQ（Product Quantization）：压缩存储

### 向量数据库对比

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| **Milvus** | 开源，分布式 | 大规模生产 |
| **Qdrant** | Rust实现，高性能 | 中小规模 |
| **Weaviate** | 内置混合检索 | 快速原型 |
| **Chroma** | 轻量，Python原生 | 本地开发 |
| **Pinecone** | 全托管SaaS | 无运维需求 |
| **pgvector** | PostgreSQL扩展 | 已有PG环境 |

### 进阶RAG技术

| 技术 | 原理 | 效果 |
|------|------|------|
| **查询重写** | 用LLM重写查询，提升检索效果 | 提高召回 |
| **混合检索** | 向量检索 + BM25关键词检索 | 兼顾语义和精确匹配 |
| **重排序** | 用Cross-Encoder对Top-K重新排序 | 提高精度 |
| **多路召回** | 并行多种检索策略 | 提高召回 |
| **上下文压缩** | 压缩检索结果，去除冗余 | 节省token |
| **自查询** | LLM自动生成检索元数据过滤 | 精确检索 |
| **递归检索** | 先检索摘要，再检索原文 | 处理长文档 |

### 多模态RAG

- **图文混合检索**：图像和文本统一向量化
- **表格RAG**：结构化数据的检索增强
- **多模态问答**：基于检索到的图像+文本回答

### RAG评估

| 维度 | 指标 | 说明 |
|------|------|------|
| **检索质量** | Recall@K, Precision@K | 检索准确性 |
| **生成质量** | Faithfulness, Relevance | 回答忠实度和相关性 |
| **端到端** | Answer Correctness | 最终答案正确性 |
| **工具** | RAGAS, TruLens | 自动化评估 |

## 发展演进

基础RAG（向量检索+生成）→ 进阶RAG（重排序+混合检索）→ Agentic RAG（LLM自主决定检索策略）→ 多模态RAG

## 应用领域

- **企业知识库**：内部文档问答
- **客服系统**：基于产品文档的智能客服
- **法律/医疗**：专业领域知识检索
- **代码助手**：基于代码库的编程辅助
- **教育**：基于教材的智能辅导
