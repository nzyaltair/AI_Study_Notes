# 模型服务化

## 背景与发展

将训练好的[[大语言模型]]部署为可用的在线服务，是 AI 落地的关键环节。早期模型服务化通过 Flask/FastAPI 手动封装推理逻辑，缺乏批处理和显存管理能力。随着 LLM 规模增长，专用推理引擎相继出现：vLLM 提出 PagedAttention，TGI 提供连续批处理，TensorRT-LLM 实现 NVIDIA GPU 上的最优性能。服务化架构也从单机部署演进到多副本负载均衡、弹性扩缩容和 Serverless 推理。

## 核心思想

选择合适的推理引擎，设计可水平扩展的服务架构，通过 API 网关统一管理请求路由、认证和限流，在保证低延迟和高吞吐的前提下实现模型服务的可靠运行。核心挑战在于 LLM 自回归生成的特性：每个请求的生成长度不确定，导致批处理调度与传统 Web 服务有本质差异。

## 技术原理

### 推理引擎

| 引擎 | 核心技术 | 适用场景 |
|------|----------|----------|
| **vLLM** | PagedAttention，连续批处理 | 通用 LLM 服务，高吞吐 |
| **TGI** | Rust 核心，连续批处理 | HuggingFace 生态 |
| **TensorRT-LLM** | NVIDIA 内核级优化 | 生产级低延迟 |
| **llama.cpp** | CPU/GPU 混合，GGUF 量化 | 本地/边缘部署 |
| **SGLang** | RadixAttention，结构化生成 | Agent 应用 |

### 服务架构

- **负载均衡**：Round-Robin / Least-Connections / 基于 GPU 利用率调度
- **高可用**：多副本部署 + 健康检查 + 自动故障转移
- **弹性伸缩**：基于 QPS 或 GPU 利用率触发 HPA 自动扩缩容
- **请求路由**：按模型大小或任务类型路由到不同实例

### 流式响应

LLM 生成具有自回归特性，流式响应显著降低用户感知延迟。基于 Server-Sent Events（SSE）实现：

```
data: {"token": "Hello"}
data: {"token": " world"}
data: [DONE]
```

### API 设计

OpenAI 兼容的 Chat Completions 接口已成为事实标准：

```python
POST /v1/chat/completions
{
    "model": "llama-3-70b",
    "messages": [{"role": "user", "content": "Hello"}],
    "temperature": 0.7,
    "max_tokens": 1024,
    "stream": true
}
```

## 发展演进

Flask/FastAPI 手动部署 → TF Serving/TorchServe 通用推理服务 → vLLM/TGI 专用 LLM 推理引擎 → Serverless LLM → 边缘 LLM 部署

## 关键算法·模型

- **PagedAttention**：将 [[KV-Cache]] 划分为固定大小的页（block），按需分配，减少显存碎片化带来的浪费，vLLM 实测可将吞吐提升 2-4 倍
- **连续批处理**（Continuous Batching）：与传统静态批处理不同，每个请求完成生成后立即释放资源，新请求动态加入批次，消除队头阻塞
- **多 GPU 并行**：张量并行（TP）将模型层内切分到多 GPU，流水线并行（PP）将模型层间分布到多 GPU

## 应用场景

- **LLM API 服务**：提供 OpenAI 兼容接口的公共或私有 API
- **企业私有化**：数据安全要求高的金融、医疗场景
- **Agent 平台**：[[Agent智能体|Agent]] 后端推理服务
- **多模型路由**：简单任务路由到小模型，复杂任务路由到大模型
