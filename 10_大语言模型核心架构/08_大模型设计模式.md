# 大模型设计模式

## 1. 概述

大模型设计模式是围绕LLM的架构设计、训练方法、推理优化和应用开发形成的一系列可复用的最佳实践。这些模式总结了工程实践中的经验法则，帮助开发者系统性地理解LLM技术栈全貌并快速构建高效可靠的LLM系统。

设计模式涵盖四个层面：**架构模式**（模型结构设计）、**训练模式**（预训练→微调→对齐）、**推理模式**（KV-Cache/量化/解码策略）、**应用模式**（CoT/RAG/Agent）。每一层都有经过大规模验证的技术组合。

## 2. 发展历史

| 时期 | 架构演进 | 训练演进 | 推理演进 | 应用演进 |
|------|---------|---------|---------|---------|
| 2017-2018 | Enc/Dec/Enc-Dec三分 | 全量微调 | Greedy/Beam | Prompt初探 |
| 2019-2020 | Decoder-only崛起 | 规模化预训练 | 采样策略 | Few-shot |
| 2021-2022 | MoE探索 | PEFT（LoRA） | 量化（INT8） | CoT/ReAct |
| 2023 | LLaMA技术栈统一 | SFT+RLHF/DPO | vLLM/FlashAttn | RAG/Agent |
| 2024 | MLA/SSM创新 | RL推理训练 | Speculative/FP8 | 推理模型/Agent |
| 2025 | MoE+推理融合 | 测试时计算扩展 | 极致推理优化 | 多Agent系统 |

## 3. 核心概念

### 3.1 设计模式的四个层面

1. **架构模式**：模型结构的技术选型（注意力变体、归一化、激活函数等）
2. **训练模式**：从预训练到部署的完整训练流程
3. **推理模式**：推理时的计算优化和输出策略
4. **应用模式**：基于LLM的应用架构设计

### 3.2 技术栈收敛

现代LLM已收敛到一套标准技术栈：
- **架构**：Decoder-only + RoPE + SwiGLU + RMSNorm + Pre-Norm + GQA
- **训练**：NTP预训练 → SFT → 偏好对齐（DPO/RLHF）
- **推理**：KV-Cache + FlashAttention + 连续批处理 + 量化
- **应用**：System Prompt + RAG + Tool Use + 结构化输出

## 4. 技术原理

### 4.1 架构设计模式

#### 模型架构选型

| 模式 | 特点 | 适用场景 | 代表模型 |
|------|------|---------|---------|
| Decoder-only | 自回归生成，Scaling Laws友好 | 通用LLM | GPT、LLaMA、Qwen |
| Encoder-only | 双向注意力，适合理解 | 文本理解 | BERT、RoBERTa |
| Encoder-Decoder | 编码器理解+解码器生成 | 翻译/摘要 | T5、BART |
| MoE稀疏 | 总参数大，激活参数少 | 大规模高效推理 | Mixtral、DeepSeek |

#### 组件演进

| 组件 | 演进路径 | 当前最优 | 理由 |
|------|---------|---------|------|
| 归一化 | Post-LN → Pre-LN → RMSNorm | RMSNorm | 去除均值，更快更稳 |
| 激活函数 | ReLU → GELU → SwiGLU | SwiGLU | 门控机制更强 |
| 注意力 | MHA → MQA → GQA → MLA | GQA/MLA | KV-Cache优化 |
| 位置编码 | 正弦 → 可学习 → RoPE | RoPE | 相对位置+外推 |
| 残差连接 | 标准 → Pre-Norm | Pre-Norm | 深层训练稳定 |

### 4.2 训练设计模式

#### 预训练任务

| 任务 | 描述 | 代表 |
|------|------|------|
| 自回归语言建模（NTP/Causal LM） | 预测下一个token | GPT、LLaMA |
| 掩码语言建模（MLM） | 预测被遮盖的token | BERT |
| 去噪自编码（Denoising Autoencoding） | 重建被噪声破坏的文本 | T5、BART |
| Multi-Token Prediction | 同时预测多个未来token | DeepSeek-V3 |

#### 对齐训练

| 模式 | 方法 | 特点 | 适用 |
|------|------|------|------|
| SFT | 监督微调 | 指令跟随能力，需标注数据 | 所有LLM |
| RLHF | 强化学习+人类反馈 | 对齐人类偏好，需奖励模型 | ChatGPT、Claude |
| DPO | 直接偏好优化 | 无需奖励模型，简化训练 | Zephyr、Llama 3 |
| KTO | Kahneman-Tversky优化 | 不需成对偏好数据 | 研究 |
| RLAIF | AI反馈强化学习 | 用AI代替人类标注 | Constitutional AI |
| Constitutional AI | 自我修正对齐 | AI自我批评和改进 | Claude |

#### 参数高效微调（PEFT）

| 模式 | 原理 | 可训练参数 | 适用场景 |
|------|------|-----------|---------|
| LoRA | 低秩矩阵分解 $W = W_0 + BA$ | ~0.1% | 通用微调 |
| QLoRA | 量化基座+LoRA | ~0.1%（更少显存） | 资源受限微调 |
| Adapter | 插入小型适配器层 | ~1% | 多任务共享 |
| Prefix Tuning | 训练输入前缀 | ~0.01% | 生成任务 |
| Prompt Tuning | 训练软提示 | ~0.01% | 轻量适配 |

#### 数据策略

- **数据混合配比**：不同语言/领域/任务的数据比例
- **课程学习**：从简单到困难逐步训练
- **质量过滤**：去重、去噪、质量评分
- **合成数据**：用强模型生成训练数据（如Phi系列）
- **数据消融**：验证不同数据配比对性能的影响

### 4.3 推理设计模式

#### KV-Cache

缓存已计算的Key和Value，避免自回归生成时的重复计算。

| 变体 | 原理 | 适用场景 |
|------|------|---------|
| 标准KV-Cache | 缓存所有历史KV | 通用 |
| Prefix Caching | 缓存共享前缀 | 系统提示复用 |
| PagedAttention | 分页管理KV-Cache | vLLM高吞吐 |
| KV量化 | 量化为INT8/FP8 | 减少内存 |
| KV淘汰 | 淘汰不重要KV | StreamingLLM |

#### 解码策略

| 策略 | 原理 | 适用场景 | 优缺点 |
|------|------|---------|--------|
| Greedy | 每步取最大概率token | 确定性输出 | 简单但可能重复 |
| Beam Search | 保留Top-B候选序列 | 翻译、摘要 | 质量好但计算贵 |
| Top-k采样 | 从Top-k中采样 | 创意写作 | 控制候选范围 |
| Top-p（核采样） | 累积概率≥p的token中采样 | 通用生成 | 自适应候选 |
| Temperature | 调整softmax温度 | 控制随机性 | T<1更确定，T>1更随机 |
| Min-p采样 | 设最小概率阈值 | 通用 | 兼顾多样性和质量 |
| Contrastive Decoding | 对比大小模型输出 | 提升质量 | 减少幻觉 |

#### 量化

| 精度 | 压缩比 | 方法 | 适用场景 |
|------|--------|------|---------|
| FP32 → FP16/BF16 | 2x | 直接转换 | 训练+推理 |
| FP16 → INT8 | 4x | GPTQ、LLM.int8() | 推理 |
| FP16 → INT4 | 8x | GPTQ、AWQ、GGUF | 端侧部署 |
| FP8 | 2x | 原生FP8（H100） | 训练+推理 |

#### 推理加速

| 技术 | 原理 | 加速比 |
|------|------|--------|
| 连续批处理 | 动态组batch | 2-10x吞吐 |
| Speculative Decoding | 小模型草拟+大模型验证 | 2-3x延迟 |
| 张量并行 | 跨GPU切分模型 | 线性加速 |
| 流水线并行 | 跨GPU切分层 | 线性加速 |
| 专家并行 | MoE专家跨设备 | MoE专用 |
| 前缀共享 | 共享系统提示计算 | 场景依赖 |

### 4.4 应用设计模式

#### Prompt工程

| 模式 | 描述 | 适用 |
|------|------|------|
| Zero-shot | 直接提问 | 简单任务 |
| Few-shot | 提供示例 | 格式控制、分类 |
| Chain-of-Thought | 引导逐步推理 | 复杂推理 |
| Self-Consistency | 多次采样取多数 | 数学推理 |
| ReAct | 推理+行动交替 | Agent |

#### RAG模式

```
Naive RAG → Advanced RAG → Modular RAG
```

| 阶段 | 特点 | 技术 |
|------|------|------|
| Naive RAG | 简单检索+拼接 | 向量检索+拼接 |
| Advanced RAG | 查询改写+重排序 | Query改写、HyDE、Cross-encoder重排 |
| Modular RAG | 路由+多路检索+融合 | 混合检索、多跳检索、自适应路由 |

#### Agent模式

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| ReAct Agent | 推理-行动循环 | 通用Agent |
| Plan-and-Execute | 先规划后执行 | 复杂多步任务 |
| Multi-Agent | 多智能体协作 | 角色分工任务 |
| Tool-Use Agent | 工具调用 | 需要外部能力 |
| Reflexion | 自我反思改进 | 迭代优化 |

#### 结构化输出

| 方法 | 原理 | 适用 |
|------|------|------|
| JSON Mode | 约束输出为有效JSON | 结构化数据 |
| Function Calling | 函数调用格式 | 工具使用 |
| Constrained Decoding | 语法约束解码 | 严格格式 |
| Pydantic验证 | 类型校验+重试 | 数据完整性 |

## 6. 优势与局限

### 优势

- **系统化方法论**：覆盖从架构到应用的完整技术栈
- **经验验证**：每种模式都有大规模工程验证
- **可复用性**：设计模式可跨项目复用
- **指导性**：为技术选型和工程决策提供参考

### 局限

- **快速变化**：LLM领域发展极快，模式可能很快过时
- **场景依赖**：不同场景的最优模式可能不同
- **组合复杂**：多种模式的组合效果不一定是各部分之和
- **缺乏理论**：很多模式是经验总结，理论解释不足

## 7. 应用场景

- **模型选型**：根据任务选择架构和微调策略
- **推理部署**：量化+批处理+KV-Cache组合优化

- **评估迭代**：自动评估+LLM-as-Judge+A/B测试

## 9. 前沿发展

### 9.1 训练范式演进

- **测试时计算扩展**：推理模型（o1/R1）代表新的Scaling维度
- **合成数据训练**：Phi系列证明合成数据的有效性
- **自我博弈训练**：模型通过自我对弈提升能力
- **课程学习2.0**：基于模型能力的自适应课程

### 9.2 推理优化

- **FP8推理**：H100/B200原生FP8支持
- **Speculative Decoding 2.0**：多模型协作、自适应草稿长度
- **KV-Cache压缩**：MLA+量化+淘汰的组合优化
- **编译器优化**：torch.compile、TensorRT-LLM

### 9.3 应用模式演进

- **Agentic Workflow**：从单步Agent到多步工作流
- **Multi-Agent系统**：多智能体协作解决复杂问题
- **自进化Agent**：Agent通过反馈持续改进
- **LLM OS**：LLM作为操作系统的核心调度器

### 9.4 评估与对齐

- **LLM-as-Judge**：用强模型评估弱模型
- **Constitutional AI**：AI自我对齐
- **过程评估**：评估推理过程而非仅最终结果
- **安全对齐**：红队测试+价值对齐
