# 基础RAG

## 1. 概述

基础RAG（Basic RAG / Naive RAG）是检索增强生成的最基本形式：在LLM生成回答前，先从外部知识库中检索与用户查询相关的文档片段，将其注入提示上下文，使LLM基于检索到的知识生成回答。

基础RAG是所有高级检索增强技术的基础范式，由Lewis et al.（2020）在论文"Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"中提出。理解基础RAG是掌握 `进阶RAG` 和 `多模态RAG` 的前提。

## 2. 发展历史

| 时间 | 里程碑 | 意义 |
|------|--------|------|
| 2017 | BM25成熟应用 | 传统关键词检索的黄金标准 |

| 2023 | LangChain/LlamaIndex普及 | RAG系统开发门槛大幅降低 |
| 2023 | BGE/E5嵌入模型发布 | 中文RAG性能大幅提升 |

## 3. 核心概念

### 基本流程

基础RAG包含五个核心阶段：

```
用户查询 → 查询理解 → 向量/关键词检索 → 文档重排 → 提示构造 → LLM生成 → 答案输出
```

### 关键术语

- **Chunking**：将长文档切分为适合检索的小片段
- **Embedding**：将文本映射为高维向量表示
- **Top-K检索**：返回相似度最高的K个文档
- **Context Window**：LLM一次能处理的上下文token数
- **Grounding**：将LLM输出锚定到检索到的外部知识

## 4. 技术原理

### 阶段1：文档处理与索引

**文档分块（Chunking）**：

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| 固定长度 | 按固定token数切分 | 通用，简单 |
| 句子/段落分割 | 按自然边界切分 | 保持语义完整性 |
| 递归分割 | 先大段后小段递归切分 | 层次化检索 |
| 语义分块 | 按语义相似度切分 | 高质量检索 |
| 滑动窗口 | 重叠切分，保留上下文 | 避免信息截断 |

```python
# 典型分块配置
chunk_size = 512      # 每块512 token
chunk_overlap = 50    # 相邻块重叠50 token
```

**向量化**：使用嵌入模型将每个文档块编码为向量，存入 `向量数据库` 。

### 阶段2：查询理解

对用户查询进行清洗和向量化：

- 查询清洗：去除噪声、标准化格式
- 查询向量化：使用嵌入模型编码查询

### 阶段3：检索

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 向量检索 | 基于语义相似度的向量匹配 | 语义理解强 | 计算成本高 |
| 关键词检索 | 基于TF-IDF/BM25词频匹配 | 速度快，精确匹配 | 语义理解有限 |
| 混合检索 | 结合向量和关键词检索 | 兼顾速度与精度 | 需要权重调优 |

**BM25公式**：

$$\text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t, d) \cdot (k_1 + 1)}{f(t, d) + k_1 \cdot (1 - b + b \cdot \frac{|d|}{\text{avgdl}})}$$

其中 $f(t,d)$ 为词$t$在文档$d$中的频率，$|d|$为文档长度，$\text{avgdl}$为平均文档长度，$k_1$和$b$为调节参数（典型值$k_1=1.5, b=0.75$）。

### 阶段4：文档重排

使用Cross-Encoder对初步检索结果精细化排序。Cross-Encoder将查询和文档拼接后计算相关性分数，精度高于双塔模型但速度较慢。

```
双塔模型（DPR）：query → vector, doc → vector, cosine similarity
Cross-Encoder：[query, doc] →相关性分数（更精确但更慢）
```

主流重排模型：BGE-Reranker（中文优化）、Cohere Rerank、ColBERT（延迟交互）。

### 阶段5：提示构造与生成

将检索到的文档整合到LLM提示中：

```python
RAG_PROMPT = """请基于以下上下文回答问题。如果上下文中没有相关信息，请说明"根据已知信息无法回答"。

上下文：
{context}

问题：{question}

回答："""
```

**上下文窗口管理**：按相关性分数选择最重要的文档，控制总token数不超过LLM上下文限制。

## 6. 优势与局限

### 优势

- 实现简单，开发门槛低
- 即时生效，无需训练模型
- 可溯源，支持事实核查
- 知识库可实时更新

### 局限

1. **静态检索**：检索与生成分离，缺乏动态调整
2. **缺乏反馈闭环**：无法根据生成结果优化检索
3. **上下文利用不足**：简单拼接文档，缺乏深度理解
4. **多跳推理困难**：难以处理需要多步推理的复杂问题
5. **查询-文档语义鸿沟**：用户查询与文档的嵌入表示可能存在语义不匹配

## 7. 应用场景

- 企业知识库问答系统
- 文档检索与摘要生成
- 客服系统与产品文档问答
- 法律/医疗等专业领域问答
- FAQ自动回复

## 8. 与其他技术关系

- 基础RAG是 `进阶RAG` 和 `多模态RAG` 的基础
- `向量数据库` 是RAG的存储基础
- `提示工程` 影响RAG提示构造质量
- `Agent系统` 将RAG作为知识来源
- 基础RAG的局限催生了Self-RAG、FLARE等高级范式

## 9. 前沿发展

- **从静态到动态**：基础RAG的静态检索范式正在被Self-RAG、Agentic RAG等动态检索范式取代
- **从文本到多模态**：RAG从纯文本检索扩展到图像、视频、音频等多模态检索
- **从检索到推理**：RAG不再只是检索文档，而是检索知识并进行推理整合
- **从通用到领域**：领域专属的嵌入模型和重排模型不断涌现，提升垂直领域RAG性能
- **评估标准化**：RAGAS等评估框架使RAG系统质量可量化、可比较
