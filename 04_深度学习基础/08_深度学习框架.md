# 深度学习框架

## 1. 概述

深度学习框架是连接数学理论与工程实践的桥梁。它提供自动微分、GPU加速、算子抽象、分布式训练等基础设施，使研究人员和工程师无需从零实现反向传播和底层矩阵运算，专注于模型设计和算法创新。

- **核心价值**：将"数学公式→梯度计算→参数更新"的完整流程自动化，使深度学习从学术实验工具发展为工业级生产力平台
- **框架的三重角色**：研究实验平台（快速验证想法）、训练基础设施（分布式大规模训练）、部署运行时（模型服务化）

## 2. 发展历史

| 年份 | 框架/事件 | 核心贡献 |
|:---|:---|:---|
| **2007** | Theano (Montreal大学) | 首个支持自动微分的深度学习库，开创"定义-编译-运行"范式 |
| **2013** | Caffe (UC Berkeley) | 视觉领域专用，Model Zoo概念先行；配置文件定义网络，无需写代码 |
| **2014** | Torch7 | Lua语言，灵活的自动微分，PyTorch的直系前身 |
| **2015** | TensorFlow 1.0 (Google) | 静态计算图 + 分布式训练，工业级部署能力，定义-运行分离 |
| **2016** | PyTorch (Meta/Facebook) | **动态计算图**，Pythonic API，调试友好，成为学术研究主导框架 |
| **2017** | Keras 并入 TensorFlow | 高层API降低入门门槛 |
| **2018** | TensorFlow 2.0 | 默认Eager模式，拥抱动态图，Keras作为官方API |
| **2020** | JAX (Google) | 函数式自动微分，XLA编译优化，`vmap`/`pmap`/`jit`一等公民 |
| **2022** | PyTorch 2.0 | `torch.compile`编译优化，动态图+编译加速的融合范式 |
| **2023+** | PyTorch + HuggingFace 生态 | LLM时代的绝对主导，DeepSpeed/FSDP/FlashAttention深度集成 |

**当前格局**：PyTorch 在研究界占据 >80% 的使用率，大模型生态（HuggingFace, DeepSpeed, vLLM）几乎全部建立在 PyTorch 之上；JAX 在 Google 内部和 TPU 场景中有独特优势；TensorFlow 逐步退居工业部署和边缘端。

## 3. 核心概念

### 3.1 计算图范式：静态 vs 动态

| 特性 | 静态图（TF 1.x / JAX JIT） | 动态图（PyTorch / TF Eager） |
|:---|:---|:---|
| **构建方式** | 先定义完整计算图，后执行 | 边执行边构建计算图 |
| **调试体验** | 难（图与执行分离） | **易**（标准Python调试器） |
| **控制流** | `tf.cond`/`tf.while_loop` 专属操作 | 原生Python `if`/`for`/`while` |
| **编译优化** | 图全局优化，性能优 | 需 `torch.compile` 额外编译 |
| **部署** | 图可独立导出（TF Serving） | 需 `torch.export`/ONNX 导出 |

### 3.2 Tensor与Autograd

- **Tensor**：框架的核心数据结构，多维度数组 + 梯度追踪。PyTorch中通过 `.requires_grad=True` 标记，TensorFlow中通过 `tf.GradientTape` 记录
- **Autograd**：记录所有Tensor操作，构建计算图，在 `.backward()` 时沿图逆向计算梯度（VJP, Vector-Jacobian Product）
- **nn.Module / tf.keras.layers.Layer**：可学习参数的容器，管理参数注册、设备转移、序列化

### 3.3 模型部署路径

```
训练模型（PyTorch .pt / TF SavedModel）
    ↓
导出中间格式：ONNX / TorchScript / TensorFlow SavedModel
    ↓
推理优化：TensorRT / OpenVINO / ONNX Runtime / TVM
    ↓
服务化：TorchServe / TF Serving / Triton Inference Server / vLLM
```

## 4. 技术原理

### 4.1 PyTorch的训练范式

```python
import torch
import torch.nn as nn
from torch.utils.data import DataLoader

# 1. 模型定义
class MLP(nn.Module):
    def __init__(self, in_dim, hidden_dim, out_dim):
        super().__init__()
        self.fc1 = nn.Linear(in_dim, hidden_dim)
        self.fc2 = nn.Linear(hidden_dim, out_dim)

    def forward(self, x):
        x = torch.relu(self.fc1(x))
        return self.fc2(x)

# 2. 训练配置
model = MLP(784, 256, 10)
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)
loss_fn = nn.CrossEntropyLoss()

# 3. 训练循环
for epoch in range(num_epochs):
    for x, y in dataloader:
        optimizer.zero_grad()         # 清零梯度缓存
        logits = model(x)             # 前向：构建计算图
        loss = loss_fn(logits, y)     # 计算损失
        loss.backward()               # 反向：自动微分为所有参数计算梯度
        optimizer.step()              # 更新：应用优化器规则
```

### 4.2 PyTorch关键机制

**`loss.backward()` 的内部流程**：
1. 遍历计算图（DAG），从loss节点开始拓扑逆序访问
2. 对每个操作调用其预注册的 `backward()` 函数（携带上游梯度）
3. 将计算出的局部梯度累加到对应Tensor的 `.grad` 属性
4. 释放不需要的中间结果（`retain_graph=False` 时）

**`optimizer.step()` 与 `zero_grad()`**：
- `.grad` 属性默认**累加**（不清零），便于梯度累积
- `zero_grad()` 必须手动调用，否则梯度会无限累加
- `step()` 遍历参数组，调用各参数的更新规则

### 4.3 关键框架对比

| 维度 | PyTorch | JAX | TensorFlow/Keras |
|:---|:---|:---|:---|
| **编程范式** | 命令式（动态图） | **函数式**（纯函数+变换） | 命令式（TF2 Eager） |
| **自动微分** | `loss.backward()` | `jax.grad(f)` | `tf.GradientTape` |
| **编译** | `torch.compile` (JIT) | **XLA JIT编译（原生）** | `@tf.function` (JIT) |
| **并行API** | DDP / FSDP / DTensor | **`pmap` / `pjit` (一等公民)** | `tf.distribute.Strategy` |
| **LLM生态** | **全能（HuggingFace/DeepSpeed）** | 次之（Flax/T5X/Pax） | 边缘 |
| **部署** | TorchServe / ONNX | TF Serving / TensorRT | TFLite / TF Serving |
| **学习曲线** | 平缓 | 陡峭（函数式思维） | 平缓（Keras） |
| **最佳场景** | 研究+大模型训练 | TPU+科学计算+函数式变换 | 生产部署+边缘端 |

### 4.4 分布式训练的框架支持

| 需求 | PyTorch方案 | JAX方案 |
|:---|:---|:---|
| 数据并行 | `DDP` (DistributedDataParallel) | `pmap` |
| 全分片数据并行 | `FSDP` (FullyShardedDataParallel) | `pjit` with sharding |
| ZeRO | DeepSpeed ZeRO-1/2/3 | — |
| 流水线并行 | `torch.distributed.pipelining` | `pjit` + manual stage |
| 张量并行 | Megatron-LM / `DTensor` | `pjit` with mesh |

## 5. 关键方法/模型

### 关键项目与生态

- **HuggingFace Transformers**：提供了数千个预训练模型的统一API，是LLM时代的"GitHub of Models"
- **DeepSpeed**（微软）：ZeRO优化器、混合精度、流水线并行的工业级实现
- **Megatron-LM**（NVIDIA）：张量并行+流水线并行的参考实现
- **vLLM**：高吞吐LLM推理引擎，PagedAttention管理KV Cache
- **PyTorch Lightning**：训练循环标准化，减少样板代码
- **Weights & Biases / MLflow**：实验追踪，详见 [[09_实验管理与复现]]

### PyTorch 2.0 的编译革命

```python
# PyTorch 2.0 的一行编译优化
model = torch.compile(model)  # 动态图 → 编译后的静态图

# 相比 PyTorch 1.x 的静态图方案（TorchScript）：
# - 无需修改代码，自动捕获计算图
# - 支持 Python 控制流（动态 shape / data-dependent 分支）
# - 性能提升 30%-200%（通过算子融合和内存优化）
```

## 6. 优势与局限

### PyTorch

| 优点 | 局限 |
|:---|:---|
| Pythonic，学习曲线平缓 | 动态图历史版本部署复杂 |
| 学术界绝对主导，论文代码首选 | XLA编译不如JAX原生 |
| LLM生态无敌（HuggingFace/DeepSpeed集成） | 在大规模TPU上不如JAX |
| PyTorch 2.0编译后性能接近JAX | 动态图overhead（已由compile缓解） |

### JAX

| 优点 | 局限 |
|:---|:---|
| 函数式纯净，易做数学变换 | 学习曲线陡峭（无状态、无副作用） |
| XLA编译原生支持，性能极致 | 生态不完整，无官方数据加载器 |
| `vmap`/`pmap`/`grad` 一等公民 | 调试困难（JIT编译后不可见） |
| TPU支持最佳 | 社区资源远少于PyTorch |

## 7. 应用场景

- **学术研究**：PyTorch 绝对主导（>80%论文复现使用）
- **LLM训练与推理**：PyTorch + DeepSpeed + HuggingFace + FlashAttention + vLLM → 完整技术栈
- **计算视觉**：PyTorch + torchvision / timm
- **科学计算与AI4Science**：JAX 在物理模拟、分子动力学、PDE求解中快速增长
- **边缘/移动端部署**：TensorFlow Lite / ONNX Runtime → 对模型大小和延迟敏感的场景
- **工业级推理服务**：TensorRT / Triton Inference Server / TorchServe

## 8. 与其他技术关系

- **与深度学习基础的关系**：框架将 [[01_神经网络基础]] 中的数学原语（张量、矩阵乘、激活函数）封装为高性能操作
- **与反向传播的关系**：框架的autograd引擎是 [[03_反向传播与自动微分]] 的工业级实现
- **与训练技巧的关系**：混合精度、分布式训练、梯度检查点等技术通过框架API落地（详见 [[07_训练技巧与稳定性]]）
- **与实验管理的关系**：框架的checkpoint机制与 [[09_实验管理与复现]] 中的模型注册表对接
- **与部署的关系**：ONNX是跨框架模型交换的标准格式，连接训练端（PyTorch/TF）和推理端（TensorRT/ONNX Runtime）（详见 [[18_模型部署与工程化]]）

## 9. 前沿发展

1. **PyTorch 2.x 编译生态深化**
   - `torch.compile` 仍在快速迭代，计划覆盖更多动态控制流模式
   - `torch.export` 为部署提供更严格的图导出保证

2. **JAX 生态完善**
   - Google 内部 Pax/Flax 框架持续发展
   - 更多开源项目（如 `equinox`）为 JAX 提供 `nn.Module` 风格的便利性

3. **推理框架的军备竞赛**
   - vLLM, TensorRT-LLM, llama.cpp, MLC-LLM 等在LLM推理领域激烈竞争
   - PagedAttention、Continuous Batching、Speculative Decoding 等新技术快速集成

4. **框架-硬件协同设计**
   - Triton（OpenAI）允许用户编写接近Metal/CUDA的GPU Kernel
   - PyTorch `torch.compile` 后端可接 Triton 生成高度优化的算子

5. **MLIR 中间表示的统一**
   - Google发起的MLIR编译器基础设施试图统一PyTorch/TF/JAX的中间表示
   - 长远目标是消除框架碎片化，实现"写一次，处处优化运行"
