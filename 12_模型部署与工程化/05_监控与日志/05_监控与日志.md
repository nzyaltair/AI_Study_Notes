# 监控与日志

## 背景与发展

LLM 服务上线后，持续监控性能、成本和质量是运维的核心。传统 Web 服务监控以 QPS 和延迟为主，而 LLM 的自回归生成特性引入了新的维度：首 Token 延迟（TTFT）、每 Token 生成延迟（TPOT）、Token 级计费和内容安全。随着 LangSmith、Phoenix 等 LLM 专用可观测性工具出现，监控从基础设施层延伸到模型行为层，形成覆盖请求链路、模型输出和资源利用的立体监控体系。

## 核心思想

建立多维监控体系——性能、资源、成本、质量四个维度——配合告警和自动化响应，确保 LLM 服务稳定、高效、安全运行。结构化日志作为故障排查的基础，分布式追踪串联完整请求链路，实时指标驱动告警和自动扩缩容决策。

## 技术原理

### 监控指标体系

| 维度 | 指标 | 说明 |
|------|------|------|
| **性能** | TTFT / TPOT | 首 Token 延迟 + 生成每 Token 延迟 |
| **性能** | QPS / 吞吐量 | 每秒请求数和 Token 生成速率 |
| **资源** | GPU 利用率 / 显存 | GPU 计算和内存使用情况 |
| **成本** | Token 消耗 | 输入/输出 Token 计费统计 |
| **质量** | 错误率 / 内容安全 | 请求失败比例和有害内容拦截率 |

### 监控工具链

| 层级 | 工具 | 作用 |
|------|------|------|
| **指标收集** | Prometheus | 时序数据采集与存储 |
| **可视化** | Grafana | 仪表盘展示与告警 |
| **日志** | ELK Stack | 日志收集、索引与分析 |
| **链路追踪** | Jaeger / Zipkin | 分布式请求链路追踪 |
| **LLM 专用** | LangSmith / Phoenix | LLM 调用追踪与质量评估 |
| **告警** | AlertManager | 告警规则管理与通知分发 |

### 结构化日志

LLM 服务的日志需记录请求生命周期的关键信息：

```json
{
    "timestamp": "2026-01-01T00:00:00Z",
    "request_id": "abc123",
    "model": "llama-3-70b",
    "input_tokens": 150,
    "output_tokens": 300,
    "latency_ms": 2500,
    "ttft_ms": 200,
    "status": "success",
    "user_id": "user_456"
}
```

### 分布式追踪

OpenTelemetry 提供统一的 Trace/Span 模型。一次 LLM 请求的典型追踪链：

```
Trace: api-request-123
├── Span: http-server-receive (0.1ms)
├── Span: authentication (0.5ms)
├── Span: model-router (0.3ms)
├── Span: vllm-generate (500ms)
│   ├── Span: tokenizer (10ms)
│   ├── Span: model-inference (480ms)
│   └── Span: detokenizer (10ms)
└── Span: http-server-send (0.2ms)
```

### 故障排查

- **高延迟**：检查 GPU 利用率、[[KV-Cache]] 命中率、批处理配置
- **OOM**：减小 batch size、启用量化、排查显存泄漏
- **质量下降**：排查数据漂移、模型版本变更、Prompt 模板修改
- **服务不可用**：健康检查失败、自动重启、故障转移

## 发展演进

传统监控（CPU/内存） → APM 应用性能管理 → LLM 专用监控 → 全链路可观测性 → 自愈系统

## 关键算法·模型

- **P50/P95/P99 延迟分位数**：对延迟分布的统计描述，P99 反映长尾用户体验。计算方式为将 $N$ 个请求的延迟排序后取第 $0.99N$ 个值
- **滑动窗口统计**：在时间窗口 $W$ 内维护请求计数和延迟分布，用于实时指标计算
- **异常检测**：基于统计阈值（均值 $\pm 3\sigma$）或机器学习模型识别指标异常

## 应用场景

- **LLM API 服务**：实时监控性能、成本和内容安全
- **Agent 系统**：追踪 Agent 行为链路，评估工具调用质量
- **RAG 系统**：监控检索质量和生成质量
- **企业 AI 平台**：多模型统一监控与告警

## 与其他技术关系

- 监控依赖 [[模型服务化]] 的指标暴露和日志输出
- [[Docker]] 容器提供监控数据来源（cAdvisor、kube-state-metrics）
- [[版本管理与回滚]] 通过监控指标触发自动回滚
- [[成本控制与配额]] 依赖监控数据进行成本归因和预算告警
- [[数据隐私与合规]] 要求日志中的敏感信息脱敏处理
