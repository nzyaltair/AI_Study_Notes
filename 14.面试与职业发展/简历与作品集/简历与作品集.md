# 简历与作品集

## 目标定位

本文档聚焦于如何打造一份技术扎实、亮点突出、匹配岗位需求的 AI 领域简历与作品集，强调可操作性、行业标准与差异化策略。通过系统性梳理简历与作品集构建的完整流程，为 AI 开发人员（应届生、转行者与资深工程师）提供从内容设计到呈现形式的全方位指导，帮助打造个人品牌资产并提升求职竞争力。

## 系统性梳理

### 简历核心模块设计

#### 教育背景
- **应届生**：重点突出相关课程、GPA、学术奖项、研究经历
- **转行者**：强调相关技能培训、项目经历、行业认证
- **资深工程师**：简要列出学位，重点放在工作经验和成就上

#### 技术栈
- **分类展示**：编程语言、框架库、工具平台、领域知识
- **技能层级**：可分为"精通"、"熟悉"、"了解"三个层级
- **关键词优化**：包含岗位 JD 中的核心技术词，提高 ATS 通过率

#### 项目经历
- **STAR 原则**：情境（Situation）、任务（Task）、行动（Action）、结果（Result）
- **量化成果**：使用具体数字和指标展示项目价值（如"提升推理速度 3 倍"、"准确率达到 SOTA"）
- **技术深度**：突出技术选型理由、关键算法、优化策略
- **团队协作**：说明在团队中的角色和贡献

#### 开源贡献
- **GitHub 活跃度**：提交次数、PR 数量、Star 数
- **贡献项目**：知名项目的具体贡献内容
- **自主项目**：项目描述、技术栈、使用情况

#### 论文/专利
- **论文发表**：期刊/会议名称、作者顺序、引用次数
- **专利申请**：专利类型、申请号、技术要点

### AI 岗位简历侧重点差异

#### 算法工程师
- **重点**：算法创新、模型性能、理论基础
- **展示**：论文发表、竞赛成绩、SOTA 模型实现
- **关键词**：深度学习、强化学习、计算机视觉、NLP、模型压缩

#### 大模型研究员
- **重点**：大模型训练、微调、对齐、评估
- **展示**：LLM 相关项目、预训练/微调经验、评估基准成绩
- **关键词**：Transformer、LLM、RLHF、LoRA、评估基准

#### ML 工程师
- **重点**：工程实现、部署优化、系统集成
- **展示**：MLOps 经验、模型部署方案、性能优化成果
- **关键词**：PyTorch、TensorFlow、ONNX、Docker、Kubernetes

### 作品集形式选择

#### GitHub 仓库
- **优势**：代码可审查、版本控制、展示工程能力
- **最佳实践**：`[[GitHub 作品集最佳实践]]`、清晰的 README、代码注释、CI/CD 配置
- **适用场景**：工程岗、算法岗的代码实现展示

#### 个人网站
- **优势**：个性化设计、多媒介展示、SEO 友好
- **技术选型**：React、Vue、Next.js、Hugo
- **部署方案**：Vercel、Netlify、GitHub Pages

#### Hugging Face Space
- **优势**：专门为 ML 模型设计、易于部署交互式 Demo、社区曝光
- **适用场景**：大模型微调、多模态应用、NLP 工具

#### Demo 视频
- **优势**：直观展示产品功能、用户体验
- **制作工具**：OBS Studio、ScreenFlow、剪映
- **发布平台**：YouTube、Bilibili、LinkedIn

#### 技术博客
- **优势**：展示技术深度、表达能力、持续学习
- **平台选择**：Medium、Dev.to、个人博客
- **内容建议**：项目复盘、技术原理、实践心得

### 项目描述的 STAR 原则与量化成果

#### STAR 原则应用
- **情境（Situation）**：项目背景和挑战
- **任务（Task）**：你的具体职责和目标
- **行动（Action）**：采取的技术方案和步骤
- **结果（Result）**：取得的具体成果和影响

#### 量化成果表达
- **性能指标**：准确率、F1 分数、推理速度、显存占用
- **业务指标**：用户增长、收入提升、成本降低
- **技术指标**：代码行数、API 调用延迟、模型大小

### 技术关键词优化与常见错误

#### ATS 友好性
- **关键词布局**：在技能栏、项目描述中自然融入关键词
- **格式规范**：避免使用表格、特殊字符，使用标准字体
- **内容完整**：确保关键词有足够的上下文支撑

#### 常见错误
- **堆砌术语**：无意义的技术名词罗列
- **缺乏上下文**：项目描述过于简略，缺乏背景信息
- **虚假信息**：夸大成果，虚构经验
- **格式混乱**：排版不规范，字体不一致

## 实用性强化

### 可复用模板片段

#### 项目描述范例
```markdown
**大语言模型微调与部署**
- **情境**：企业内部知识库问答系统准确率低，响应速度慢
- **任务**：设计并实现基于 LLaMA-2 的领域微调模型，优化推理性能
- **行动**：
  - 使用 PEFT/LoRA 进行参数高效微调
  - 构建 RAG 系统增强事实性
  - 部署 vLLM 提升推理速度
- **结果**：
  - 问答准确率提升至 85%（+30%）
  - 推理延迟降低至 200ms（-75%）
  - 支持 100+ 并发请求
```

#### 技能栏写法
```markdown
**技术栈**
- **编程语言**：Python (NumPy, Pandas, PyTorch), Java, C++
- **深度学习**：Transformer, LLM, RLHF, LoRA, RAG
- **部署工具**：Docker, Kubernetes, AWS SageMaker, vLLM
- **开发工具**：Git, GitHub Actions, Jupyter, VS Code
- **领域知识**：NLP, 计算机视觉, 推荐系统
```

### 优秀 vs 普通简历案例

#### 优秀简历片段
```markdown
**项目经历**
- **大模型 RAG 系统**：设计并实现基于 LangChain 的 RAG 系统，整合向量数据库和知识图谱，提升问答准确率 40%，支持多文档检索和实时更新。
- **模型压缩**：使用知识蒸馏和量化技术，将 BERT 模型大小减少 75%，推理速度提升 3 倍，在移动设备上实现实时 NLP 推理。
```

#### 普通简历片段
```markdown
**项目经历**
- 参与大模型项目，负责部分功能开发。
- 做过模型压缩相关工作。
```

#### 关键改进点
- **量化成果**：从"参与"到"提升准确率 40%"的具体数据
- **技术细节**：从"部分功能"到"基于 LangChain 的 RAG 系统"的技术深度
- **问题解决**：从描述工作到展示解决方案和效果

#### 大模型/AI 热门方向展示建议

##### LLM 微调
- **展示形式**：Hugging Face Space + GitHub 仓库
- **重点**：微调方法、数据集、评估结果、推理优化
- **Demo**：提供对话界面，展示模型能力

##### RAG 系统
- **展示形式**：交互式 Demo + 技术博客
- **重点**：检索策略、知识库构建、上下文管理、评估指标
- **Demo**：上传文档，进行问答测试

##### 多模态应用
- **展示形式**：视频 Demo + GitHub 仓库
- **重点**：模态融合方法、跨模态对齐、应用场景
- **Demo**：图文生成、视频理解等功能展示

## 前瞻性探讨

### AI 时代简历新趋势

#### 动态作品集
- **可交互模型 Demo**：让招聘方直接体验模型能力
- **实时数据**：展示模型在真实场景中的表现
- **个性化内容**：根据岗位需求动态调整展示内容

#### AI 辅助简历生成
- **风险**：内容同质化、信息不准确、缺乏个人特色
- **边界**：使用 AI 辅助但保持个人风格，确保信息真实性
- **最佳实践**：用 AI 生成初稿，人工优化细节和个性化内容

#### GitHub 即简历（Code as CV）
- **理念**：代码质量、项目结构、提交历史反映工程能力
- **实践**：维护高质量 GitHub 个人资料，README 清晰，代码规范
- **优势**：直接展示编程能力，避免简历与实际能力不符

### 复合展示策略

#### 工程能力
- **代码质量**：GitHub 仓库的代码规范、测试覆盖率
- **系统设计**：架构图、技术选型文档
- **部署经验**：CI/CD 配置、容器化部署方案

#### 科研思维
- **问题定义**：清晰的问题描述和目标设定
- **方法论**：实验设计、评估指标、结果分析
- **创新点**：技术创新、算法改进、性能突破

#### 产品意识
- **用户需求**：从用户角度出发的功能设计
- **体验优化**：界面设计、交互流程、响应速度
- **商业价值**：成本降低、效率提升、收入增长
