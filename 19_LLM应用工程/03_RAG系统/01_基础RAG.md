# 基础RAG

## 背景与发展

检索增强生成（RAG）由Lewis et al.（2020）在论文"Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks"中提出。其核心思想是通过从外部知识库中检索相关信息来增强[[大语言模型]]的生成能力，解决LLM的知识时效性、事实准确性和可追溯性问题。基础RAG是所有高级检索增强技术的基础范式。

## 核心思想

RAG = 检索器 + 生成器。在LLM生成回答前，先从外部知识库中检索与查询相关的文档片段，注入提示上下文，使LLM基于检索知识生成更准确的回答。

## 技术原理

### 基础RAG流程

```
用户查询 → 查询理解 → 向量/关键词检索 → 文档重排 → 提示构造 → LLM生成 → 答案输出
```

### 阶段1：查询理解

对用户查询进行清洗、扩展和重写，生成优化后的查询表示：

- 查询清洗：去除噪声、标准化
- 查询扩展：添加同义词、相关术语
- 查询重写：转换为更适合检索的形式
- 查询向量化：使用嵌入模型编码

### 阶段2：检索

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 向量检索 | 基于语义相似度的向量匹配 | 语义理解能力强 | 计算成本较高 |
| 关键词检索 | 基于TF-IDF/BM25词频匹配 | 速度快，实现简单 | 语义理解有限 |
| 混合检索 | 结合向量和关键词检索 | 兼顾速度与精度 | 需要权重调优 |

### 阶段3：文档重排

使用Cross-Encoder对初步检索结果精细化排序。Cross-Encoder将查询和文档拼接后计算相关性分数，精度高于双塔模型但速度较慢。

主流重排模型：BGE-Reranker（中文优化）、Cohere Rerank、ColBERT（延迟交互）。

### 阶段4：提示构造

将检索到的文档整合到LLM提示中：

```python
RAG_PROMPT = """请基于以下上下文回答问题。如果上下文不足，请说明。

上下文：
{context}

问题：{question}
"""
```

上下文窗口管理：按相关性分数选择最重要的文档，控制总token数不超过LLM上下文限制。

### 阶段5：生成

LLM基于构造的提示生成最终答案，可添加来源引用和置信度评估。

### 嵌入模型对比

| 模型 | 维度 | 特点 | 适用场景 |

### 主流框架实现

| 框架 | 优点 | 适用场景 |
|------|------|----------|
| LangChain | 生态丰富，组件齐全 | 复杂应用，高度定制 |
| LlamaIndex | 文档处理能力强 | 文档密集型应用 |
| Haystack | 管道设计清晰 | 快速原型，研究 |

## 发展演进

基础RAG（2020）→ 混合检索（2023）→ 进阶RAG（查询重写/HyDE/多向量检索）→ Self-RAG/CRAG（2024）→ Agentic RAG（2024）

### 基础RAG的局限

1. **静态检索**：检索与生成分离，缺乏动态调整
2. **缺乏反馈闭环**：无法根据生成结果优化检索
3. **上下文利用不足**：简单拼接文档，缺乏深度理解
4. **多跳推理困难**：难以处理需要多步推理的复杂问题

## 关键算法·模型

- **DPR**（Dense Passage Retrieval）：双塔密集检索器（Karpukhin et al., 2020）
- **BGE**：中文优化的嵌入模型
- **Cross-Encoder**：BERT-based查询-文档对精排
- **BM25**：基于词频和逆文档频率的关键词检索
- **HNSW**：分层可导航小世界图ANN索引

## 应用场景

- 企业知识库问答系统
- 文档检索与摘要生成
- 客服系统与产品文档问答
- 法律/医疗等专业领域问答
