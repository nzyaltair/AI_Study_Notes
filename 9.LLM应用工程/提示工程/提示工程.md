# 提示工程

## 1. 概述

### 1.1 目标定位
本文聚焦于**提示工程（Prompt Engineering）在真实 LLM 应用场景中的系统方法论**，内容超越基础技巧，深入策略设计、评估机制与工程集成，旨在帮助AI开发人员与应用工程师构建高效、可靠、可扩展的提示系统。

### 1.2 核心价值
- 充分发挥LLM能力，提升应用性能和用户体验
- 降低模型依赖，增强跨模型兼容性
- 提高开发效率，加速LLM应用落地
- 增强系统可解释性和可控性
- 降低幻觉风险，提升输出可靠性

## 2. 基础概念与方法

### 2.1 零样本提示（Zero-shot Prompting）
- **定义**：直接向模型提问，不提供任何示例
- **适用场景**：简单任务、模型已训练过的常见任务
- **示例**：
  ```
  请总结以下文本的主要内容：
  提示工程是构建LLM应用的关键技术，涉及设计、优化和评估提示词，以引导模型生成高质量输出。
  ```
- **优缺点**：
  - 优点：简单易用、无需示例数据
  - 缺点：复杂任务效果不佳、缺乏上下文指导

### 2.2 少样本提示（Few-shot Prompting）
- **定义**：提供少量示例，引导模型学习任务模式
- **适用场景**：复杂任务、特定领域任务
- **示例**：
  ```
  请将以下文本分类为正面、负面或中性：
  
  示例1：
  文本：这款产品非常好用，我很喜欢！
  分类：正面
  
  示例2：
  文本：这个服务一般，没有特别的亮点。
  分类：中性
  
  待分类：
  文本：界面设计很差，操作也不流畅。
  分类：
  ```
- **最佳实践**：
  - 示例数量：3-5个（过多会增加token消耗）
  - 示例质量：确保示例准确、多样化
  - 格式一致性：保持示例格式统一

### 2.3 思维链提示（Chain-of-Thought, CoT）^思维链
- **定义**：引导模型逐步思考，展示推理过程
- **适用场景**：数学推理、逻辑问题、复杂决策
- **示例**：
  ```
  问题：如果一个商店有100个苹果，卖出30个，然后又进货50个，现在有多少个苹果？
  
  让我们逐步思考：
  1. 初始苹果数量：100个
  2. 卖出30个后：100 - 30 = 70个
  3. 进货50个后：70 + 50 = 120个
  
  答案：120个
  ```
- **变体**：
  - 标准CoT：提供完整推理示例
  - 零样本CoT：在提示中加入"让我们逐步思考"等引导语

### 2.4 自洽性（Self-Consistency）
- **定义**：生成多个推理路径，通过投票选择最一致的答案
- **适用场景**：需要高准确性的复杂推理任务
- **示例流程**：
  1. 对同一问题生成多个不同的推理路径
  2. 收集所有推理结果
  3. 统计出现次数最多的答案作为最终结果
- **优势**：
  - 提高推理准确性
  - 降低单一推理路径的偶然性
  - 提供置信度评估

## 3. 高级提示模式

### 3.1 ReAct^ReAct
- **定义**：结合推理（Reasoning）和行动（Action）的提示框架
- **核心思想**：让模型先思考，然后执行行动，再根据结果调整策略
- **适用场景**：需要外部工具调用、多步骤任务
- **示例格式**：
  ```
  你是一个助手，需要完成用户的请求。请按照以下格式回答：
  
  思考：[你的推理过程]
  行动：[调用的工具和参数]
  结果：[工具返回的结果]
  
  用户请求：请查询2026年1月26日北京的天气，然后告诉我是否适合户外活动。
  
  思考：我需要先查询北京的天气，然后根据天气情况判断是否适合户外活动。
  行动：查询天气(城市="北京", 日期="2026-01-26")
  ```
- **工具集成**：可与搜索、计算器、数据库等外部工具结合

### 3.2 Self-Ask
- **定义**：引导模型通过自问自答的方式解决复杂问题
- **核心思想**：将大问题分解为多个子问题，逐步求解
- **适用场景**：需要多步骤推理、信息检索的任务
- **示例**：
  ```
  你是一个助手，需要通过自问自答的方式解决问题。
  
  问题：爱因斯坦出生于哪一年？
  
  回答：我需要知道爱因斯坦的出生日期。爱因斯坦出生于1879年3月14日。
  ```

### 3.3 Tree-of-Thoughts (ToT)
- **定义**：构建思考树，探索多种可能的推理路径
- **核心思想**：类似人类的发散思维，考虑多种可能性，然后评估筛选
- **适用场景**：需要探索多种可能性的复杂任务
- **示例流程**：
  1. 问题分解：将问题分解为多个子问题
  2. 并行思考：为每个子问题生成多种可能的解决方案
  3. 评估筛选：评估每种方案的可行性，筛选出最优路径
  4. 组合求解：将最优子方案组合成最终答案

### 3.4 Program-Aided Language Models (PAL)
- **定义**：结合自然语言和编程语言，利用代码执行辅助推理
- **核心思想**：让模型生成代码来解决问题，然后执行代码得到结果
- **适用场景**：数学计算、数据分析、逻辑推理等任务
- **示例**：
  ```
  问题：计算1到100的和。
  
  请生成Python代码来解决这个问题：
  
  sum = 0
  for i in range(1, 101):
      sum += i
  print(sum)
  ```
- **优势**：
  - 提高计算准确性
  - 降低模型幻觉
  - 适合复杂计算任务

## 4. 动态提示技术

### 4.1 上下文学习（In-Context Learning, ICL）
- **定义**：模型从上下文中的示例中学习，无需参数更新
- **核心思想**：通过精心设计的示例，引导模型理解任务要求
- **优化策略**：
  - 示例选择：选择最相关、最多样化的示例
  - 示例排序：按难度递增或相关度排序
  - 示例数量：平衡效果和token消耗

### 4.2 提示模板自动优化
- **定义**：使用算法自动优化提示模板，提升模型性能
- **方法**：
  - 基于规则的优化：根据模型特性调整提示结构
  - 基于搜索的优化：使用遗传算法、贝叶斯优化等搜索最优提示
  - 基于模型的优化：使用另一个模型生成或优化提示
- **工具**：
  - PromptTools：OpenAI开发的提示测试和优化工具
  - LangChain PromptTemplate：支持动态模板生成
  - DSPy^DSPy：用于自动优化提示和程序

### 4.3 提示集成
- **定义**：将提示与其他系统组件集成，实现动态调整
- **实现方式**：
  - 基于用户输入动态生成提示
  - 结合上下文信息（如对话历史、用户偏好）调整提示
  - 根据模型反馈优化提示
- **示例**：
  ```python
  from langchain import PromptTemplate
  
  template = """你是一个专业的{field}助手，用户是{role}。
  请根据用户的问题提供专业、准确的回答：
  
  用户问题：{question}
  回答："""
  
  prompt_template = PromptTemplate(
      input_variables=["field", "role", "question"],
      template=template
  )
  
  # 动态生成提示
  prompt = prompt_template.format(
      field="金融",
      role="初学者",
      question="什么是股票？"
  )
  ```

## 5. 安全与鲁棒性

### 5.1 对抗提示与防御
- **对抗提示**：精心设计的提示，旨在绕过模型的安全机制
- **防御策略**：
  - 输入过滤：检测和阻止恶意提示
  - 输出约束：限制模型输出范围
  - 强化学习：通过RLHF增强模型的安全对齐
  - 多轮验证：对模型输出进行多轮检查

### 5.2 越狱防御
- **越狱攻击**：通过欺骗性提示，让模型生成不当内容
- **防御方法**：
  - 强化系统提示：明确禁止不当内容生成
  - 输出检测：使用分类模型检测不当输出
  - 上下文限制：限制模型的知识范围
  - 审计日志：记录所有提示和输出，便于事后分析

### 5.3 输出约束
- **定义**：限制模型输出的格式、内容和范围
- **实现方式**：
  - 格式约束：要求模型按照特定格式输出（如JSON、XML）
  - 内容约束：限制模型输出的主题、语言风格
  - 长度约束：限制输出的token数量
- **示例**：
  ```
  请按照以下JSON格式输出结果：
  {
    "answer": "[你的回答]",
    "confidence": "[高/中/低]",
    "source": "[信息来源]"
  }
  
  问题：什么是人工智能？
  ```

## 6. 工具链集成

### 6.1 LangChain集成
- **核心优势**：提供丰富的提示模板库和集成能力
- **关键组件**：
  - PromptTemplate：用于创建和管理提示模板
  - FewShotPromptTemplate：支持少样本提示
  - ChatPromptTemplate：用于对话场景
  - OutputParser：用于解析模型输出
- **示例**：
  ```python
  from langchain.prompts import ChatPromptTemplate, SystemMessagePromptTemplate, HumanMessagePromptTemplate
  
  # 系统提示
  system_template = "你是一个专业的{field}顾问，提供准确、专业的建议。"
  system_message_prompt = SystemMessagePromptTemplate.from_template(system_template)
  
  # 人类提示
  human_template = "请解释{topic}的基本概念。"
  human_message_prompt = HumanMessagePromptTemplate.from_template(human_template)
  
  # 组合提示
  chat_prompt_template = ChatPromptTemplate.from_messages([
      system_message_prompt,
      human_message_prompt
  ])
  
  # 生成提示
  prompt = chat_prompt_template.format_prompt(
      field="金融",
      topic="区块链"
  ).to_messages()
  ```

### 6.2 LlamaIndex集成
- **核心优势**：专注于检索增强生成（RAG）场景
- **关键组件**：
  - PromptHelper：用于生成和优化提示
  - PromptTemplate：支持自定义提示模板
  - ResponseSynthesizer：结合检索结果生成响应
- **示例**：
  ```python
  from llama_index import PromptHelper
  
  # 创建PromptHelper
  prompt_helper = PromptHelper(
      context_window=4096,
      num_output=256,
      chunk_overlap_ratio=0.1
  )
  
  # 生成提示
  prompt = prompt_helper.get_prompt_for_query(context_str, query_str)
  ```

### 6.3 DSPy集成^DSPy
- **核心优势**：自动优化提示和程序，提升模型性能
- **关键组件**：
  - Signature：定义任务的输入输出格式
  - Module：封装提示逻辑
  - Optimizer：自动优化提示和程序
- **示例**：
  ```python
  import dspy
  
  # 定义任务签名
  class QuestionAnswering(dspy.Signature):
      """回答用户的问题。"""
      question = dspy.InputField()
      answer = dspy.OutputField()
  
  # 创建模块
  class SimpleQA(dspy.Module):
      def forward(self, question):
          return dspy.Predict(QuestionAnswering)(question=question)
  
  # 初始化模型
  lm = dspy.OpenAI(model="gpt-4")
  dspy.settings.configure(lm=lm)
  
  # 使用模块
  qa = SimpleQA()
  result = qa(question="什么是机器学习？")
  ```

## 7. 工程实践与评估

### 7.1 提示版本管理
- **必要性**：跟踪提示的演变，支持回滚和对比
- **实现方式**：
  - 代码化管理：将提示模板存储在代码仓库中
  - 版本标签：为每个提示版本添加唯一标签
  - 变更日志：记录每次提示变更的原因和影响
- **最佳实践**：
  ```
  # 提示版本示例
  prompt_v1.0: 基础版本，支持零样本提示
  prompt_v1.1: 增加思维链提示，提高推理准确性
  prompt_v2.0: 升级为ReAct模式，支持工具调用
  ```

### 7.2 A/B测试
- **定义**：对比不同提示版本的性能差异
- **关键指标**：
  - 准确性：回答的正确性
  - 延迟：从输入到输出的时间
  - token消耗：生成答案所使用的token数量
  - 用户满意度：用户对输出的评分
- **实现流程**：
  1. 设计测试方案：确定测试目标、指标和样本量
  2. 分配流量：将用户流量分配到不同的提示版本
  3. 收集数据：记录每个版本的性能指标
  4. 分析结果：比较不同版本的性能，选择最优版本

### 7.3 监控指标
- **提示相关指标**：
  - 提示长度：平均提示token数
  - 模板使用率：各提示模板的使用比例
  - 提示成功率：成功生成有效输出的提示比例
- **模型输出指标**：
  - 输出准确性：正确回答的比例
  - 输出多样性：生成内容的多样性
  - 幻觉率：生成内容与事实不符的比例
  - 格式符合率：按要求格式输出的比例
- **系统指标**：
  - 延迟：端到端延迟
  - 吞吐量：每秒处理的请求数
  - 错误率：请求失败的比例

### 7.4 常见陷阱与缓解手段

| 陷阱 | 描述 | 缓解手段 |
|------|------|----------|
| **格式漂移** | 模型输出偏离预期格式 | 使用明确的格式约束、输出解析器、多轮验证 |
| **幻觉放大** | 模型生成的错误信息被后续提示强化 | 引入外部知识源、增加事实核查、限制生成长度 |
| **上下文过载** | 提示过长导致模型忽略重要信息 | 优化提示结构、使用检索增强、分割长提示 |
| **过拟合特定模型** | 提示仅在特定模型上表现良好 | 设计通用提示结构、测试多种模型、避免模型特定语法 |
| **性能退化** | 模型性能随时间下降 | 定期评估提示效果、更新提示模板、监控模型输出 |

## 8. 前瞻性探讨

### 8.1 模型能力演进下的角色变迁
- **当前状态**：提示工程是LLM应用的核心技术之一
- **未来趋势**：
  - 内置推理模块：模型可能内置更强大的推理能力，减少对显式提示的依赖
  - 自动提示优化：工具将自动生成和优化提示，降低人工干预
  - 模型即服务：MaaS平台可能提供预优化的提示模板
- **长期影响**：
  - 提示工程可能从"艺术"转变为"科学"，有更系统的方法论
  - 开发者可能更关注提示的业务逻辑，而非底层技术细节

### 8.2 2024–2026年新兴方向

#### 8.2.1 编译式提示
- **定义**：将自然语言提示编译为更高效的模型输入格式
- **优势**：
  - 提高模型推理效率
  - 减少token消耗
  - 增强提示的可读性和可维护性

#### 8.2.2 神经符号提示
- **定义**：结合神经网络和符号推理的提示技术
- **优势**：
  - 提高推理准确性
  - 增强可解释性
  - 支持复杂逻辑推理

#### 8.2.3 多模态提示模板
- **定义**：支持文本、图像、音频等多种模态的提示模板
- **应用场景**：
  - 图像描述生成
  - 视频内容分析
  - 多模态对话系统
- **示例**：
  ```
  请描述以下图像的内容，然后生成一段相关的故事。
  [图像]
  ```

#### 8.2.4 自适应提示
- **定义**：根据模型反馈和用户需求动态调整提示
- **优势**：
  - 适应不同用户的需求
  - 自动优化提示效果
  - 提高用户满意度
