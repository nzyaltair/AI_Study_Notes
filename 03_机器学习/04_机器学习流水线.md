# 机器学习流水线

## 背景与发展

机器学习流水线（ML Pipeline）是将 ML 项目的各个阶段——从数据收集到模型部署和监控——自动化和标准化的流程。它解决了传统 ML 开发中手动操作多、实验难复现、部署不规范的问题，是工程化思维在[[机器学习]]领域的应用。

## 核心思想

将数据处理、[[特征工程]]、模型训练、评估和部署等步骤串联为模块化的工作流，确保整个流程可复现、可测试、可维护。每个阶段有明确的输入输出和质量标准。

## 技术原理

### 流水线典型架构

```
数据收集 → 数据预处理 → 特征工程 → 模型训练与评估 → 模型部署 → 模型监控
```

### 各阶段职责

| 阶段 | 职责 | 关键工具 |
|------|------|---------|
| **数据收集** | 数据获取、质量检查、版本控制 | DVC、Git LFS |
| **数据预处理** | 清洗缺失值、异常值、类型转换、数据合并 | Pandas、Scikit-learn |
| **特征工程** | 特征提取、选择、构造、转换 | Featuretools、Scikit-learn |
| **模型训练** | 超参调优、交叉验证、模型选择 | Optuna、MLflow |
| **模型评估** | 指标计算、模型解释 | SHAP、LIME |
| **模型部署** | API 服务、容器化、灰度发布 | Docker、FastAPI、KServe |
| **模型监控** | 数据漂移检测、性能监控 | Prometheus、Evidently AI |

### 数据漂移与概念漂移

- **数据漂移（Data Drift）**：输入数据分布 $P(X)$ 变化，导致模型性能下降。检测方法包括 KS 检验、AD 检验；应对策略为重新训练模型
- **概念漂移（Concept Drift）**：特征与目标关系 $P(Y|X)$ 变化。通过监控性能指标变化检测；应对策略为更新训练数据、重新定义问题

### 流水线自动化框架

| 框架 | 特点 | 适用场景 |
|------|------|---------|
| **Airflow** | DAG 定义工作流，生态丰富 | ETL、通用流水线 |
| **Kubeflow** | 基于 K8s，端到端 ML | 大规模分布式训练 |
| **MLflow** | 实验跟踪 + 模型管理 + 部署 | 实验管理、模型版本控制 |
| **Metaflow** | Netflix 开源，原型到生产 | 数据科学原型开发 |
| **Prefect** | 现代工作流管理 | 数据工程、ML 流水线 |

### MLOps 核心原则

- **自动化**：数据处理 $\to$ 训练 $\to$ 评估 $\to$ 部署全流程自动化
- **可重复性**：确保实验和模型训练可复现
- **版本控制**：代码、数据、模型三重版本管理
- **持续监控**：性能指标、数据分布、业务指标全面监控
- **CI/CD**：模型持续集成与持续部署

## 发展演进

手动执行 $\to$ 脚本化 $\to$ 框架化（Airflow/Kubeflow）$\to$ 平台化 $\to$ AutoML 自动化。从传统[[机器学习]]的手动流程，发展到 MLOps 体系化的工程实践，再到 LLMOps 针对大模型的特殊运维需求。

## 关键算法·模型

- **CI/CD 流水线**：代码提交触发自动训练、评估、部署
- **数据版本控制**：DVC 管理数据集版本，保证可复现性
- **模型注册表**：MLflow Model Registry 管理模型版本和阶段

## 应用场景

- **企业 ML 系统**：金融风控、推荐系统、广告投放
- **深度学习项目**：模型训练 $\to$ 部署 $\to$ 监控全生命周期
- **LLM 应用**：LLMOps——模型压缩、量化、安全对齐、推理优化

## 与其他技术关系

- 流水线依赖软件工程的模块化设计和 CI/CD 实践
- [[特征工程]]是流水线的核心环节
- [[模型选择与调参]]与评估嵌入流水线的训练阶段
- LLMOps 是 MLOps 在大模型时代的延伸
