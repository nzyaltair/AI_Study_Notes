# 结构化输出

## 1. 概述

结构化输出（Structured Output）是使大语言模型输出严格符合预定义格式（如JSON Schema、XML、函数参数）的技术方法。它是连接LLM自由文本生成与确定性编程系统的关键桥梁。

LLM天然倾向于生成自由文本，而实际应用需要可编程的结构化数据——API调用参数、数据库写入、函数输入等。结构化输出解决了"自由文本→可编程数据"的可靠转换问题，是 `工具调用` 和 `Agent系统` 的基础设施。

## 3. 核心概念

### 方法分类

| 方法类型 | 实现层面 | 核心机制 | 可靠性 |
|----------|----------|----------|--------|
| 提示工程 | 输入侧 | 在提示中描述格式要求 | 低（~70%） |
| JSON Mode | 模型层 | 模型强制输出合法JSON | 中高（~90%） |
| Function Calling | 模型层 | 模型原生工具调用参数生成 | 高（~95%） |
| Schema约束 | 模型层 | response_format指定Schema | 高（~95%） |
| Constrained Decoding | 解码层 | 生成时实时过滤非法token | 极高（~100%） |
| Grammar-based Decoding | 解码层 | CFG/BNF文法约束 | 极高（~100%） |

### 关键术语

- **JSON Schema**：描述JSON数据结构的规范，定义字段类型、必填项、取值范围
- **Constrained Decoding**：在token采样时实时过滤不符合格式约束的token
- **CFG（Context-Free Grammar）**：上下文无关文法，用于定义合法的输出语法
- **BNF（Backus-Naur Form）**：巴科斯范式，描述语言语法的元语言
- **Logits Processor**：在logits层面干预token采样的组件

## 4. 技术原理

### 提示工程方法

在提示中嵌入格式说明或Schema示例：

```python
schema = {
    "type": "object",
    "properties": {
        "product_name": {"type": "string"},
        "price": {"type": "number"},
        "in_stock": {"type": "boolean"}
    },
    "required": ["product_name", "price"]
}
prompt = f"请按以下JSON Schema输出产品信息：\n{json.dumps(schema, indent=2)}"
```

局限：LLM可能忽略格式要求、字段类型错误、生成多余文本。

### 模型原生支持

**OpenAI Structured Output**：

```python
from pydantic import BaseModel

class Product(BaseModel):
    name: str
    price: float
    in_stock: bool

response = client.beta.chat.completions.parse(
    model="gpt-4o-2024-08-06",
    messages=[{"role": "user", "content": "提取产品信息"}],
    response_format=Product,  # 直接传递Pydantic模型
)
product = response.choices[0].message.parsed  # 类型安全的解析结果
```

**OpenAI Strict Mode**：启用 `strict: true` 后，模型保证输出100%符合Schema。

### Constrained Decoding（解码层约束）

核心原理：在每一步token生成时，根据当前已生成文本和格式规则，计算合法的token集合，将非法token的logits设为 $-\infty$。

```
生成状态：{"name": "iPhone", "price":|
  → 合法token：数字、小数点
  → 非法token（设为-∞）：字母、特殊字符

生成状态：{"name": "iPhone", "price": 999.|
  → 合法token：数字、}
  → 非法token：字母（因为price是number类型）
```

**实现原理**：

$$\text{logits}'_i = \begin{cases} \text{logits}_i & \text{if } token_i \in \text{ValidTokens(state)} \\ -\infty & \text{otherwise} \end{cases}$$

### Grammar-based Decoding

通过定义形式文法（CFG/BNF）约束模型生成的token序列，确保输出符合特定语法规则：

```ebnf
# BNF语法定义JSON子集
root        ::= object
object      ::= "{" ws (pair (ws "," ws pair)*)? ws "}"
pair        ::= string ws ":" ws value
value       ::= string | number | object | "true" | "false" | "null"
string      ::= "\"" ([^"\\] | "\\" .)* "\""
number      ::= [0-9]+ ("." [0-9]+)?
ws          ::= [ \t\n]*
```

Llama.cpp、vLLM等推理引擎原生支持grammar约束，直接将文法编译为状态机，在解码时实时验证。

### 后处理与校验

当模型不支持原生结构化输出时的兜底方案：

- **正则提取**：用正则表达式提取JSON部分并尝试解析，失败时尝试修复
- **LLM-as-a-Judge**：使用另一个LLM验证和修正输出
- **Pydantic校验**：用Pydantic模型验证类型和约束，失败时重试

## 5. 关键方法与模型

### 主流工具对比

| 工具 | 核心机制 | 性能 | 适用场景 |
|------|----------|------|---------|
| OpenAI Structured Output | 模型层Schema约束 | 高 | OpenAI模型应用 |
| Outlines | Constrained Decoding | 极高 | 开源模型，生产环境 |
| Guidance（微软） | 条件生成 + 格式约束 | 极高 | 复杂条件生成 |
| lm-format-enforcer | 轻量级格式约束 | 极高 | 高性能场景 |
| vLLM Logits Processor | 推理引擎原生支持 | 极高 | 大规模部署 |
| Llama.cpp Grammar | BNF文法约束 | 极高 | 本地部署 |
| Instructor | Pydantic + 重试 | 中高 | 快速原型 |

### 性能权衡

| 维度 | 提示工程 | JSON Mode | Constrained Decoding |
|------|---------|-----------|---------------------|
| 格式可靠性 | ~70% | ~90% | ~100% |
| 输出质量 | 高 | 中高 | 可能降低（强制约束影响生成质量） |
| 延迟开销 | 无 | 低 | 中（每步计算合法token） |
| 模型兼容性 | 所有模型 | 支持的模型 | 需要推理引擎支持 |

## 6. 优势与局限

### 优势

- **可靠性**：Constrained Decoding可达100%格式合规
- **可编程性**：LLM输出可直接被程序消费，无需解析
- **类型安全**：Schema约束保证字段类型和结构正确
- **生态集成**：与API调用、数据库写入、函数调用无缝对接

### 局限

- **质量权衡**：严格约束可能影响输出内容质量（模型被迫选择非最优token）
- **延迟开销**：Constrained Decoding的每步验证增加推理延迟
- **模型限制**：并非所有模型/推理引擎都支持高级约束
- **复杂Schema**：深层嵌套或复杂约束可能超出模型理解能力
- **幻觉内容**：格式正确但内容可能错误（结构化≠准确）

## 7. 应用场景

- **API调用**：LLM生成Function Calling参数
- `RAG`结果解析：从检索文档中提取结构化信息
- `Agent`工具调用：Agent生成工具调用参数
- **数据标注**：自动从文本中提取结构化标注
- **数据库写入**：LLM输出直接作为数据库记录
- **表单填充**：从自然语言提取表单字段
- **信息抽取**：NER、关系抽取、事件抽取

## 8. 与其他技术关系

- 结构化输出是 `工具调用与MCP协议` 的基础，工具调用参数依赖结构化输出
- `提示工程` 影响结构化输出的内容质量
- `Agent系统` 依赖结构化输出实现工具调用和数据传递
- `RAG` 利用结构化输出解析检索结果和生成结构化回答
- 推理引擎的 `解码与采样策略` 直接影响结构化输出的实现方式

## 9. 前沿发展

- **原生结构化输出模型**：模型训练阶段即引入Schema遵循能力，无需解码层干预
- **Strict Mode普及**：主流API厂商普遍支持100%格式合规的严格模式
- **多模态结构化输出**：从纯文本扩展到图文混合的结构化输出
- **性能优化**：通过编译文法为高效状态机，降低Constrained Decoding的延迟开销
- **Schema推理**：LLM根据任务自动推断最优输出Schema，减少人工定义
- **流式结构化输出**：在流式生成中实时解析部分结构化结果，降低首token延迟
