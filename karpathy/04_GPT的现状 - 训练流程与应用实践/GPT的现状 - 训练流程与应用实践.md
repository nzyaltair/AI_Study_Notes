# GPT 的现状 - 训练流程与应用实践

## 核心概述

本笔记整理自 Andrej Karpathy 在 Microsoft Build 2023 上的演讲 "The State of GPT"。演讲分为两大部分：**如何训练 GPT 助手**和**如何有效使用 GPT 助手**。

**为什么重要**：这份演讲系统性地梳理了从基础模型到可用助手模型的完整训练流水线，并从实践角度给出了提示工程、工具增强、检索增强生成等前沿技术的使用指南。理解这些内容是掌握现代大语言模型生态的关键。

**解决什么问题**：
- 基础模型与助手模型有何本质区别
- GPT 助手的四阶段训练流水线（预训练 → SFT → RM → RL）各自做什么
- RLHF 为什么比纯 SFT 效果更好
- 如何针对 LLM 的认知特点设计有效的提示策略
- 何时使用提示工程、何时使用微调、何时使用 RAG

> [!note] 核心论点
> GPT 助手的训练分为四个阶段：**预训练**（99% 算力，互联网规模数据）、**监督微调（SFT）**（少量高质量对话数据）、**奖励建模（RM）**（人类偏好排序数据）、**强化学习（RL）**（基于奖励模型优化策略）。RLHF 之所以比纯 SFT 更有效，是因为**人类判断优劣的能力远强于从零创作优质内容的能力**。在使用层面，LLM 本质上是"令牌模拟器"——每个 token 的计算量大致相同，不会自我反思，不知道自己不知道什么。提示工程的核心就是弥补 LLM 与人类认知架构之间的差距。

## 知识体系

### 1. GPT 助手的四阶段训练流水线

GPT 助手的训练是一个**四阶段串行流水线**，每个阶段由特定的数据集、训练算法和模型产物组成：

```
预训练 ──→ 监督微调(SFT) ──→ 奖励建模(RM) ──→ 强化学习(RL)
  │              │                │                │
  互联网数据    提示-回复对      偏好排序数据     奖励模型评分
  99%算力       少量GPU          少量GPU          少量GPU
  数月          数小时~数天      数小时~数天      数小时~数天
  │              │                │                │
  基础模型      SFT模型          奖励模型          RLHF模型
```

> [!important] 计算资源分配
> 预训练阶段占据整个训练过程中 **99%** 的计算时间和浮点运算量。其余三个微调阶段仅需少量 GPU，耗时数小时到数天。这意味着预训练是决定模型基础能力的核心阶段。

### 2. 第一阶段：预训练（Pretraining）

#### 2.1 数据收集

预训练需要互联网规模的海量数据。以 Meta 的 LLaMA 模型为例，数据混合包含：

| 数据类型 | 来源 | 特点 |
|----------|------|------|
| 网页爬取 | CommonCrawl, C4 | 量大，质量参差 |
| 代码 | GitHub | 结构化，高质量 |
| 百科 | Wikipedia | 知识性强 |
| 书籍 | Books | 长文本，深度内容 |
| 论文 | ArXiv | 学术内容 |
| 问答 | StackExchange | 问答对 |

这些数据按指定比例混合后采样。

#### 2.2 分词（Tokenization）

训练前需将原始文本转为整数序列——这是 GPT 的原生输入形式。分词是**无损转换**。

常用算法：**字节对编码（BPE, Byte Pair Encoding）**——通过迭代合并高频短文本片段形成标记（token）。

> [!info] 词表大小与上下文长度
> - 词汇量：通常为数万（如 GPT-2 为 50,257）
> - 上下文长度：GPT-3 时代约 2,000~4,000，如今可达 100,000+
> - 上下文长度决定了输入序列的最大长度

#### 2.3 关键超参数对比

| 参数 | GPT-3 | LLaMA |
|------|-------|-------|
| 参数量 | 175B | 65B |
| 训练 Token 数 | 3,000 亿 | 14 万亿 |
| GPU 规模 | 数千块 | 2,000 块 |
| 训练时长 | 数月 | ~21 天 |
| 费用 | — | 数百万美元 |

> [!important] 参数量 ≠ 模型能力
> LLaMA 的参数量（65B）远少于 GPT-3（175B），但能力明显更强。核心原因是训练时长显著更长（14 万亿 vs 3,000 亿 token）。**不能仅凭参数数量衡量模型能力**，训练数据量和质量同样关键。

#### 2.4 预训练的工作方式

训练数据被组织为大型数组：

```
维度: (B, T)
B = 批量大小（每行一个独立样本）
T = 最大上下文长度

┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
│<文档1> │<eos>│  <文档2>  │<eos>│...
└──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
```

多个文档用文本结束标记（`<eos>`）拼接填入行中。Transformer 对序列中**每个位置**同时预测下一个 token，并据此调整权重。

#### 2.5 预训练的直观效果

以莎士比亚文本训练小型 GPT 为例：
- **初始**：权重完全随机，输出随机字符
- **训练中**：模型逐渐掌握词汇、空格、标点的使用
- **训练后**：输出越来越连贯稳定

训练关注的核心指标是**损失（Loss）**：损失越低，模型对下一个词的预测越准确。

#### 2.6 基础模型的能力与局限

预训练产生的模型称为**基础模型（Base Model）**。业界发现基础模型在语言建模过程中学会了**强大的通用表征**，可高效微调适配各类下游任务。

**演进历程**：
- **GPT-1**：预训练 + 微调范式——大模型 + 少量例子即可高效微调
- **GPT-2**：发现仅靠**提示（Prompting）**效果可媲美甚至超越微调——拉开"提示优于微调"的时代序幕

> [!warning] 基础模型 ≠ 助手
> 基础模型的功能是**补全文档**，而非回答问题。
> 
> 如果你要求它"写一首关于面包和奶酪的诗"，它可能反过来提出更多问题——因为它在补全它认为的文档内容。
> 
> 可用特定方式引导基础模型（如"假设这是一首关于面包和奶酪的诗："），但效果不可靠。

**当前可用的基础模型**：
- GPT-3 基础模型：可通过 API 调用（`davinci`）
- GPT-2 基础模型：可从 GitHub 下载权重
- LLaMA 系列：Meta 发布，但不可商用
- GPT-4 基础模型：从未发布

### 3. 第二阶段：监督微调（SFT）

#### 3.1 核心思路

用**少量但高质量**的对话数据替换预训练时使用的大量低质量互联网文档。算法上没有任何改变——仍是语言建模，只是替换了训练数据集。

```
预训练数据：量大、质量低（互联网文档）
SFT 数据：量少、质量高（提示-理想回复对）
```

#### 3.2 数据收集

- 邀请**专业人员**整理提示与理想回复
- 通常收集**数万条**数据
- 标注员按照详细规范操作，要求做到：**有用、真实、无害**

#### 3.3 SFT 模型的特点

- 是**真正的助手**，可以实际部署
- 在一定程度上能用
- 但还不是最优——后续的 RLHF 可进一步提升

> [!note] 算法不变，数据改变
> SFT 的关键洞察：无需改变训练算法，只需用高质量的问答对话数据继续训练。模型从"文档补全器"转变为"问答助手"。

---

### 4. 第三阶段：奖励建模（Reward Modeling）

#### 4.1 核心思路

从"绝对创作"转向"相对比较"——让人类标注员**比较**模型生成的多个回复，按优劣排序。

#### 4.2 训练流程

1. 对同一提示，用已训练的 SFT 模型生成**多个不同回答**
2. 请人类标注员对所有回答**两两比较**并排序
3. 训练 Transformer 为每个回复打分（绿色奖励信号）
4. 损失函数使评分结果与人类排序一致——优质回复评分高于劣质回复

```
提示: "写一个判断回文字符串的函数"
  ├── 回答A ──→ 奖励模型评分: 高
  ├── 回答B ──→ 奖励模型评分: 中
  └── 回答C ──→ 奖励模型评分: 低
```

> [!warning] 奖励模型不能直接用作助手
> 奖励模型的功能是为回复打分，而非生成回复。它的价值在于为下一阶段的强化学习提供评分信号。

#### 4.3 评估的困难

判断哪个回答更好往往很困难——有时评估一对问答就要花上数小时。这反映了奖励建模的挑战性。

---

### 5. 第四阶段：强化学习（RL）

#### 5.1 训练流程

1. 使用一批**新的提示词**
2. 从 SFT 模型出发，用当前策略生成若干回复（黄色文本）
3. **奖励模型保持不变**，为每条回复打分
4. 沿用语言建模的损失函数，但只训练生成的回复部分
5. 根据评分调整训练权重：高分回复的 token 被加强，低分回复的 token 被削弱
6. 对大量提示和批次反复执行

```
提示 → 策略模型生成回复 → 奖励模型评分 → 调整策略权重
         ↑                                          │
         └──────────────────────────────────────────┘
                        循环迭代
```

#### 5.2 RLHF 模型的特点

- 最终可部署的助手模型（如 ChatGPT 就是 RLHF 模型）
- 其他模型如 Vicuna-13B 等是 SFT 模型
- RLHF 模型在人类评估中普遍优于 SFT 模型

#### 5.3 为什么 RLHF 比 SFT 更有效

> [!important] 比较容易，创作困难
> 一个核心洞察：**人类判断优劣的能力远强于从零创作优质内容的能力**。
> 
> 以"写一首关于回形针的诗"为例：
> - 作为 SFT 标注员：你可能不擅长写诗，创作困难
> - 作为 RM 标注员：给你几首诗，你更容易判断哪些好、哪些不好
> 
> 这种**不对称性**说明：通过比较来发挥人类的判断力，比要求人类直接创作优质内容更有效。RLHF 正是利用了这种不对称性。

#### 5.4 RLHF 的代价：多样性下降

> [!warning] RLHF 模型的多样性不如基础模型
> RLHF 模型的输出变得更趋单一，生成内容的差异变小。基础模型本身随机性强，能生成多样化、富有创意的内容。
> 
> **基础模型更适用的场景**：当你手头有若干样本，希望生成更多风格相近的内容时（如生成宝可梦名字、风格化文本等），基础模型的多样性优势显著。

### 6. 模型生态全景

#### 6.1 三类模型

| 模型类型 | 特点 | 典型代表 |
|----------|------|----------|
| 基础模型 | 文档补全，非助手 | GPT-3 `davinci`, LLaMA, GPT-2 |
| SFT 模型 | 真正的助手，可部署 | Vicuna, Koala, Alpaca |
| RLHF 模型 | 性能最优的助手 | ChatGPT, Claude, GPT-4 |

#### 6.2 性能排名（2023 年，按医疗评分体系）

1. **GPT-4**（RLHF）— 首屈一指
2. **Claude**（RLHF）
3. **GPT-3.5**（RLHF）
4. 其他模型（多为 SFT 训练）

> [!note] 排名前三的模型均为 RLHF 训练，表明 RLHF 在助手性能上的显著优势。

### 7. LLM 的认知架构：与人类思维的对比

#### 7.1 人类的思维过程

以"写一句话比较加州和阿拉斯加人口"为例，人类的大脑中发生了复杂的过程：

1. **意识到需求**：需要比较两个州的人口
2. **自我认知**：知道自己记不住这些数据 → 使用工具
3. **信息检索**：查维基百科获取人口数据
4. **计算**：用计算器算 39.2 ÷ 0.74 ≈ 53
5. **合理性检查**：53 合理吗？加州人口最多，所以没问题
6. **创意写作**：尝试写，不满意就删除重写
7. **独立审查**：写作时有一个独立的流程在检查内容是否合适

#### 7.2 LLM 的处理方式

对 GPT 来说，这只是一个**标记序列**。关键差异：

| 维度 | 人类 | LLM |
|------|------|-----|
| 计算分配 | 不同步骤投入不同计算量 | 每个 token 计算量大致相同 |
| 内部独白 | 有复杂的内部思考过程 | 无独立思考过程 |
| 自我认知 | 知道自己知道什么/不知道什么 | 不知道自己不知道什么 |
| 自我反思 | 会检查、纠正自己的错误 | 默认不会反思或纠错 |
| 工具使用 | 主动选择合适工具 | 需被明确告知使用工具 |

#### 7.3 LLM 的认知优势

尽管有上述局限，LLM 也有其认知优势：

1. **庞大的事实性知识**：数十亿参数可存储大量事实信息
2. **相对较大的短期记忆**：上下文窗口内的内容可通过自注意力机制被完美调用

> [!important] 提示工程的本质
> 提示工程就是在**弥补 LLM 与人类认知架构之间的差距**。理解 LLM 的认知特点（令牌模拟器、不反思、不知不足），才能设计出有效的提示策略。

### 8. 提示工程技术

#### 8.1 分散推理到多个 token

> [!important] 核心原则
> 不能期望 Transformer 在单个 token 中完成复杂推理。**推理需要分散到更多 token 中**——Transformer 需要 token 来"思考"。

**方法一：少样本提示（Few-shot Prompting）**

提供几个示例，展示思考过程，让 Transformer 模仿这种模式：

```
示例1：[问题] → [详细推理过程] → [答案]
示例2：[问题] → [详细推理过程] → [答案]
问题：[你的问题]
```

**方法二："让我们一步步思考"**

通过这种引导性措辞，让 Transformer 进入展示思考过程的状态，每个 token 的计算量减少，更有可能成功。

#### 8.2 自洽性（Self-Consistency）

- 对同一问题**多次采样**生成多个回复
- 筛选优质样本或进行**多数表决**
- 原理：Transformer 可能运气不佳抽到不好的 token，陷入推理死胡同——多次采样可以规避

#### 8.3 自我检查（Self-Reflection）

> [!example] LLM 知道自己出错了
> 假如要求模型生成一首不押韵的诗，但它生成了押韵的诗。直接问 GPT-4："你完成任务了吗？"——它会承认没有，并主动重试。
> 
> **关键**：如果不提醒它检查，它不会自己检查。你需要在提示中明确要求它检查。

#### 8.4 思维树（Tree of Thoughts）

- 为每个提示保持**多个完成选项**
- 在过程中评分，保留表现较好的
- 需要与 Python 辅助代码结合使用（维护多个提示、树搜索算法）
- 类比 AlphaGo：模仿人类策略 + 多维探索，保留有效分支

> [!note] AlphaGo 类比
> AlphaGo 下棋时既有人类模仿策略，又会在脑海中设想多种可能性并保留有效的。思维树将类似方法应用于文本生成。

#### 8.5 ReAct 模式

将回答结构化为 **思考 → 行动 → 观察** 的循环：

```
Thought: 我需要查一下这个信息
Action: search("关键词")
Observation: 搜索结果...
Thought: 现在我可以回答了
Action: finish(最终答案)
```

在这些步骤中，模型可以使用工具（搜索、计算器等）。

#### 8.6 AutoGPT

让 LLM 管理任务清单，持续递归拆分任务。虽然目前还不实用，但代表了"让模型系统具备思考能力"的方向。

#### 8.7 角色设定的 U 型曲线

```
"你是这个领域的顶尖专家"     → 效果提升
"假设你的智商是 120"        → 效果较好
"假设你的智商是 400"        → 效果变差（超出数据分布，进入科幻题材）
```

> [!warning] 角色设定不要过度
> 要求过高的智商或能力可能让模型超出训练数据分布，反而表现更差。存在一个 **U 型曲线**——需要找到合适的角色设定水平。

#### 8.8 "LLM 不想成功"

> [!important] LLM 的心理倾向
> LLM 的训练数据中包含各种层次的质量——从完全错误的学生答案到专家解答。Transformer 无法区分，倾向于复制所有内容。
> 
> **你希望成功，应该主动去争取**——在提示中明确要求高质量输出，以正确答案为前提条件，可以让 Transformer 表现更佳（因为它不需要将概率分配给低质量方案）。

### 9. 工具增强

#### 9.1 为什么需要工具

如同人类依赖计算工具一样，LLM 也需要工具来弥补其不足。但关键前提是：

> [!warning] LLM 不知道自己不知道什么
> 默认情况下，Transformer 不了解自己擅长什么或不擅长什么。你需要在提示中**明确告诉它**。

#### 9.2 常用工具

- **计算器**：明确告诉 LLM"你不太擅长心算，做大量数字运算时请使用计算器"
- **代码解释器**：执行 Python 代码进行精确计算
- **搜索能力**：检索最新信息

#### 9.3 使用方式

```python
# 在提示中明确指示工具使用
prompt = """
你不太擅长心算。每次需要做大量数字的加法、乘法等运算时，
请使用提供的计算器工具。

可用工具：
- calculator(expression): 计算数学表达式

问题：加州人口是阿拉斯加的多少倍？
"""
```

### 10. 检索增强生成（RAG）

#### 10.1 从检索到记忆的光谱

```
纯检索（传统） ◄─────────────────► 纯记忆（LLM 内部知识）
                    ▲
              RAG 位于中间
```

#### 10.2 RAG 的工作原理

以 LlamaIndex 为例：

1. **索引阶段**：
   - 选取相关文档
   - 分割为若干片段
   - 对片段进行嵌入处理，得到嵌入向量
   - 存储到**向量数据库**中

2. **推理阶段**：
   - 向向量数据库发起查询
   - 获取与当前任务相关的片段
   - 将片段加入提示中
   - LLM 基于增强的上下文进行生成

> [!note] 上下文窗口 = 工作内存
> Transformer 的上下文窗口就是它的工作内存。如果能用任务相关信息填充工作内存，模型表现会非常好——因为它可以完美记住上下文窗口里的所有内容。
> 
> RAG 的价值：就像人类解决复杂问题时查阅教科书或文档一样，即使你大致记得内容，亲自查一遍效果更好。

---

### 11. 约束生成（Constrained Generation）

#### 11.1 核心思路

让 LLM 的输出遵循特定格式。以微软的 Guidance 为例：

- 设定输出必须是 **JSON 格式**
- 干预 Transformer 输出的 token 概率，锁定某些 token
- Transformer 只需在限定范围内"填空"
- 可对空白处可填充的内容施加更多限制

> [!tip] 实用价值
> 约束生成在需要结构化输出的场景中非常有用（如 API 响应、数据提取、代码生成等），避免了生成后解析失败的问题。

### 12. 模型微调实践

#### 12.1 LoRA：高效参数微调

**LoRA（Low-Rank Adaptation）** 是一种高效参数微调技术：

- 只训练模型中**少量分散的部分**
- 模型大部分固定在基础版本，只有少数参数可调
- 实测表现良好，成本大幅降低
- 固定部分可用低精度推理（不通过梯度下降更新）

#### 12.2 微调 vs 提示工程

| 维度 | 提示工程 | 微调 |
|------|----------|------|
| 技术难度 | 较低 | 较高 |
| 迭代速度 | 快 | 慢 |
| 数据需求 | 无需训练数据 | 需要标注数据 |
| 成本 | 低 | 高 |
| 适用阶段 | 首选 | 提示工程用尽后 |

> [!important] 实践建议
> 1. **监督微调（SFT）**：可实现，因为本质上是继续语言建模任务，相对简单
> 2. **RLHF**：目前仍处于研究阶段，非常不稳定，训练困难，**不推荐初学者自行实现**
> 3. 先充分发挥提示工程的作用，持续使用一段时间后再考虑微调

#### 12.3 开源模型

- LLaMA：质量不错，但可能没有商业许可
- 基于开源模型进行 LoRA 微调是目前最实用的定制化路径

### 13. LLM 的局限性

> [!warning] 使用 LLM 时需牢记的局限
> - **偏见**：模型可能带有训练数据中的偏见
> - **幻觉**：可能编造或虚构信息
> - **推理错误**：在复杂推理任务中可能出错
> - **能力边界**：在某些类别的应用中力不从心
> - **知识时效**：知识有截止日期（如不了解某日期之后的内容）
> - **安全攻击**：提示注入、越狱攻击、数据污染等

**实践建议**：
- 在**低风险场景**下使用 LLM
- 始终配合**人工监督**
- 当作**灵感和建议的来源**、协作者
- 不应视为能独自完成任务的全自动代理

### 14. 实践路线图

#### 14.1 达到最佳性能

1. **使用 GPT-4**：当前能力最强的模型
2. **使用非常详细的提示**：
   - 包含大量任务内容及相关信息
   - 设想如果无法通过邮件沟通，你会如何指导任务承包人
   - 但记住：任务承包人是人，有自己的想法且聪明；LLM 不具备这些特点
3. **深入理解 LLM 的心理机制**，据此设计提示语
4. **使用少样本示例**：不仅告诉它，更要展示给它
5. **使用工具和插件**减轻 LLM 负担
6. **考虑链式提示和反思过程**：不只是单一问答
7. **充分使用提示工程一段时间后**，再考虑微调

#### 14.2 优化性能与降低成本

- 使用低容量模型或较短的提示
- 考虑 LoRA 微调开源模型
- 探索约束生成减少后处理
- RAG 减少对大上下文窗口的依赖

#### 14.3 决策流程

```
任务开始
  │
  ├──→ 提示工程（首选）
  │      ├── 少样本示例
  │      ├── 链式思考
  │      ├── 工具增强
  │      └── RAG
  │
  ├──→ 效果不够？
  │      └──→ 监督微调（SFT + LoRA）
  │
  └──→ 仍不够？
         └──→ RLHF（需专家级研究，高风险）
```

### 15. 关键概念总结

#### 15.1 训练流程核心

| 阶段 | 数据 | 算法 | 产出 |
|------|------|------|------|
| 预训练 | 互联网文档 | 语言建模（预测下一个 token） | 基础模型 |
| SFT | 提示-回复对 | 语言建模（继续） | SFT 模型 |
| 奖励建模 | 偏好排序数据 | 回归（评分匹配排序） | 奖励模型 |
| 强化学习 | 新提示 + 奖励评分 | 策略优化（PPO） | RLHF 模型 |

#### 15.2 三种模型的核心区别

- **基础模型**：补全文档，非助手，多样性高
- **SFT 模型**：真正的助手，可部署，效果尚可
- **RLHF 模型**：性能最优，但多样性下降

#### 15.3 LLM 认知特点速查

- 本质是**令牌模拟器**，每个 token 计算量大致相同
- **不反思**、不自我纠错、不做合理性检查
- **不知不足**——不知道自己擅长什么/不擅长什么
- 拥有庞大的事实性知识（参数即记忆）
- 上下文窗口内的短期记忆近乎完美

#### 15.4 提示工程的核心原则

1. **分散推理**：不要期望单个 token 完成复杂推理
2. **展示而非告知**：少样本示例比纯指令更有效
3. **明确要求检查**：LLM 不会自动自我检查
4. **角色设定适度**：存在 U 型曲线，过度设定适得其反
5. **主动要求高质量**：LLM 倾向于模仿所有质量层次的内容
6. **明确工具使用**：LLM 不知道自己需要工具
