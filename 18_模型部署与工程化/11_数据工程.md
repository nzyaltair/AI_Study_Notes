# 数据工程

## 1. 概述

数据工程是支撑 AI 系统的数据获取、存储、处理和分发的技术体系。它解决从原始数据到可供模型训练/推理使用的数据制品的全链路工程问题，是 ML 系统中工程量最大但常被低估的环节。

数据工程解决的核心问题：
- **海量数据存储**：TB 到 PB 级数据的可靠存储与高效查询
- **分布式处理**：单机无法处理的超大规模数据计算
- **实时与批处理**：离线训练需要批量处理，在线推理需要实时特征
- **数据质量保障**：从噪声数据中提取高质量训练样本
- **特征管理**：特征的共享、复用、版本控制和一致性保障

Google 2015 年论文《Hidden Technical Debt in Machine Learning Systems》指出，ML 系统中代码只占很小一部分，周边数据管道和基础设施构成了大量"技术债"。

## 3. 核心概念

### 3.1 数据架构模式

| 架构 | 特点 | 适用场景 |
|:---|:---|:---|
| 数据仓库 | 结构化数据、SQL 查询、ETL 加载 | BI 分析、报表 |
| 数据湖 | 原始数据、任意格式、Schema-on-Read | ML 训练、探索分析 |
| Lakehouse | 数据湖 + 数据仓库特性（事务、Schema） | 统一分析与 ML |
| 数据网格 | 领域驱动的去中心化数据架构 | 大型组织 |
| 流处理 | 实时数据管道、低延迟 | 在线特征、监控 |

### 3.2 批处理 vs 流处理

| 维度 | 批处理 | 流处理 |
|:---|:---|:---|
| 延迟 | 分钟~小时 | 毫秒~秒 |
| 数据量 | TB~PB | 单条~MB/s |
| 模型 | MapReduce、Spark | Kafka Streams、Flink |
| 一致性 | 最终一致 | 精确一次（Exactly-once） |
| ML 场景 | 离线训练数据准备 | 在线特征实时计算 |

### 3.3 特征存储（Feature Store）

Feature Store 是连接离线训练和在线推理的特征管理平台：

```
                    ┌─────────────────┐
                    │   Feature Store  │
                    │                  │
  离线特征 ────────→│  离线存储 (Parquet)│──────→ 模型训练
  (批量计算)        │  在线存储 (Redis)  │──────→ 在线推理
  实时特征 ────────→│  特征注册表        │
  (流式计算)        │  版本与血缘追踪    │
                    └─────────────────┘
```

**核心能力**：
- **特征注册**：定义特征名、类型、数据源、计算逻辑
- **离线/在线一致性**：训练和推理使用相同特征变换逻辑
- **特征共享**：跨团队复用已定义特征
- **时间旅行**：按时间点查询特征值，避免数据泄露
- **血缘追踪**：特征到数据源的依赖关系

### 3.4 数据版本管理

| 工具 | 类型 | 特点 |
|:---|:---|:---|
| DVC | Git 扩展 | 数据版本控制，类似 Git LFS |
| Delta Lake | 数据湖格式 | ACID 事务、时间旅行、Schema 演化 |
| Apache Iceberg | 数据湖格式 | 快照隔离、分区演化、增量查询 |
| Apache Hudi | 数据湖格式 | 增量更新、CDC 支持 |
| LakeFS | Git-for-Data | 对象存储上的版本控制 |

## 4. 技术原理

### 4.1 MapReduce 模型

分布式数据处理的基础范式：

```
Map 阶段:  (k1, v1) → list(k2, v2)    # 并行处理每个分片
Shuffle 阶段: 按 k2 分组               # 网络传输
Reduce 阶段: (k2, list(v2)) → list(v3) # 聚合结果
```

**Word Count 示例**：
```
Map:    "hello world hello" → (hello, 1), (world, 1), (hello, 1)
Shuffle: (hello, [1, 1]), (world, [1])
Reduce: (hello, 2), (world, 1)
```

### 4.2 Spark RDD 与内存计算

Spark 的核心抽象 RDD（Resilient Distributed Dataset）：

- **内存计算**：中间结果保留在内存中，避免反复磁盘读写
- **懒求值**：Transformations 延迟执行，Actions 触发计算
- **血统（Lineage）**：记录 RDD 的变换历史，节点故障时可重建
- **分区（Partition）**：数据分布在不同节点上并行处理

Spark 执行计划：
```
代码 → 逻辑计划 → 优化（Catalyst 优化器）→ 物理计划 → RDD 执行
```

### 4.3 Kafka 消息队列

分布式流处理的基础设施：

```
Producer → Topic (Partition 0, Partition 1, ...) → Consumer Group
                    ↓
              Broker 集群（持久化、复制）
```

- **分区并行**：Topic 分为多个 Partition，支持水平扩展
- **消费者组**：同组内消费者分担分区，实现并行消费
- **持久化**：消息持久化到磁盘，支持回放
- **Exactly-once**：事务性生产者 + 幂等消费者

### 4.4 LLM 预训练数据管线

大模型预训练的数据处理流水线：

```
原始数据 (CommonCrawl/GitHub/...)
    ↓
格式标准化 (HTML→文本, 代码解析)
    ↓
质量过滤 (启发式规则 + 模型评分)
    ↓
去重 (MinHash / SimHash)
    ↓
安全过滤 (PII 脱敏, 有害内容过滤)
    ↓
Token 化 (BPE/SentencePiece)
    ↓
数据配比 (领域混合比例优化)
    ↓
训练就绪数据集
```

**关键去重算法 — MinHash**：

通过 Jaccard 相似度检测近似重复文档：

$$J(A, B) = \frac{|A \cap B|}{|A \cup B|}$$

MinHash 通过随机哈希函数逼近 Jaccard 相似度：
1. 对文档的 n-gram 集合施加 $k$ 个哈希函数
2. 每个哈希函数取最小值作为签名
3. 签名相同的文档对为候选重复

### 4.5 数据标注工程

| 标注类型 | 方法 | 工具 |
|:---|:---|:---|
| 人工标注 | 众包平台、专业标注团队 | Label Studio, Doccano |
| 主动学习 | 模型选择信息量大的样本标注 | ModAL |
| 弱监督 | 规则/启发式自动标注 | Snorkel |
| 合成数据 | 模型生成训练数据 | Self-Instruct, Evol-Instruct |
| 标注蒸馏 | 大模型标注小模型数据 | GPT-4 标注 |

**标注质量控制**：
- 一致性检验（Inter-Annotator Agreement, Cohen's Kappa）
- 黄金标准集交叉验证
- 标注者能力评估与淘汰
- 多人标注投票/置信度加权

## 6. 优势与局限

### 6.1 价值

- **可复现性**：数据版本化保障训练实验可复现
- **特征复用**：Feature Store 避免重复开发特征
- **一致性保障**：训练-推理特征一致性消除线上线下偏差
- **规模化处理**：分布式框架处理 PB 级数据
- **实时性**：流处理支撑在线特征和实时模型

### 6.2 挑战

- **工程复杂度高**：数据管道涉及众多组件，维护成本大
- **数据质量难控**：原始数据噪声多、分布漂移、标注不一致
- **隐私合规**：GDPR/CCPA 等法规对数据使用和存储的限制
- **多模态数据**：文本、图像、视频的统一存储和处理挑战
- **冷启动**：新特征上线需要回填历史数据
- **成本控制**：大规模数据存储和计算的资源消耗

## 7. 应用场景

| 场景 | 数据工程需求 | 典型方案 |
|:---|:---|:---|
| 推荐系统 | 用户行为实时特征 + 离线用户画像 | Kafka + Flink + Feature Store |
| LLM 预训练 | 万亿 token 清洗、去重、配比 | Spark + MinHash + 自定义管线 |
| 金融风控 | 实时交易特征 + 历史聚合 | Flink + Redis + Hive |
| 自动驾驶 | PB 级传感器数据标注与管理 | Ray + 数据湖 + 主动学习 |
| 搜索排序 | 用户点击流 + 实时反馈 | Kafka + Spark Streaming |
| 语音识别 | 大规模音频采集与转录标注 | 分布式存储 + 众包标注 |

## 9. 前沿发展

- **Lakehouse 统一架构**：Delta Lake/Iceberg/Hudi 融合数据湖与仓库能力
- **数据网格（Data Mesh）**：去中心化的领域数据自治架构
- **合成数据飞轮**：模型生成数据→训练更强模型→生成更高质量数据
- **实时特征计算**：Flink + 向量数据库实现毫秒级在线特征
- **多模态数据湖**：统一存储文本、图像、音频、视频的原生数据湖
- **Data-centric AI**：从模型中心转向数据中心，通过数据改进提升模型性能
- **隐私计算**：联邦学习、差分隐私、安全多方计算实现数据可用不可见
- **向量数据库**：为 LLM RAG 场景提供高效语义检索基础设施
- **数据合约（Data Contract）**：数据生产者与消费者间的 Schema 与质量协议
