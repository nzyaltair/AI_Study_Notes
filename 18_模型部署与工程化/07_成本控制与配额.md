# 成本控制与配额

## 1. 概述

随着大语言模型规模从亿级增长到千亿级，推理成本快速攀升。成本控制与配额管理成为企业级 LLM 平台的核心工程能力。

成本控制解决的核心问题：
- **推理成本优化**：在保证服务质量的前提下降低单 Token 推理成本
- **资源公平分配**：多租户/多团队共享 GPU 资源时的公平性
- **预算可控**：防止成本失控，按用户/租户/模型维度追踪和限制
- **弹性成本**：应对流量波动的弹性计费策略

通过模型层面优化（量化、蒸馏）、推理层面优化（连续批处理、推测解码）、架构层面优化（自动扩缩容、冷热分离）和运营层面优化（成本归因、预算管理）四个层次系统性降低总成本。

## 3. 核心概念

### 3.1 成本结构

| 成本类型 | 组成 | 优化手段 |
|:---|:---|:---|
| **计算成本** | GPU 实例费用 | Spot 实例、预留实例、弹性伸缩 |
| **推理成本** | Token 生成计算 | 量化、连续批处理、推测解码 |
| **存储成本** | 模型权重存储 | 模型分片、冷热分层存储 |
| **带宽成本** | 跨区传输、公网出口 | 就近部署、CDN 加速 |

### 3.2 配额管理

多级配额确保资源公平分配：
- **用户级**：每用户 QPS 限制、每日 Token 配额
- **租户级**：多租户资源隔离与优先级管理
- **系统级**：整体 QPS 上限、GPU 资源总量控制

### 3.3 成本归因

按用户/租户/模型维度追踪 Token 消耗和计算成本。关键指标：$/Token、$/Request、GPU 利用率。

## 4. 技术原理

### 4.1 GPU 实例选型

| 实例类型 | GPU | 显存 | 按需价格（$/h） | Spot 折扣 |
|:---|:---|:---|:---|:---|
| AWS g5.xlarge | A10G | 24GB | ~1.51 | ~70% |
| AWS p4d.24xlarge | A100×8 | 80GB×8 | ~32.77 | ~70% |
| GCP a2-highgpu-1g | A100 | 40GB | ~4.00 | ~70% |
| GCP a3-highgpu-8g | H100×8 | 80GB×8 | ~32.00 | ~70% |

### 4.2 模型量化优化

- INT4 量化：减少 75% 模型大小，推理速度提升 2-4 倍
- INT8 量化：减少 50%，速度提升 1.5-2 倍
- GPTQ 和 AWQ 是主流的后训练量化方法

### 4.3 推测解码

小模型（Draft Model）生成候选 Token，大模型（Target Model）并行验证。若候选 Token 被接受，大模型无需逐步自回归生成，推理速度提升 2-3 倍。

接受概率：$p = \min\left(1, \frac{p_{\text{target}}(x)}{p_{\text{draft}}(x)}\right)$

### 4.4 模型路由

简单任务路由到 7B 模型，复杂任务路由到 70B 模型，可降低 50%+ 成本。路由策略：
- 基于规则：按请求类型/长度路由
- 基于模型：小模型先评估难度，决定是否升级到大模型
- 基于学习：训练路由模型预测最优目标

### 4.5 请求缓存

缓存高频请求的响应，减少重复计算，可消除 30%+ 的重复请求。缓存策略：
- 精确匹配：完全相同的请求复用结果
- 语义缓存：语义相似的请求复用结果
- 前缀缓存：共享公共前缀的 KV-Cache

### 4.6 令牌桶限流

桶容量 $C$，填充速率 $r$，允许突发流量：

$$n(t) = \min(C, n(t_0) + r \cdot (t - t_0))$$

### 4.7 资源隔离策略

| 隔离方式 | 机制 | 适用场景 |
|:---|:---|:---|
| 物理隔离 | 不同租户使用独立硬件 | 关键租户、高安全要求 |
| 逻辑隔离 | 通过软件调度实现资源分配 | 普通租户 |
| 混合隔离 | 关键租户物理隔离，普通租户逻辑隔离 | 企业级平台 |

## 5. 关键方法与模型

### 5.1 最大最小公平算法

多租户资源分配中，先满足最小需求，再将剩余资源按比例分配，确保公平性。

### 5.2 弹性伸缩优化

- **峰值预测**：基于历史流量预测峰值，提前预热实例
- **冷热分离**：核心实例常驻，弹性实例按需启动
- **Spot 实例**：使用 Spot 实例承接弹性流量，成本降低 70%
- **自动缩容**：低流量时自动缩容，避免资源浪费

### 5.3 多模型共享 GPU

- **MIG 分区**：A100/H100 硬件分区，隔离的多实例
- **时间切片**：多个模型轮流使用同一 GPU
- **vGPU**：软件层 GPU 虚拟化

## 6. 优势与局限

### 6.1 优势

- **成本透明**：精确到 Token 级的成本追踪
- **公平分配**：多级配额确保资源公平
- **弹性优化**：多层次的成本优化策略
- **预算可控**：预算告警和自动限制

### 6.2 局限

- **量化精度损失**：INT4 量化可能导致 1-2% 精度下降
- **Spot 实例不稳定**：可能被抢占，影响服务可用性
- **冷启动延迟**：弹性扩容时模型加载耗时
- **模型路由复杂性**：路由策略不当可能导致质量下降
