# 提示工程

## 1. 概述

提示工程（Prompt Engineering）是设计与优化大语言模型输入提示的系统化方法，旨在通过结构化的提示设计引导LLM产生高质量、可控的输出。它是LLM应用的"软编程"接口——不需要修改模型参数，仅通过输入文本即可控制模型行为。

提示工程解决的核心问题：LLM的行为由输入提示决定，但"自然语言→模型行为"的映射具有高度不确定性。提示工程通过结构化设计降低这种不确定性，使LLM输出从"随机"走向"可控"。

## 3. 核心概念

### 提示的组成要素

一个完整的提示通常包含以下要素：

| 要素 | 作用 | 示例 |
|------|------|------|
| 指令（Instruction） | 明确任务要求 | "请总结以下文章" |
| 上下文（Context） | 提供背景信息 | RAG检索结果、历史对话 |
| 示例（Examples） | 引导输出格式和风格 | Few-Shot示例 |
| 输入数据（Input） | 待处理的具体内容 | 用户的问题或文档 |
| 输出格式（Output Format） | 约束输出结构 | "以JSON格式输出" |

### 关键术语

- **Zero-Shot**：不提供示例，直接让模型完成任务
- **Few-Shot**：提供少量示例引导模型行为
- **Chain-of-Thought（CoT）**：引导模型展示推理过程
- **Prompt Template**：带变量的可复用提示模板
- **Prompt Chaining**：将多个提示串联，前一个的输出作为后一个的输入
- **Prompt Injection**：恶意输入劫持模型行为的攻击手段

## 4. 技术原理

### 基础提示技术

| 技术 | 原理 | 适用场景 |
|------|------|---------|
| 零样本（Zero-Shot） | 直接提问，依赖模型预训练知识 | 简单任务、通用能力 |
| 少样本（Few-Shot） | 提供3-5个输入-输出示例 | 特定格式、领域任务 |
| 角色提示 | 设定专业角色（"你是XX专家"） | 专业领域咨询 |
| 指令提示 | 明确任务要求与约束条件 | 格式化输出、任务控制 |
| 格式提示 | 指定输出格式（JSON/Markdown/表格） | 结构化输出 |

### 思维链（Chain-of-Thought）

引导模型展示中间推理过程，适用于数学、逻辑、多步推理任务。详见 `思维链` 。

**Zero-Shot CoT**：在提示末尾添加"让我们逐步思考"即可触发推理。

**Few-Shot CoT**：在示例中包含推理过程：

```
Q: 图书馆有32本书，借出12本后又还回5本，现在有多少本？
A: 逐步推理：
1. 初始数量：32本
2. 借出12本：32 - 12 = 20本
3. 还回5本：20 + 5 = 25本
答案：25本
```

### 高级推理技术

| 技术 | 原理 | 适用场景 | 代表论文 |
|------|------|---------|---------|
| Self-Consistency | 多次采样CoT推理，取多数投票 | 提高推理准确性 | Wang et al., 2022 |
| Tree of Thoughts（ToT） | 树形搜索多条推理路径，可回溯 | 规划、搜索类任务 | Yao et al., 2023 |
| Graph of Thoughts（GoT） | 图结构推理，支持路径合并 | 复杂推理 | Besta et al., 2023 |
| Self-Refine | 生成→自我审查→修正循环 | 代码生成、写作 | Madaan et al., 2023 |
| Decomposition | 将复杂任务分解为子任务 | 多步推理 | Zhou et al., 2022 |

### ReAct框架

Thought → Action → Observation循环，将推理与工具调用交替执行。详见 `ReAct` 。

```
Thought: 用户想知道北京天气，我需要查询天气工具
Action: get_weather("北京")
Observation: 晴天 25°C
Thought: 获取到了天气信息，可以回答
Final Answer: 北京今天晴天，气温25°C。
```

### 提示工程化

将提示从"手工调参"提升为系统化工程实践：

- **模板化**：使用变量和模板引擎管理提示（LangChain PromptTemplate）
- **版本控制**：对提示进行Git版本管理，追踪变更
- **A/B测试**：对比不同提示在评估集上的效果
- **评估体系**：建立提示质量评估指标（准确率、格式合规率等）
- **自动优化**：使用DSPy等框架自动搜索最优提示

## 5. 关键方法与模型

### CoT系列方法演进

```
Zero-Shot → Few-Shot → CoT → Self-Consistency → ToT → GoT
    ↓
推理模型（o1/o3）：将CoT内化为模型能力
```

| 方法 | 核心思想 | 提升 | 论文 |
|------|---------|------|------|
| CoT | 展示推理过程 | GSM8K: 17.7%→58.1% | Wei et al., 2022 |
| Self-Consistency | 多次采样投票 | GSM8K: 58.1%→74.4% | Wang et al., 2022 |
| ToT | 树搜索推理路径 | 24点游戏: 4%→36% | Yao et al., 2023 |
| DSPy | 自动优化提示 | MATH: +15% | Khattab et al., 2023 |

### DSPy框架

DSPy将提示工程从"手工编写"转变为"声明式编程+自动优化"：

```python
import dspy

# 声明式定义任务
class QA(dspy.Signature):
    """回答问题"""
    question = dspy.InputField()
    answer = dspy.OutputField()

# DSPy自动优化提示
qa = dspy.ChainOfThought(QA)
# 使用BootstrapFewShot自动搜索最优Few-Shot示例
optimizer = dspy.BootstrapFewShot(metric=my_metric)
optimized_qa = optimizer.compile(qa, trainset=train_examples)
```

## 6. 优势与局限

### 优势

- **零成本迭代**：修改提示不需要重新训练模型
- **即时生效**：提示修改后立即可用，无需等待训练
- **灵活性高**：同一模型通过不同提示可完成多种任务
- **可解释性**：提示是自然语言，人类可直接理解

### 局限

- **脆弱性**：微小的提示修改可能导致输出大幅变化
- **不可靠性**：无法100%保证输出格式和行为
- **依赖模型能力**：同一提示在不同模型上效果差异大
- **难以系统化**：缺乏完整的理论指导，仍依赖经验
- **上下文消耗**：Few-Shot示例占用宝贵的上下文窗口

## 7. 应用场景

- LLM应用开发的基础技能，所有LLM应用都依赖提示设计
- `Agent系统`：Agent的行为完全由系统提示定义
- `RAG`：检索结果需要通过提示整合到上下文中
- 数据标注：用LLM自动标注数据
- LLM-as-Judge：用LLM评估其他LLM的输出质量
- 教育辅助：生成练习题、解题步骤、学习建议

## 8. 与其他技术关系

- 提示工程是 `Agent系统` 的核心技术，Agent行为由提示定义
- `思维链` 推理是Agent规划能力的基础
- `结构化输出` 依赖提示工程引导格式
- `RAG` 将检索结果注入提示上下文
- 提示质量直接影响 `LLM应用评估` 的结果
- 推理模型（o1/o3）部分替代了外部CoT提示的需求

## 9. 前沿发展

- **推理模型减少提示依赖**：o1/o3等模型内部具备CoT能力，复杂推理不再需要精心设计CoT提示
- **自动提示优化**：DSPy、PromptBreeder等方法自动搜索最优提示，替代人工调优
- **多模态提示**：从纯文本提示扩展到图文混合提示
- **提示安全**：Prompt Injection防御、提示水印等技术受到关注
- **提示编译**：将高层声明式提示编译为模型最优输入，类似编译器优化
- **提示标准化**：行业开始形成提示编写规范和最佳实践标准
