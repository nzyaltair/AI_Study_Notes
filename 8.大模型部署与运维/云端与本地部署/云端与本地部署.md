# 云端与本地部署

## 1. 简介

大模型部署是将训练好的大语言模型（LLM）应用于实际生产环境的关键环节。根据部署环境的不同，可分为云端部署和本地/边缘部署两大类。本文档聚焦这两类部署方式的架构设计、工具链选择、性能权衡与运维实践，为AI开发者与MLOps工程师提供可落地的工程决策依据。

%% 核心目标：建立一套完整的大模型部署知识体系，帮助团队根据实际需求选择合适的部署方案 %% ^core-objective

## 2. 云端与本地部署对比

| 维度 | 云端部署 | 本地/边缘部署 |
|------|----------|---------------|
| **硬件资源** | 弹性伸缩，按需付费 | 固定资源，一次性投入 |
| **成本模型** | 按使用量付费（计算、存储、网络） | 硬件成本+运维成本 |
| **部署难度** | 低（托管服务），高（自定义集群） | 中高（需配置硬件和软件环境） |
| **性能表现** | 高性能（GPU/TPU集群） | 受限于本地硬件，适合轻量级应用 |
| **隐私安全** | 数据需传输到云端，依赖云厂商安全机制 | 数据本地化，隐私可控 |
| **网络依赖** | 强依赖网络连接 | 无网络依赖或弱依赖 |
| **扩展性** | 支持大规模分布式部署 | 受限于本地硬件资源 |
| **更新维护** | 云厂商负责底层维护，用户负责模型更新 | 用户负责全部维护工作 |
| **适用场景** | 大规模生产服务、弹性流量、资源需求波动大 | 隐私敏感场景、离线应用、边缘设备、开发测试 |

### 2.2 本地/边缘部署方案

**核心优势**：
- 数据隐私可控
- 低延迟响应
- 无网络依赖
- 固定成本，长期划算

**适用场景**：
- 医疗、金融等隐私敏感行业
- 边缘设备（如手机、IoT设备）
- 开发测试环境
- 离线应用

## 5. 模型格式与转换

### 5.1 主流模型格式

| 格式 | 核心优势 | 适用场景 | 支持框架 |
|------|----------|----------|----------|
| **PyTorch (.pth, .pt)** | 原生格式，支持动态计算图 | 研究和开发，灵活调试 | PyTorch |
| **Safetensors (.safetensors)** | 安全，快速加载，无执行代码 | 生产环境，安全要求高 | 多种框架支持 |
| **ONNX (.onnx)** | 跨平台，高性能，模型压缩 | 跨框架部署，边缘设备 | 多种框架支持 |
| **GGUF (.gguf)** | 量化模型，小体积，高性能 | 本地部署，低资源环境 | llama.cpp, Ollama |
| **TensorRT Engine (.plan)** | NVIDIA GPU优化，极低延迟 | NVIDIA GPU生产部署 | TensorRT |
| **MLX (.mlxmodel)** | Apple Silicon优化，内存高效 | Apple设备部署 | mlx |

### 5.2 模型格式转换示例

**PyTorch → ONNX**：
```python
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "meta-llama/Llama-3-8b"
model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype=torch.float16)
tokenizer = AutoTokenizer.from_pretrained(model_name)

# 导出ONNX模型
dummy_input = tokenizer("Hello, ", return_tensors="pt")
torch.onnx.export(
    model,
    tuple(dummy_input.values()),
    "llama-3-8b.onnx",
    export_params=True,
    opset_version=17,
    do_constant_folding=True,
    input_names=["input_ids", "attention_mask"],
    output_names=["output"],
    dynamic_axes={
        "input_ids": {0: "batch_size", 1: "seq_len"},
        "attention_mask": {0: "batch_size", 1: "seq_len"},
        "output": {0: "batch_size", 1: "seq_len"}
    }
)
```

**PyTorch → GGUF**：
```bash
# 使用llama.cpp的转换脚本
python convert.py meta-llama/Llama-3-8b --outtype q4_0 --outfile llama-3-8b-q4_0.gguf
```

## 7. 典型部署模式

### 7.1 单机单卡部署

**适用场景**：
- 开发测试环境
- 小规模应用
- 模型规模适中（7B-13B参数）

**部署流程**：
1. 准备硬件环境（GPU/CPU）
2. 安装依赖软件（CUDA、Python、推理引擎）
3. 下载或转换模型
4. 启动推理服务
5. 配置API访问

**示例**：使用vLLM在A10G上部署LLaMA-3
```bash
# 安装vLLM
pip install vllm

# 启动vLLM服务
vllm serve meta-llama/Llama-3-8b-hf --port 8000 --host 0.0.0.0 --gpu-memory-utilization 0.9
```

### 7.2 多卡分布式部署

**适用场景**：
- 大规模模型（70B+参数）
- 高吞吐量需求
- 低延迟要求

**部署方式**：
- **张量并行（TP）**：将模型张量分布到多个GPU上
- **流水线并行（PP）**：将模型层分布到多个GPU上
- **数据并行（DP）**：将数据分布到多个GPU上

**示例**：使用TensorRT-LLM进行多卡部署
```bash
# 构建TensorRT-LLM引擎
trtexec --onnx=llama-3-70b.onnx --saveEngine=llama-3-70b.engine --fp16 --useDLACore=all --batch=8

# 启动多卡推理服务
trtserver --model-repository=models --cuda-memory-fraction=0.9 --allow-metrics=true
```

### 7.3 多实例弹性伸缩

**适用场景**：
- 流量波动大的应用
- 成本敏感场景
- 高可用性要求

**实现方式**：
- 使用Kubernetes进行容器编排
- 配置Horizontal Pod Autoscaler（HPA）
- 基于CPU/GPU利用率或请求量进行扩缩容
- 集成云厂商的自动扩缩容服务

**示例**：Kubernetes部署vLLM服务
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: vllm-service
  template:
    metadata:
      labels:
        app: vllm-service
    spec:
      containers:
      - name: vllm-container
        image: vllm/vllm-openai:latest
        ports:
        - containerPort: 8000
        resources:
          limits:
            nvidia.com/gpu: 1
        env:
        - name: MODEL_NAME
          value: "meta-llama/Llama-3-8b-hf"
        - name: GPU_MEMORY_UTILIZATION
          value: "0.9"
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: vllm-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: vllm-deployment
  minReplicas: 1
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```

## 9. 部署流程示例

### 9.1 示例1：使用vLLM在A10G上部署LLaMA-3

**环境准备**：
- AWS EC2 g5.xlarge实例（A10G GPU，24GB显存）
- Ubuntu 22.04 LTS
- CUDA 12.1

**部署步骤**：

1. **安装依赖**：
```bash
# 更新系统
apt update && apt upgrade -y

# 安装Python和pip
apt install -y python3-pip python3-venv

# 创建虚拟环境
python3 -m venv vllm-env
source vllm-env/bin/activate

# 安装vLLM
pip install vllm
```

2. **启动vLLM服务**：
```bash
vllm serve meta-llama/Llama-3-8b-hf \
  --port 8000 \
  --host 0.0.0.0 \
  --gpu-memory-utilization 0.9 \
  --max-num-seqs 256 \
  --max-model-len 8192
```

3. **测试API服务**：
```bash
curl -X POST http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "meta-llama/Llama-3-8b-hf", "messages": [{"role": "user", "content": "Hello!"}], "max_tokens": 100}'
```

4. **配置反向代理和SSL**（可选）：
```bash
# 安装Nginx
apt install -y nginx

# 配置Nginx
cat > /etc/nginx/sites-available/vllm << EOF
server {
    listen 80;
    server_name your-domain.com;
    
    location / {
        proxy_pass http://localhost:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade \$http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host \$host;
        proxy_cache_bypass \$http_upgrade;
    }
}
EOF

# 启用站点
ln -s /etc/nginx/sites-available/vllm /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx
```

## 10. 常见问题与解决方案

### 10.1 显存不足

**问题现象**：
- 模型加载失败，提示CUDA out of memory
- 推理过程中显存溢出

**解决方案**：
1. **模型量化**：使用INT8/INT4量化，减少显存占用
   ```bash
   # 使用llama.cpp进行量化
   python convert.py meta-llama/Llama-3-8b --outtype q4_0
   ```

2. **减少批量大小**：降低`max_num_seqs`参数
   ```bash
   vllm serve meta-llama/Llama-3-8b --max-num-seqs 64
   ```

3. **减少上下文窗口**：降低`max_model_len`参数
   ```bash
   vllm serve meta-llama/Llama-3-8b --max-model-len 4096
   ```

4. **使用更高效的推理引擎**：如vLLM、TensorRT-LLM

5. **多卡分布式部署**：将模型分布到多个GPU上

### 10.2 冷启动延迟

**问题现象**：
- 首次请求延迟高
- 模型加载时间长
- 自动扩缩容时新实例启动慢

**解决方案**：
1. **模型预热**：启动服务后发送测试请求
   ```python
   import requests
   
   def warmup_model():
       for _ in range(5):
           requests.post(
               "http://localhost:8000/v1/chat/completions",
               json={
                   "model": "meta-llama/Llama-3-8b",
                   "messages": [{"role": "user", "content": "Warmup"}],
                   "max_tokens": 10
               }
           )
   ```

2. **使用模型缓存**：将常用模型预加载到内存

3. **优化模型加载**：使用Safetensors格式，减少加载时间

4. **配置合理的自动扩缩容策略**：设置适当的扩缩容阈值和冷却时间

### 10.3 批处理调度

**问题现象**：
- 批处理效率低
- 请求等待时间长
- 资源利用率不高

**解决方案**：
1. **使用支持动态批处理的推理引擎**：如vLLM、Triton

2. **调整批处理参数**：
   ```bash
   # vLLM动态批处理配置
   vllm serve meta-llama/Llama-3-8b --max-num-seqs 256 --max-batch-prefill-tokens 4096
   ```

3. **实现请求合并**：将多个小请求合并为一个批处理请求

4. **使用合适的调度算法**：如FIFO、优先级调度等

### 10.4 模型版本管理

**问题现象**：
- 模型版本混乱
- 无法快速回滚到之前的版本
- 模型更新影响在线服务

**解决方案**：
1. **使用模型仓库**：如Hugging Face Hub、AWS S3、阿里云OSS

2. **实现模型版本控制**：
   ```bash
   # 使用git管理模型配置
   git init model-repo
   cd model-repo
   git add model.config
   git commit -m "Add model v1.0"
   ```

3. **实现灰度发布**：
   - 逐步将流量切换到新模型
   - 监控新模型性能
   - 出现问题快速回滚

4. **使用模型注册中心**：如MLflow、DVC等
