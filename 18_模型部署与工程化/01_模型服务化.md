# 模型服务化

## 1. 概述

模型服务化是将训练好的模型封装为可通过网络调用的在线服务的过程，是 AI 落地的关键工程环节。其核心目标是让模型能够以低延迟、高吞吐、高可用的方式响应来自客户端的推理请求。

模型服务化解决的核心问题：
- **推理执行**：高效执行模型前向计算，管理显存和计算资源
- **并发处理**：多请求并发下的批处理调度和资源分配
- **服务可用性**：水平扩展、负载均衡和故障恢复
- **接口标准化**：提供统一的 API 接口供客户端调用

与传统 Web 服务不同，LLM 服务化面临独特挑战：自回归生成的每个请求生成长度不确定，KV-Cache 随序列增长而膨胀，导致批处理调度具有与传统 Web 服务本质上的差异。

## 3. 核心概念

### 3.1 推理引擎

专门用于执行模型推理的软件系统，负责模型加载、计算调度、内存管理和性能优化。LLM 推理引擎需要处理自回归生成、动态序列长度和 KV-Cache 管理等特有挑战。

### 3.2 KV-Cache

Transformer 自回归生成中，缓存历史 Token 的 Key/Value 张量，避免重复计算注意力。KV-Cache 随序列长度线性增长，是 LLM 推理显存占用的主要来源。

### 3.3 批处理（Batching）

将多个并发请求合并为一个批次统一推理，提高 GPU 利用率。传统静态批处理需等待整批完成，连续批处理（Continuous Batching）允许请求完成即释放。

### 3.4 流式响应

LLM 生成具有自回归特性，流式响应将 Token 逐个推送给客户端，显著降低用户感知的首 Token 延迟（TTFT）。基于 Server-Sent Events（SSE）实现。

## 4. 技术原理

### 4.1 PagedAttention

vLLM 的核心创新，灵感来自操作系统的虚拟内存分页机制：

- 将 KV-Cache 划分为固定大小的页（block，通常 16 个 Token）
- 每个请求维护逻辑页到物理页的映射表
- 按需分配物理页，消除显存碎片
- 实测将吞吐提升 2-4 倍

### 4.2 连续批处理

与传统静态批处理的对比：

```
静态批处理：
  Request A: [========]     (8 tokens)
  Request B: [================] (16 tokens) ← 队头阻塞
  Request C: [======]       (6 tokens) ← 等待 B 完成

连续批处理：
  Request A: [========] ✓   → 完成即释放，D 加入
  Request B: [================]
  Request C: [======] ✓     → 完成即释放，E 加入
  Request D:      [=====]   → 动态加入
```

- 传统：$T_{static} = \max(L_i) \times T_{forward}$
- 连续：$T_{cont} = \frac{\sum_i L_i}{N_{max}} \times T_{forward}$

### 4.3 流式响应协议

基于 SSE 的流式响应格式：

```
data: {"choices":[{"delta":{"content":"Hello"}}]}
data: {"choices":[{"delta":{"content":" world"}}]}
data: [DONE]
```

### 4.4 服务架构

```
客户端
  │
  ▼
API 网关（认证/限流/路由）
  │
  ▼
负载均衡器（Round-Robin / Least-Connections / GPU 利用率调度）
  │
  ├──▶ 推理实例 1 (vLLM, GPU 0)
  ├──▶ 推理实例 2 (vLLM, GPU 1)
  └──▶ 推理实例 N (vLLM, GPU N)
```

### 4.5 多 GPU 并行

- **张量并行（TP）**：将模型单层内权重切分到多 GPU，需高带宽互联（NVLink）
- **流水线并行（PP）**：将模型不同层分配到不同 GPU，通信量小但存在气泡
- **数据并行（DP）**：多副本独立处理不同请求，通过负载均衡分发

## 6. 优势与局限

### 6.1 优势

- **高吞吐**：PagedAttention + 连续批处理实现 2-4 倍吞吐提升
- **低延迟**：流式响应将首 Token 延迟降至 200ms 以内
- **可扩展**：水平扩展应对流量波动
- **标准化**：OpenAI 兼容接口降低客户端集成成本

### 6.2 局限

- **冷启动慢**：大模型加载到显存需数十秒，影响弹性伸缩
- **显存瓶颈**：KV-Cache 随并发数和序列长度增长，限制最大并发
- **GPU 成本高**：高端 GPU 供应紧张，大模型推理成本高昂
- **多模型管理复杂**：同时服务多个不同模型需精细的资源调度

## 7. 应用场景

| 场景 | 特点 | 典型方案 |
|:---|:---|:---|
| LLM API 服务 | 高并发公共 API | vLLM + K8s + API 网关 |
| 企业私有化 | 数据安全要求高 | 本地 GPU + vLLM + Docker |
| Agent 平台 | 工具调用、结构化输出 | SGLang + Function Calling |

| 本地推理 | 隐私保护、离线 | Ollama / llama.cpp |

## 9. 前沿发展

- **推测解码集成**：vLLM/TGI 集成推测解码，小模型生成候选 Token、大模型并行验证
- **离散 KV-Cache**：将 KV-Cache 卸载到 CPU 内存或 SSD，突破显存限制
- **多 LoRA 动态加载**：同一基础模型服务多个 LoRA 适配器，按需切换
- **Prefix Cache 共享**：共享公共前缀的 KV-Cache，减少重复计算
- **Serverless LLM**：自动缩容到零，按请求计费
- **推理-计算分离**：KV-Cache 存储与计算节点分离，提高资源利用率
