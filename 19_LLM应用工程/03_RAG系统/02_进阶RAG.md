# 进阶RAG

## 1. 概述

进阶RAG（Advanced RAG）是在 `基础RAG` 的检索-生成流程中引入多阶段优化技术，解决基础RAG面临的四大挑战：查询-文档语义鸿沟、检索噪声与冗余、上下文窗口限制、静态检索范式缺乏反馈。

进阶RAG通过查询前优化（重写/分解/HyDE）、检索中增强（多向量/融合/递归）、检索后处理（压缩/重排）、索引结构演进（层次化/图检索）以及训练感知的联合优化，构建更准确、高效、鲁棒的检索增强生成系统。

## 3. 核心概念

### 进阶RAG优化维度

```
查询前（Pre-Retrieval）    检索中（Retrieval）     检索后（Post-Retrieval）
     ↓                        ↓                        ↓
  查询重写                  多向量检索                 上下文压缩
  HyDE                     融合检索                   多阶段重排
  子查询分解                递归检索                   去重与多样性
  查询扩展                  图检索                     来源引用
```

### 关键术语

- **HyDE**：Hypothetical Document Embeddings，先生成假设答案再检索
- **RRF**：Reciprocal Rank Fusion，倒数排名融合
- **Self-RAG**：LLM自评估是否需要检索及检索质量
- **CRAG**：Corrective RAG，检索质量评估与纠正
- **GraphRAG**：基于图结构的检索增强生成
- **Agentic RAG**：Agent式动态检索，按需多跳

## 4. 技术原理

### 查询前优化（Pre-Retrieval）

**查询重写**：使用LLM将原始查询重写为更有利于检索的形式，提升召回率15-30%。

```python
# 查询重写示例
original_query = "怎么提高模型准确率"
rewritten_query = llm.rewrite("如何提升机器学习模型的预测准确率方法")
# → 生成多个变体，分别检索后合并
```

**HyDE（Hypothetical Document Embeddings）**：先生成假设性答案文档，用该文档的嵌入进行检索，解决查询与文档在嵌入空间中的语义不匹配问题。

```python
class HyDERetriever:
    def retrieve(self, query):
        # 1. 生成假设性答案文档
        hypothetical_doc = self.llm.generate(f"请回答这个问题：{query}")
        # 2. 用假设文档（而非查询）的嵌入进行检索
        embedding = self.embedding_model.encode(hypothetical_doc)
        # 3. 检索与假设文档相似的文档
        return self.vector_db.similarity_search(embedding, k=5)
```

原理：查询通常很短，而文档通常较长，两者在嵌入空间中分布不同。通过生成假设文档，将查询"提升"到文档的嵌入空间，提升检索精度。在事实性任务上Recall@5提升20-40%。

**子查询分解**：将复杂多跳查询分解为多个子查询，分别检索后合并结果。

```
查询："比较GPT-4和Claude 3在代码生成方面的表现"
  → 子查询1："GPT-4代码生成能力评测"
  → 子查询2："Claude 3代码生成能力评测"
  → 分别检索 → 合并结果 → 对比生成
```

### 检索中增强（Retrieval）

**多向量检索（Multi-Vector Retrieval）**：为每个文档生成多个向量表示（摘要向量、关键词向量、问题向量），分别检索后融合。

**递归检索（Recursive Retrieval）**：先检索父文档，再在父文档内部进行细粒度检索。配合父子分块策略：父分块提供上下文，子分块用于精确匹配。

**融合检索（Hybrid Retrieval）**：结合稠密检索（语义）和稀疏检索（BM25），使用倒数排名融合（RRF）合并结果：

$$\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + \text{rank}_r(d)}$$

其中 $R$ 为多路检索的结果集列表，$\text{rank}_r(d)$ 为文档$d$在第$r$路结果中的排名，$k$ 通常取60。

**图检索（Graph RAG）**：将文档和实体构建为图结构，支持基于关系的检索和推理。结合知识图谱进行实体扩展和关系遍历，适合多跳推理任务。

### 检索后处理（Post-Retrieval）

**上下文压缩**：

| 策略 | 原理 | 压缩率 | 信息保留 |
|------|------|--------|---------|
| 提取式摘要 | 选择最重要的句子 | 30-50% | 高 |
| 生成式摘要 | LLM生成摘要 | 70-90% | 中 |
| 选择性压缩 | 仅保留与查询相关段落 | 40-60% | 高 |
| LLMLingua | 小模型压缩token | 2-5x | 高 |

**多阶段重排**：

```
粗排（向量相似度Top-100）→ 精排（Cross-Encoder Top-10）→ 多样性重排（去重）
```

### 索引结构演进

**层次化摘要索引（RAPTOR）**：为文档创建多级摘要索引（文档级→章节级→段落级→原始内容），实现从粗到细的层次化检索。

**图索引（GraphRAG）**：

```
原始文档 → 实体抽取 → 关系构建 → 社区检测 → 层次化摘要
                    ↓
查询 → 全局检索（社区摘要）或 局部检索（实体关系）
```

### 自适应检索

| 方法 | 核心机制 | 优势 |
|------|---------|------|
| Self-RAG | LLM自评估：是否需要检索？检索结果是否相关？回答是否有依据？ | 按需检索，避免无关检索 |
| CRAG | 检索质量评估 → 纠正（正确→使用，模糊→Web搜索兜底，不相关→直接生成） | 检索质量保障 |
| FLARE | 生成过程中检测不确定性（低置信度token）→ 动态触发检索 | 实时按需检索 |

### 训练感知RAG

**领域适配微调**：

- **嵌入模型微调**：使用领域数据通过对比学习微调嵌入模型
- **重排模型微调**：使用领域标注数据微调Cross-Encoder

**端到端可训练RAG**：构建可微分的检索-生成管道，联合优化：

$$\mathcal{L} = \alpha \cdot \mathcal{L}_{\text{answer}} + \beta \cdot \mathcal{L}_{\text{retrieval}}$$

## 6. 优势与局限

### 优势

- 检索精度大幅提升，Recall@5提升20-40%
- 支持复杂多跳推理问题
- 自适应检索减少不必要的检索开销
- 图检索支持全局视角理解
- 训练感知优化提升领域适配性

### 局限

- 系统复杂度显著增加，调试和维护成本高
- 多阶段处理增加端到端延迟
- 查询重写/HyDE等引入额外LLM调用，增加成本
- 自适应检索依赖LLM自我评估能力，可能误判
- 图索引构建成本高，维护复杂

## 7. 应用场景

- 复杂多跳问答系统
- 企业级知识库（需要高精度检索）
- 专业领域问答（法律、医疗、金融）
- 多源异构数据检索
- 需要全局理解的长文档分析
- 实时性要求高的对话系统（FLARE）

## 8. 与其他技术关系

- 进阶RAG以 `基础RAG` 为基础
- `向量数据库` 提供存储和ANN检索支持
- `Agent系统` 可将进阶RAG作为工具调用（Agentic RAG）
- `多模态RAG` 扩展到多模态内容
- 图检索与知识图谱技术结合
- `提示工程` 影响查询重写和提示构造质量

## 9. 前沿发展

- **Agentic RAG**：RAG从预定义流程进化为Agent自主决策的动态检索，LLM根据任务需要自主选择检索策略、判断检索质量、决定是否需要多跳
- **GraphRAG**：Microsoft提出的图结构RAG，通过社区检测和层次化摘要支持全局性问题（如"这个数据集的主要主题是什么？"）
- **多模态融合**：进阶RAG技术扩展到多模态领域，支持图文混合检索
- **端到端优化**：从分别优化检索和生成走向联合优化，弥合检索-生成语义鸿沟
- **RAG可解释性**：检索路径可视化、来源置信度评分、检索决策解释
- **轻量化进阶RAG**：通过模型蒸馏和缓存优化，降低进阶RAG的延迟和成本
