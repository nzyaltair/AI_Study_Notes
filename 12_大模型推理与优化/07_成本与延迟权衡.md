# 成本与延迟权衡

## 1. 概述

在大模型推理部署中，成本与延迟是两个核心且相互制约的指标。成本涵盖计算资源、内存使用、能耗和金钱消耗；延迟涉及首 token 延迟（TTFT）、逐 token 延迟（TPOT）和端到端响应时间。AI 应用开发者和系统工程师需要在两者之间找到最优平衡点，以满足业务 SLA 并保证用户体验。

随着推理优化技术的发展（量化、连续批处理、Speculative Decoding、Prefill-Decode 分离），成本-延迟的 Pareto 前沿不断向前推进。但不同应用场景对成本和延迟的敏感度不同，需要针对性的优化策略。

## 2. 发展历史

| 年代 | 里程碑 | 意义 |
|------|--------|------|
| 2020 | 单 GPU 直接部署 | 早期推理方式，成本高 |
| 2022 | 模型并行部署 | 支持超大模型，但利用率低 |
| 2022 | INT8 量化 | 成本减半，精度几乎无损 |
| 2023 | vLLM 连续批处理 | 吞吐量提升 10-20x |
| 2023 | INT4 量化（GPTQ/AWQ）| 70B 模型单卡部署 |
| 2023 | Speculative Decoding | 延迟降低 30-50% |
| 2024 | Prefill-Decode 分离 | 两阶段分别优化 |
| 2024 | FP8 推理 | 硬件原生加速 |
| 2024 | 弹性推理服务 | 按需伸缩，降低闲置成本 |

## 3. 核心概念

### 3.1 关键指标

| 指标 | 定义 | 说明 |
|------|------|------|
| TTFT（Time To First Token） | 首 token 延迟 | Prefill 阶段时间，影响用户体验 |
| TPOT（Time Per Output Token） | 逐 token 延迟 | Decode 阶段每 token 时间 |
| E2E Latency | 端到端延迟 | TTFT + TPOT × 输出长度 |
| Throughput | 吞吐量 | tokens/s，影响成本效率 |
| Cost Efficiency | 成本效率 | tokens/$，单位成本产出 |
| Goodput | 有效吞吐 | 满足 SLA 的有效请求吞吐 |

### 3.2 成本-延迟权衡维度

- **模型选型**：大模型精度高但成本高，小模型成本低但能力弱
- **量化精度**：低精度成本低但精度有损失
- **批处理策略**：大 batch 吞吐高但延迟增加
- **硬件选择**：高端 GPU 延迟低但成本高
- **并行策略**：并行度越高延迟越低但成本和通信开销增加

## 4. 技术原理

### 4.1 模型参数影响

| 模型大小 | 显存（FP16） | 推理速度（tokens/s） | 成本影响 |
|----------|-------------|---------------------|---------|
| 7B | ~13GB | 100-200 | 低 |
| 13B | ~26GB | 50-100 | 中 |
| 33B | ~66GB | 20-50 | 高 |
| 70B | ~140GB | 10-20 | 极高 |

模型越大，精度通常越高，但成本和延迟也越高。

### 4.2 量化对成本的影响

| 量化方案 | 显存节省 | 速度提升 | 精度损失 | 适用场景 |
|----------|---------|---------|----------|---------|
| FP16 | 基准 | 基准 | 无 | 高精度需求 |
| INT8 | ~50% | 1.5-2x | <1% | 通用场景 |
| INT4 | ~75% | 2-3x | 1-3% | 低延迟场景 |
| AWQ | ~75% | 2.5-3.5x | <2% | 高精度+低延迟 |
| FP8 | ~50% | 2-3x | <1% | H100 高性能 |
| GGUF | ~75-90% | 3-4x | 2-5% | 边缘设备 |

### 4.3 调度策略对延迟的影响

**连续批处理**：
- GPU 利用率从 30-40% 提升到 80-90%
- 平均延迟降低 40-60%
- 吞吐量提升 3-4x
- 但高负载时单请求延迟可能增加

**Speculative Decoding**：
- 端到端延迟降低 30-50%
- 吞吐量提升 2-3x
- 成本节省 40-60%
- 但增加 Draft 模型的计算开销

**Prefill-Decode 分离**：
- Prefill 集群优化计算吞吐
- Decode 集群优化带宽利用
- 避免两阶段相互干扰
- 但增加 KV-Cache 传输开销

### 4.4 硬件选择

| GPU 型号 | 显存 | 带宽 | 适用模型 | 成本（$/小时）|
|---------|------|------|---------|--------------|
| A10 | 24GB | 600GB/s | ≤7B | ~1.0 |
| A100 40GB | 40GB | 1.5TB/s | ≤13B | ~2.0 |
| A100 80GB | 80GB | 1.9TB/s | ≤33B | ~3.0 |
| H100 | 80GB | 3.3TB/s | ≤70B（INT4）| ~8.0 |
| H200 | 141GB | 4.8TB/s | ≤70B（FP8）| ~10.0 |

**关键观察**：Decode 阶段是带宽密集型，高带宽 GPU（H100/H200）的性价比远高于低带宽 GPU。

### 4.5 成本-延迟优化公式

- **成本效率**：$\text{CE} = \text{Throughput} / \text{Cost}$（tokens/$）
- **延迟效率**：$\text{LE} = 1 / (\text{Latency} \times \text{Cost})$（1/(ms·$)）
- **性价比**：$\text{Value} = \text{Quality} / (\text{Latency} \times \text{Cost})$

### 4.6 弹性伸缩

- **自动伸缩**：根据请求队列深度动态调整 GPU 实例数
- **峰值-非峰值策略**：非峰值时段降低实例数，节省 40-60% 成本
- **预热策略**：提前加载模型，避免冷启动延迟
- **混合部署**：在线请求用高性能 GPU，离线批处理用低成本 GPU

## 5. 关键方法与模型

### 5.1 场景化方案推荐

| 场景 | 模型 | 量化 | 推理框架 | 调度 | 硬件 |
|------|------|------|---------|------|------|
| 低延迟（P99<100ms）| 7B | INT4/AWQ | TensorRT-LLM | 连续批处理 | H100 |
| 高精度 | >7B | FP8/INT8 | TensorRT-LLM | Speculative | H100 SXM5 |
| 低成本 | 7-13B | INT4/GGUF | llama.cpp/vLLM | 连续批处理 | A100 |
| 超大模型 | >13B | MoE+INT4 | vLLM | 连续批处理 | A100×N |
| 边缘部署 | 7B | GGUF INT4 | llama.cpp | - | 消费级 |
| 批量离线 | 大模型 | INT4 | vLLM | 大 batch | A100 |

### 5.2 推理框架成本对比

| 框架 | 优势 | 成本特点 |
|------|------|---------|
| **vLLM** | 内存利用率高，高并发最优 | 吞吐量比 HF 高 10-20x |
| **TensorRT-LLM** | 延迟最低 | INT8 延迟比 vLLM 低 20-30% |
| **llama.cpp** | 硬件成本低 | 7B 可在 8GB 内存运行 |
| **DeepSpeed** | 支持超大模型 Offloading | 成本降低 20-40%，延迟增加 10-30% |

### 5.3 MoE 模型的成本优势

MoE 架构在成本-延迟权衡中具有独特优势：

- **Mixtral 8x7B**：47B 总参数，13B 激活 → 以 13B 的推理成本获得 47B 的模型能力
- **DeepSeek-V2**：236B 总参数，21B 激活 → 以 21B 的推理成本接近 236B 的质量
- 稀疏激活使大模型能力的推理成本大幅降低

## 6. 优势与局限

**优势**：
- 量化+连续批处理可使成本降低 5-10x
- Speculative Decoding 同时改善延迟和吞吐
- 灵活的硬件配置适应不同 SLA 需求
- 弹性伸缩消除闲置浪费

**局限**：
- 成本与延迟通常需要权衡，无法同时最优
- 量化精度损失在敏感任务中可能不可接受
- 高并发下单请求延迟可能恶化
- 弹性伸缩有冷启动延迟
- MoE 模型需加载所有专家，显存需求大

## 7. 应用场景

- **实时对话**：7B INT4 + vLLM + 连续批处理 + A100，P99 < 500ms
- **高精度 API**：33B INT8 + TensorRT-LLM + Speculative Decoding + H100，P99 < 1s
- **边缘部署**：7B GGUF INT4 + llama.cpp + 消费级 CPU/GPU，延迟 < 2s
- **批量处理**：大模型 + 高吞吐配置，优化单位成本吞吐量
- **弹性服务**：自动弹性伸缩，非峰值时段降低 40-60% 成本
- **Agent 应用**：结构化输出 + 高并发，平衡延迟和成本

## 9. 前沿发展

- **Prefill-Decode 分离架构**：DistServe 等系统将两阶段部署到不同集群，各自优化成本-延迟
- **MoE 推理优化**：针对稀疏 MoE 模型的专用推理引擎，降低显存需求
- **推理时计算扩展**：o1 模式以更多推理时计算换取更高质量，重新定义成本-质量权衡
- **Serverless 推理**：按请求计费，消除闲置成本（如 Modal、Replicate）
- **多模型路由**：根据请求难度路由到不同大小的模型，简单请求用小模型降低成本
- **CXL 池化**：CXL 协议实现 GPU 显存池化，提高资源利用率
- **能效优化**：关注每 token 的能耗成本，推动绿色 AI 推理
