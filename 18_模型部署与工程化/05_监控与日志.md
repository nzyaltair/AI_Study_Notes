# 监控与日志

## 1. 概述

LLM 服务上线后，持续监控性能、成本和质量是运维的核心。传统 Web 服务监控以 QPS 和延迟为主，而 LLM 的自回归生成特性引入了新的维度：首 Token 延迟（TTFT）、每 Token 生成延迟（TPOT）、Token 级计费和内容安全。

监控与日志解决的核心问题：
- **性能保障**：实时发现延迟异常和吞吐瓶颈
- **成本追踪**：Token 级精确计费和成本归因
- **质量监控**：检测模型输出质量和内容安全
- **故障排查**：结构化日志和分布式追踪支持快速定位问题

随着 LangSmith、Phoenix 等 LLM 专用可观测性工具出现，监控从基础设施层延伸到模型行为层，形成覆盖请求链路、模型输出和资源利用的立体监控体系。

## 3. 核心概念

### 3.1 监控维度

| 维度 | 指标 | 说明 |
|:---|:---|:---|
| **性能** | TTFT / TPOT | 首 Token 延迟 + 生成每 Token 延迟 |
| **性能** | QPS / 吞吐量 | 每秒请求数和 Token 生成速率 |
| **资源** | GPU 利用率 / 显存 | GPU 计算和内存使用情况 |
| **成本** | Token 消耗 | 输入/输出 Token 计费统计 |
| **质量** | 错误率 / 内容安全 | 请求失败比例和有害内容拦截率 |

### 3.2 可观测性三支柱

- **指标（Metrics）**：时序数值数据，如 QPS、延迟、GPU 利用率
- **日志（Logs）**：离散事件记录，用于故障排查
- **追踪（Traces）**：请求在分布式系统中的完整调用链路

### 3.3 延迟分位数

P50/P95/P99 反映延迟分布：
- P50：中位数，反映典型用户体验
- P95：95% 请求的延迟上限
- P99：99% 请求的延迟上限，反映长尾用户体验

## 4. 技术原理

### 4.1 监控工具链

| 层级 | 工具 | 作用 |
|:---|:---|:---|
| **指标收集** | Prometheus | 时序数据采集与存储 |
| **可视化** | Grafana | 仪表盘展示与告警 |
| **日志** | ELK Stack / Loki | 日志收集、索引与分析 |
| **链路追踪** | Jaeger / Zipkin | 分布式请求链路追踪 |
| **LLM 专用** | LangSmith / Phoenix | LLM 调用追踪与质量评估 |
| **告警** | AlertManager | 告警规则管理与通知分发 |

### 4.2 结构化日志

LLM 服务的日志需记录请求生命周期的关键信息：

```json
{
    "timestamp": "2026-01-01T00:00:00Z",
    "request_id": "abc123",
    "model": "llama-3-70b",
    "input_tokens": 150,
    "output_tokens": 300,
    "latency_ms": 2500,
    "ttft_ms": 200,
    "status": "success",
    "user_id": "user_456"
}
```

### 4.3 分布式追踪

OpenTelemetry 提供统一的 Trace/Span 模型，关联日志、指标和追踪数据。一次 LLM 请求的典型追踪链：

```
Trace: api-request-123
├── Span: http-server-receive (0.1ms)
├── Span: authentication (0.5ms)
├── Span: model-router (0.3ms)
├── Span: vllm-generate (500ms)
│   ├── Span: tokenizer (10ms)
│   ├── Span: model-inference (480ms)
│   └── Span: detokenizer (10ms)
└── Span: http-server-send (0.2ms)
```

### 4.4 异常检测

- **统计阈值**：均值 $\pm 3\sigma$ 检测异常
- **滑动窗口统计**：在时间窗口 $W$ 内维护请求计数和延迟分布
- **机器学习模型**：基于历史数据训练异常检测模型，识别非 obvious 异常

### 4.5 告警策略

| 告警级别 | 触发条件 | 响应方式 |
|:---|:---|:---|
| P0 紧急 | 服务不可用、错误率 > 10% | 电话 + 短信 + 自动故障转移 |
| P1 严重 | P99 延迟 > 阈值、GPU 利用率 > 95% | 短信 + 自动扩容 |
| P2 警告 | 吞吐量下降 20%、显存 > 90% | 邮件 + 人工关注 |
| P3 信息 | 成本超预算 80% | 邮件 |

## 5. 关键方法与模型

### 5.1 LLM 质量监控

- **输出质量评估**：使用评判模型或人工评估生成质量
- **幻觉检测**：对比生成内容与知识库的一致性
- **内容安全**：检测有害、偏见或不当内容
- **Prompt 漂移**：监控 Prompt 模板变更对输出的影响

### 5.2 数据漂移检测

监控输入数据分布变化，当分布偏离训练数据时触发告警：
- 输入 Token 长度分布变化
- 请求主题分布变化
- 用户群体变化

### 5.3 故障排查方法论

| 问题 | 排查方向 |
|:---|:---|
| 高延迟 | 检查 GPU 利用率、KV-Cache 命中率、批处理配置 |
| OOM | 减小 batch size、启用量化、排查显存泄漏 |
| 质量下降 | 排查数据漂移、模型版本变更、Prompt 模板修改 |
| 服务不可用 | 健康检查失败、自动重启、故障转移 |

## 6. 优势与局限

### 6.1 优势

- **全链路可观测**：从请求到响应的完整追踪
- **实时告警**：问题发生后秒级发现
- **成本透明**：Token 级精确计费
- **数据驱动决策**：基于监控数据优化配置

### 6.2 局限

- **LLM 质量评估难**：生成质量难以自动化评估
- **监控开销**：日志和追踪增加额外计算开销
- **指标过载**：过多指标导致告警疲劳
- **隐私风险**：日志可能包含敏感用户数据
