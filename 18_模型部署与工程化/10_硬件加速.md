# 硬件加速

## 1. 概述

硬件加速是指利用专为并行计算和低精度运算优化的硬件器件（GPU、TPU、NPU、FPGA 等）来加速 AI 模型的训练与推理。与通用 CPU 相比，专用加速器通过大规模并行计算单元、高带宽显存和定制化数据路径，实现了数量级的性能提升。

硬件加速解决的核心问题：
- **计算密度**：深度学习模型包含海量矩阵乘法，CPU 的少量 ALU 无法高效处理
- **内存带宽**：模型参数和激活值的访存开销远大于计算开销，需要高带宽存储
- **能效比**：数据中心推理成本中电力消耗占比高，专用芯片能效比 CPU 高 10-100 倍
- **规模化训练**：千亿参数模型的训练需要数百卡协同，硬件互联成为瓶颈

LLM 推理是**访存密集型**任务（Arithmetic Intensity ≈ 1），而训练则是**计算密集型**任务，两者对硬件的需求侧重点不同。

## 3. 核心概念

### 3.1 计算密度与 Roofline 模型

Roofline 模型描述硬件性能上限与计算访存比的关系：

$$\text{Performance} = \min\left(\text{Peak Compute}, \quad \text{Peak Bandwidth} \times \text{Arithmetic Intensity}\right)$$

其中 Arithmetic Intensity（AI）= FLOPs / Bytes。

- **计算上限**：硬件峰值算力（TFLOPS）
- **带宽上限**：显存带宽 × AI
- **平衡点**：$AI^* = \frac{\text{Peak Compute}}{\text{Peak Bandwidth}}$

以 A100 为例：FP16 峰值 312 TFLOPS，带宽 2.0 TB/s，$AI^* = 156$ FLOP/Byte。LLM 推理的 $AI \approx 1$，远低于平衡点，是访存密集型。

### 3.2 内存层次

AI 加速器的内存层次直接影响性能：

| 层级 | 容量 | 带宽 | 延迟 | 用途 |
|:---|:---|:---|:---|:---|
| 寄存器 | 256 KB/SM | ~20 TB/s | 1 cycle | 即时计算 |
| 共享内存 / L1 Cache | 192 KB/SM | ~10 TB/s | ~30 cycles | 块内数据复用 |
| L2 Cache | 40 MB | ~4 TB/s | ~200 cycles | 跨 SM 数据复用 |
| HBM（全局显存） | 80 GB | 2.0 TB/s | ~400 cycles | 模型权重与 KV-Cache |
| 主机内存 | 1-2 TB | ~100 GB/s | ~10 μs | CPU-GPU 数据传输 |

FlashAttention 的核心优化就是通过分块计算（tiling）最大化利用 SRAM，减少 HBM 访问。

### 3.3 Tensor Core

NVIDIA GPU 中的专用矩阵计算单元，执行矩阵乘加（MMA）操作：

```
D = A × B + C   (A: m×k, B: k×n, C: m×n)
```

- 一个 Tensor Core 在一个时钟周期内执行 $16 \times 8 \times 16$ 或类似维度的矩阵乘加
- A100 的 432 个 Tensor Core 提供 312 TFLOPS (FP16) 算力
- H100 的 Transformer Engine 自动选择 FP8/FP16 精度

### 3.4 数据格式与精度

| 格式 | 位宽 | 指数 | 尾数 | 动态范围 | 适用场景 |
|:---|:---|:---|:---|:---|:---|
| FP64 | 64 | 11 | 52 | ~10^308 | 科学计算 |
| FP32 | 32 | 8 | 23 | ~10^38 | 训练默认精度 |
| TF32 | 19 | 8 | 10 | ~10^38 | A100 训练加速 |
| BF16 | 16 | 8 | 7 | ~10^38 | LLM 训练标准 |
| FP16 | 16 | 5 | 10 | ~65504 | 训练/推理 |
| FP8 (E4M3) | 8 | 4 | 3 | ~448 | H100 推理加速 |
| FP8 (E5M2) | 8 | 5 | 2 | ~57344 | H100 训练加速 |
| INT8 | 8 | — | — | [-128, 127] | 量化推理 |
| INT4 | 4 | — | — | [-8, 7] | LLM 量化推理 |
| FP4 | 4 | 2 | 1 | ~6 | B200 极致推理 |

BF16 相比 FP16 的优势：保留 FP32 的 8 位指数，动态范围相同，训练不易溢出。

## 4. 技术原理

### 4.1 GPU 并行计算模型

CUDA 编程模型将计算组织为三级层次：

```
Grid → Block → Thread
  ↓       ↓       ↓
 GPU     SM    CUDA Core
```

- **Thread**：最小执行单元，拥有私有寄存器
- **Block**：线程块，映射到一个 SM（Streaming Multiprocessor），块内通过共享内存通信和 `__syncthreads()` 同步
- **Grid**：所有线程块的集合，在 GPU 上全局调度

**矩阵乘法的并行映射**（以 $C = A \times B$ 为例）：

```
每个 Block 计算输出矩阵 C 的一个 B×B 子块
  → 从全局内存加载 A 和 B 的对应子块到共享内存
  → 每个 Thread 计算子块内一个或多个元素
  → 使用共享内存进行块内规约
```

### 4.2 算子融合

将多个连续算子合并为一个自定义 kernel，减少中间结果的显存读写：

**未融合**（PyTorch 默认 eager 模式）：
```
Y₁ = MatMul(X, W₁)    → 写 HBM → 读 HBM
Y₂ = LayerNorm(Y₁)     → 写 HBM → 读 HBM
Y₃ = GELU(Y₂)          → 写 HBM → 读 HBM
Y₄ = MatMul(Y₃, W₂)    → 读 HBM
```
HBM 读写次数：6 次（3 写 + 3 读）

**融合后**：
```
Y₄ = FusedMLP(X, W₁, W₂)  → 仅读 X, W₁, W₂，仅写 Y₄
```
HBM 读写次数：4 次（3 读 + 1 写），中间结果保留在寄存器/共享内存中。

**融合策略**：
- **水平融合**：将多个独立算子合并（如多 Head 的 QKV 投影）
- **垂直融合**：将流水线上的连续算子合并（如 MatMul + Bias + GELU + LayerNorm）
- **模板化融合**：Triton、TVM 等工具通过模板自动生成融合 kernel

### 4.3 FlashAttention

FlashAttention 是最重要的硬件感知算法优化之一，通过分块计算避免实例化完整 $n \times n$ 注意力矩阵：

**标准 Attention**：
$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d}}\right) V$$

- 需要存储 $n \times n$ 的注意力矩阵，HBM 读写 $\mathcal{O}(n^2 d)$

**FlashAttention**：
1. 将 $Q, K, V$ 分为大小为 $B_r \times B_c$ 的块
2. 逐块计算 $S_{ij} = Q_i K_j^T / \sqrt{d_k}$
3. 使用**在线 softmax** 增量更新归一化因子
4. 累积输出 $O_i += P_{ij} V_j$

- HBM 访问降为 $\mathcal{O}(n^2 d / M)$（$M$ 为 SRAM 大小）
- 空间复杂度从 $\mathcal{O}(n^2)$ 降为 $\mathcal{O}(n)$
- 实测 2-4 倍加速

**在线 Softmax** 是关键创新——无需存储完整 softmax 矩阵，通过维护运行最大值 $m$ 和归一化因子 $l$ 增量更新：

$$\tilde{m}_{new} = \max(\tilde{m}_{old}, m_{block}), \quad \tilde{l}_{new} = e^{(\tilde{m}_{old} - \tilde{m}_{new})} \tilde{l}_{old} + e^{(m_{block} - \tilde{m}_{new})} l_{block}$$

### 4.4 KV-Cache 的硬件影响

LLM 自回归推理中，KV-Cache 随序列长度线性增长，是显存占用的主要来源：

$$\text{KV-Cache Size} = 2 \times n_{layers} \times n_{kv\_heads} \times d_{head} \times seq\_len \times \text{dtype\_size}$$

以 LLaMA-2-70B 为例（FP16）：
- 80 层 × 8 KV heads × 128 dim × 2 (K+V) × 2 bytes = 327,680 bytes/token ≈ 320 KB/token
- 序列长度 4096 时，KV-Cache 占用约 1.3 GB
- 批处理 64 个请求时，KV-Cache 占用约 83 GB，超过模型权重（140 GB 的 59%）

这意味着**显存容量而非算力是 LLM 推理的瓶颈**，推动了 PagedAttention、GQA/MQA、KV-Cache 量化等技术。

### 4.5 模型编译与图优化

模型编译器将高层计算图优化为硬件特定的高效代码：

```
PyTorch 模型 → 计算图 → 图优化 → 内核选择 → 代码生成 → 可执行引擎
```

**图优化**：
- 常量折叠：预计算可静态确定的子图
- 死代码消除：移除不影响输出的算子
- 算子融合：合并相邻算子
- 布局转换：优化数据排布（NHWC vs NCHW）

**主要编译器**：

| 编译器 | 来源 | 特点 |
|:---|:---|:---|
| XLA | Google | TensorFlow/JAX 编译后端，算子融合与内核自动调优 |
| TensorRT | NVIDIA | NVIDIA 推理优化引擎，支持 INT8/FP8 量化 |
| TVM | Apache | 跨平台编译器，AutoTVM 自动调优 |
| TorchInductor | Meta | PyTorch 2.0 编译后端，Triton kernel 生成 |
| Triton | OpenAI | Python DSL 编写 GPU kernel，无需 CUDA C |
| ONNX Runtime | Microsoft | 跨平台推理引擎，Execution Provider 架构 |

### 4.6 GPU 互联技术

多 GPU 训练需要高速数据交换，互联带宽直接决定并行效率：

| 互联技术 | 带宽 | 延迟 | 特点 |
|:---|:---|:---|:---|
| PCIe 4.0 x16 | 32 GB/s | ~6 μs | 通用标准，成本低 |
| PCIe 5.0 x16 | 64 GB/s | ~5 μs | 最新标准 |
| NVLink 3.0 | 50 GB/s/GPU | ~1.5 μs | V100 时代 |
| NVLink 4.0 | 100 GB/s/GPU | ~1 μs | A100，8 卡全互联 |
| NVLink 5.0 | 200 GB/s/GPU | ~0.8 μs | H100，72 卡互联 |
| NVSwitch | 900 GB/s 聚合 | — | DGX A100/H100 内全互联 |
| InfiniBand HDR | 200 Gb/s | ~1 μs | 节点间互联 |
| InfiniBand NDR | 400 Gb/s | ~0.5 μs | 最新节点间互联 |

**张量并行**对互联带宽要求最高（每层通信），适合 NVLink 互联的同节点 GPU；**流水线并行**通信量小但存在气泡，适合跨节点部署。

## 6. 关键优化技术

### 6.1 混合精度训练

利用低精度格式加速计算，同时保持训练稳定性：

```
FP32 Master Weights → FP16/BF16 Forward/Backward → FP32 Gradient Update
```

- **权重主副本**：FP32 存储，避免累积舍入误差
- **前向/反向**：FP16/BF16 计算，利用 Tensor Core 加速
- **梯度缩放**（FP16）：$\text{loss}_{scaled} = \text{loss} \times S$，防止小梯度下溢
- **BF16 无需缩放**：动态范围与 FP32 相同

### 6.2 FP8 训练与推理

H100 引入 FP8 支持，两种格式分别适用于不同场景：

- **E4M3**（4 位指数，3 位尾数）：精度更高，适合前向传播
- **E5M2**（5 位指数，2 位尾数）：动态范围更大，适合反向梯度

FP8 训练流程：
```
权重: E4M3 存储 → 前向计算 (E4M3) → 激活: E4M3
梯度: E5M2 反向计算 → 梯度缩放 → FP32 权重更新
```

### 6.3 推理量化部署

| 量化方案 | 精度 | 硬件支持 | 加速比 | 适用场景 |
|:---|:---|:---|:---|:---|
| FP16 | 16-bit | 所有 GPU | 1×（基准） | 通用推理 |
| BF16 | 16-bit | A100+ | 1× | 训练/推理一致 |
| INT8 (W8A8) | 8-bit | T4+ | 2× | 通用量化推理 |
| SmoothQuant | 8-bit | A100+ | 2× | 激活值异常值平滑 |
| GPTQ | 4-bit (权重) | A100+ | 3-4× | 后训练权重量化 |
| AWQ | 4-bit (权重) | A100+ | 3-4× | 激活感知量化 |
| FP8 | 8-bit | H100+ | 2× | H100 原生支持 |
| FP4 | 4-bit | B200 | 4-8× | B200 原生支持 |

### 6.4 推测解码的硬件利用

推测解码利用小模型生成候选 Token，大模型并行验证。从硬件角度：
- **减少串行访存**：大模型每次前向只读一次权重，验证 $k$ 个 Token
- **提高计算密度**：将 $AI \approx 1$ 的逐 Token 生成变为 $AI \approx k$ 的批量验证
- **硬件利用率提升**：大模型的 Tensor Core 利用率从 ~5% 提升到 ~30%

## 7. 优势与局限

### 7.1 优势

- **数量级加速**：GPU 相比 CPU 在矩阵运算上有 10-100 倍加速
- **能效比**：专用芯片单位功耗的计算量远高于通用 CPU
- **规模化**：多卡集群支持千亿参数模型训练
- **精度灵活**：从 FP64 到 FP4，支持精度-性能-显存的多维权衡
- **生态成熟**：CUDA 生态覆盖几乎所有深度学习框架和工具

### 7.2 局限与挑战

- **访存瓶颈**：LLM 推理是访存密集型，算力利用率低（~5%）
- **显存容量限制**：大模型权重 + KV-Cache 超出单卡显存
- **互联瓶颈**：跨节点通信延迟限制大规模并行效率
- **供应链风险**：高端 GPU 产能受限，地缘政治影响供应
- **能耗与散热**：H100 功耗 700W，B200 功耗 1000W，数据中心散热挑战大
- **成本高昂**：H100 单卡价格约 3 万美元，集群投入达数亿美元
- **编程复杂度**：手写 CUDA kernel 门槛高，依赖 Triton 等高级工具降低难度

## 10. 前沿发展

- **FP4 推理**：B200 的 FP4 格式将推理速度推向新高度，精度损失可控
- **存算一体**：在存储单元内直接执行计算，消除冯·诺伊曼瓶颈
- **光子计算**：利用光信号进行矩阵乘法，理论能效比远超电子芯片
- **可重构芯片**：FPGA-like 架构，动态适配不同模型结构
- **3D 芯片堆叠**：通过 TSV 技术垂直堆叠计算和存储层，缩短数据路径
- **超大规模式加速器**：Cerebras WSE 等整晶圆级芯片，单芯片集成万亿晶体管
- **液冷与浸没式冷却**：支持 1000W+ TDP 芯片的散热方案
- **CXL 互联**：Cache 一致性互联协议，实现 CPU-GPU-存储统一寻址
- **NPU 大模型训练**：华为昇腾等国产 NPU 逐步支持大规模 LLM 训练
- **量子-AI 混合加速**：量子计算处理特定子问题的潜力探索
