# 成本控制与配额

## 背景与发展

随着[[大语言模型]]规模从亿级增长到千亿级，推理成本快速攀升。2023 年前，LLM 推理主要依赖按需 GPU 实例，成本高昂且利用率低。vLLM 的 PagedAttention 将吞吐提升 2-4 倍，连续批处理和量化技术进一步降低单 Token 成本。云厂商陆续推出 Spot 实例、预留实例和 Serverless 推理等多种计费模式。多级配额管理和模型路由策略成为企业级 LLM 平台控制成本的核心手段。

## 核心思想

在性能、可用性和预算之间取得平衡。通过模型层面优化（量化、蒸馏）、推理层面优化（连续批处理、推测解码）、架构层面优化（自动扩缩容、冷热分离）和运营层面优化（成本归因、预算管理）四个层次系统性降低总成本。配额管理通过多级限流和资源隔离确保公平性和可控性。

## 技术原理

### 成本结构

| 成本类型 | 组成 | 优化手段 |
|----------|------|----------|
| **计算成本** | GPU 实例费用 | Spot 实例、预留实例、弹性伸缩 |
| **推理成本** | Token 生成计算 | 量化、连续批处理、推测解码 |
| **存储成本** | 模型权重存储 | 模型分片、冷热分层存储 |
| **带宽成本** | 跨区传输、公网出口 | 就近部署、CDN 加速 |

### GPU 实例选型

| 实例类型 | GPU | 显存 | 按需价格（$/h） | Spot 折扣 |
|----------|-----|------|-----------------|-----------|
| AWS g5.xlarge | A10G | 24GB | ~1.51 | ~70% |
| AWS p4d.24xlarge | A100×8 | 80GB×8 | ~32.77 | ~70% |
| GCP a2-highgpu-1g | A100 | 40GB | ~4.00 | ~70% |
| GCP a3-highgpu-8g | H100×8 | 80GB×8 | ~32.00 | ~70% |

### 成本优化技术

**模型量化**：INT4 量化减少 75% 模型大小，推理速度提升 2-4 倍；INT8 量化减少 50%，速度提升 1.5-2 倍。GPTQ 和 AWQ 是主流的后训练量化方法。

**推测解码**（Speculative Decoding）：小模型生成候选 Token，大模型并行验证。若候选 Token 被接受，则大模型无需逐步自回归生成，可将推理速度提升 2-3 倍。

**模型路由**：简单任务路由到 7B 模型，复杂任务路由到 70B 模型，可降低 50%+ 成本。

**请求缓存**：缓存高频请求的响应，减少重复计算，可消除 30%+ 的重复请求。

### 配额与限流

**多级配额**：
- 用户级：每用户 QPS 限制、每日 Token 配额
- 租户级：多租户资源隔离与优先级管理
- 系统级：整体 QPS 上限、GPU 资源总量控制

**令牌桶限流**：桶容量 $C$，填充速率 $r$，允许突发流量。时间 $\Delta t$ 内可处理的最大请求数为 $\min(C, C_{\text{current}} + r \cdot \Delta t)$。

**资源隔离**：
- 物理隔离：不同租户使用独立硬件
- 逻辑隔离：通过软件调度实现资源分配
- 混合隔离：关键租户物理隔离，普通租户逻辑隔离

### 成本归因

按用户/租户/模型维度追踪 Token 消耗和计算成本，生成成本报告。关键指标：$/Token、$/Request、GPU 利用率。

## 发展演进

按需实例 → Spot/预留实例 → Token 级计费 → Serverless LLM → 边缘-云协同推理 → 绿色 AI

## 关键算法·模型

- **令牌桶算法**：以速率 $r$ 填充令牌至容量 $C$，请求消耗令牌，公式 $n(t) = \min(C, n(t_0) + r(t - t_0))$
- **最大最小公平算法**：多租户资源分配中，先满足最小需求，再将剩余资源按比例分配
- **推测解码**：Draft 模型生成 $k$ 个候选 Token，Target 模型并行计算概率，接受概率 $p = \min(1, p_{\text{target}}/p_{\text{draft}})$ 的 Token

## 应用场景

- **LLM API 平台**：多租户成本分摊和配额管理
- **企业 AI 平台**：部门级预算控制和资源调度
- **边缘-云协同**：边缘处理简单请求降低云端成本
- **流量波动场景**：弹性伸缩 + Spot 实例应对峰谷
