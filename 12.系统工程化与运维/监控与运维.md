# 监控与运维

## 背景

LLM服务上线后，持续监控性能、成本和安全状况是运维的核心。LLM的特殊性（自回归生成、长尾延迟、token计费）使其监控需求与传统Web服务不同。

## 核心思想

建立多维监控体系（性能、成本、安全、质量），配合告警和自动化响应，确保LLM服务稳定、高效、安全运行。

## 技术原理

### 监控指标体系

| 维度 | 指标 | 说明 |
|------|------|------|
| **性能** | P50/P95/P99延迟 | 首 token 延迟(TTFT) + 生成延迟 |
| **性能** | QPS/吞吐量 | 每秒请求数和token生成速率 |
| **资源** | GPU利用率/显存 | GPU资源使用情况 |
| **成本** | Token消耗 | 输入/输出token计费统计 |
| **成本** | API调用量 | 请求次数和并发数 |
| **质量** | 错误率 | 请求失败比例 |
| **质量** | 内容安全 | 有害内容拦截率 |

### 监控工具栈

| 层级 | 工具 | 作用 |
|------|------|------|
| **指标收集** | Prometheus | 时序数据采集 |
| **可视化** | Grafana | 仪表盘展示 |
| **日志** | ELK Stack | 日志收集与分析 |
| **链路追踪** | Jaeger/Zipkin | 请求链路追踪 |
| **LLM专用** | LangSmith/Phoenix | LLM调用追踪与评估 |
| **告警** | AlertManager | 告警规则与通知 |

### 版本管理与回滚

- **模型版本**：MLflow/Weights & Biases管理模型版本
- **配置版本**：Git管理模型配置和提示模板
- **灰度发布**：逐步切换流量到新版本
- **快速回滚**：异常时秒级回退到上一个稳定版本
- **A/B测试**：并行运行多版本对比效果

### 成本控制

| 策略 | 方法 | 效果 |
|------|------|------|
| **模型路由** | 简单任务用小模型，复杂任务用大模型 | 降50%+成本 |
| **缓存** | 缓存常见请求的响应 | 降30%重复请求 |
| **批量处理** | 非实时请求合并处理 | 提升GPU利用率 |
| **量化** | INT8/INT4量化减少计算 | 降2-4x推理成本 |
| **弹性伸缩** | 低谷期缩容 | 降闲置成本 |
| **Token优化** | 压缩提示、限制输出长度 | 降token消耗 |

### 故障排查

- **高延迟**：检查GPU利用率、KV-Cache大小、批处理配置
- **OOM**：减少batch size、启用量化、检查内存泄漏
- **质量下降**：数据漂移、模型版本问题、提示变更
- **服务不可用**：健康检查、自动重启、故障转移

## 发展演进

传统监控（CPU/内存） → APM → LLM专用监控 → 全链路可观测性 → 自愈系统
