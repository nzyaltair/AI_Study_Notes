# 常见问题与算子兼容

## 背景与发展

[[ONNX]] 转换过程中最频繁出现的问题集中在三个方面：算子不兼容、数据类型不匹配、内存不足。这些问题随着 ONNX Opset 版本演进和工具链成熟有所缓解，但大模型（数十亿参数）的转换仍面临严峻挑战。掌握系统化的诊断和解决方案是高效完成模型部署的前提。

## 核心思想

问题排查遵循**诊断-定位-解决**三步法：先用 `onnx.checker` 和 ONNX Runtime 验证模型完整性，再用 Netron 可视化或代码遍历定位具体问题节点，最后根据算子类型选择合适的解决方案（Opset 升级、算子替换、扩展库或自定义实现）。

## 技术原理

### 算子兼容性诊断

```python
import onnx
from collections import Counter

model = onnx.load("model.onnx")
op_counts = Counter([node.op_type for node in model.graph.node])
print("算子统计:", op_counts)

# 检查已知问题算子
problematic = {'Unique', 'RepeatInterleave', 'GridSample', 'ScatterND'}
issues = [op for op in op_counts if op in problematic]
```

**常见不兼容算子速查**：

| 算子 | 状态 | 解决方案 |
|------|------|---------|
| `aten::unique` | 核心不支持 | onnxruntime-extensions |
| `aten::repeat_interleave` | Opset<14 不支持 | 重构为 expand+reshape |
| `aten::grid_sample` | 实验性 | onnxruntime-extensions |
| `aten::scatter_*` | 部分支持 | 使用 ScatterElements/ScatterND |
| `Einsum` | Opset 12+ 部分 | 替换为 MatMul+Reshape |
| `tf.ragged.*` | 不支持 | 转为 dense tensor + padding |

### 解决方案策略

**方案1：提升 Opset 版本**

```python
torch.onnx.export(model, dummy_input, "model.onnx", opset_version=14)
```

**方案2：onnxruntime-extensions 加载扩展算子**

```python
from onnxruntime_extensions import get_library_path
import onnxruntime as ort

sess_options = ort.SessionOptions()
sess_options.register_custom_ops_library(get_library_path())
session = ort.InferenceSession("model.onnx", sess_options)
```

**方案3：算子替换（模型重构）**

将 `repeat_interleave` 替换为等价的 `expand` + `reshape`：

```python
# 原始（不兼容）
# x.repeat_interleave(repeats, dim=1)

# 重构后（兼容）
shape = list(x.shape)
shape[1] = shape[1] * repeats
return x.unsqueeze(2).expand(*shape[:2], repeats, *shape[2:]).reshape(shape)
```

**方案4：ONNX GraphSurgeon 子图替换**

```python
import onnx_graphsurgeon as gs
import onnx

graph = gs.import_onnx(onnx.load("model.onnx"))
for node in graph.nodes:
    if node.op == 'Einsum':
        # 替换为 MatMul
        matmul = gs.Node(op="MatMul", name=node.name+"_matmul",
                         inputs=[node.inputs[0], node.inputs[1]],
                         outputs=[node.outputs[0]])
        graph.nodes.append(matmul)
        graph.nodes.remove(node)
graph.cleanup()
onnx.save(gs.export_onnx(graph), "model_fixed.onnx")
```

### 数据类型匹配

ONNX Runtime 主要支持 `float32` 和 `float16`，不原生支持 `float64`。常见问题及解决方案：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| PyTorch float64 输出 | Python float 默认 float64 | `model.float()` + `input.float()` |
| TF shape 为 int64 | TF 形状操作输出 int64 | 添加 `tf.cast(x, tf.int32)` |
| 常量类型错误 | `torch.tensor(2.0)` 隐式类型 | `torch.tensor(2.0, dtype=torch.float32)` |
| 已有模型类型错误 | 导出后才发现 | 用 GraphSurgeon 插入 Cast 节点 |

**ONNX 数据类型映射**：

| 框架类型 | ONNX 类型 | 支持情况 |
|---------|----------|---------|
| float32 | FLOAT (1) | 完全支持，推荐 |
| float16 | FLOAT16 (10) | GPU 支持 |
| float64 | DOUBLE (11) | 不支持 |
| int32 | INT32 (6) | 支持 |
| int64 | INT64 (7) | 仅部分 ops |

### 内存优化策略

**大模型导出 OOM 解决方案**：

1. **CPU 导出**：强制 `model.cpu()` 和 `dummy_input.cpu()`，避免 GPU 内存限制
2. **增量导出**：将大模型拆分为子模块分别导出，再合并
3. **关闭常量折叠**：`do_constant_folding=False`，节省导出时内存峰值
4. **外部数据格式**：权重存储在 `.data` 文件中，突破 protobuf 2GB 限制

**文件大小优化**：

```python
from onnxruntime.quantization import quantize_dynamic, QuantType

# FP16 转换：文件减半，GPU 推理加速
# INT8 动态量化：文件减少 75%，CPU 推理加速 2-3x
quantize_dynamic("model_fp32.onnx", "model_int8.onnx",
                 weight_type=QuantType.QInt8, per_channel=True)
```

| 优化方法 | 文件大小减少 | 精度影响 | 推理加速 |
|---------|-------------|---------|---------|
| FP16 | 50% | <0.5% | 1.5-2x |
| INT8 动态量化 | 75% | ~1% | 2-3x |
| INT8 静态量化 | 75% | 1-2% | 3-4x |
| 移除未使用初始器 | 5-30% | 0% | 无 |

### 动态维度常见错误

**错误1：Rank mismatch** — 确保所有输入在导出时具有相同的 rank。

**错误2：常量折叠干扰动态 shape** — 对于复杂动态 shape，尝试 `do_constant_folding=False`。

**错误3：Opset 版本过低** — dynamic_axes 需 Opset >= 11，推荐 13 或 14。

## 发展演进

早期 ONNX 算子覆盖不足，大量模型需要自定义算子实现。随着 Opset 版本递增和 onnxruntime-extensions 库的丰富，常见不兼容算子逐步得到支持。PyTorch 2.x 引入的 `torch.export` 和 TorchDynamo 提供了新的导出路径，有望进一步减少算子兼容性问题。

## 关键算法·模型

- **onnx-simplifier（onnxsim）**：通过形状推断和常量折叠简化计算图，消除冗余 Identity 节点
- **ONNX GraphSurgeon**：图级别编辑工具，支持节点增删改、子图替换、常量折叠
- **IOBinding 零拷贝**：ONNX Runtime 的 IOBinding 机制，避免 GPU↔CPU 间的数据拷贝
- **内存模式优化**：`enable_mem_pattern=True` 预分配内存块，减少推理时动态分配开销

## 应用场景

- **大模型转换**：LLM 等数十亿参数模型的分层导出和内存管理
- **边缘设备部署**：INT8 量化 + 剪枝，将模型压缩到设备内存限制内
- **生产环境故障排查**：快速定位转换失败原因，选择最小成本修复方案
- **跨框架迁移**：从 PyTorch 迁移到 TensorFlow 或反向时处理算子差异
