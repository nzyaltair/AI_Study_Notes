# 长上下文扩展

## 1. 概述

长上下文扩展技术旨在使大语言模型能高效处理超长序列输入（从 2K 到 1M+ token），同时控制推理成本。随着应用需求增长——长文档分析、多轮对话、代码库理解、RAG 检索增强——模型的有效上下文长度成为核心能力指标。

LLM 推理时，长上下文带来三大挑战：
- [[01_KV-Cache机制|KV-Cache]] 显存随序列长度线性增长
- 注意力计算 $O(n^2)$ 复杂度
- 推理延迟随序列长度增加

长上下文扩展从推理优化角度解决问题，与模型架构层面的长上下文机制（位置编码设计、记忆机制）互补。核心技术包括位置编码外推、KV-Cache 管理、注意力计算优化和分布式长序列处理。

## 3. 核心概念

### 3.1 上下文长度的瓶颈

| 瓶颈 | 原因 | 影响 |
|------|------|------|
| KV-Cache 显存 | 随序列长度线性增长 | batch=1 时 128K 上下文可能占用数十 GB |
| 注意力计算 | $O(n^2)$ 复杂度 | Prefill 阶段计算量急剧增长 |
| 位置编码外推 | 训练时未见过的位置 | 长度泛化能力下降 |
| 注意力稀释 | 长序列中注意力分散 | "Lost in the Middle" 问题 |

### 3.2 长上下文的两层优化

- **架构层**：位置编码外推、注意力模式设计、记忆机制
- **推理层**：KV-Cache 管理、注意力计算优化、分布式处理

本文聚焦推理优化层，架构层参见 [[10_大语言模型核心架构]] 的位置编码与记忆机制。

## 4. 技术原理

### 4.1 位置编码外推

RoPE（旋转位置编码）是大模型最常用的位置编码，但在训练长度外推时性能下降。外推方法通过修改 RoPE 频率实现长度扩展。

#### 位置插值（Position Interpolation, PI）

**原理**：将超出训练长度的位置缩放到训练范围内：

$$\theta'_i = \theta_i / s, \quad s = L_{\text{target}} / L_{\text{train}}$$

- 简单有效，扩展 4-8x
- 需少量长序列微调
- 缺点：所有频率统一缩放，高频信息损失

#### NTK-aware

**原理**：调整 RoPE 基频 $\theta$，对高频和低频差异化处理：

$$\theta' = \theta \cdot s^{d/(d-2)}$$

- 高频部分保持不变（保留细粒度位置信息）
- 低频部分进行插值（扩展范围）
- 扩展 8-32x，无需微调即可保持质量

#### YaRN（Yet another RoPE extensioN）

**原理**：分频段插值 + 温度缩放：

1. 将 RoPE 频率分为多个波段
2. 高频波段保持原始，低频波段插值，中间波段平滑过渡
3. 对注意力分数施加温度缩放补偿

- 扩展 16-64x
- 少量微调即可获得高质量外推
- LLaMA-2 等模型的标准外推方法

#### LongRoPE

**原理**：通过进化搜索寻找每个频率维度的最优缩放因子：

- 不假设统一的缩放模式
- 搜索空间覆盖所有可能的非均匀插值
- 支持 128x+ 扩展（2M 上下文）
- Microsoft 2024 提出

### 4.2 KV-Cache 管理

| 技术 | 原理 | 效果 |
|------|------|------|
| PagedAttention | 分页管理 KV-Cache | 减少碎片，吞吐提升 2-4x |
| KV-Cache 压缩 | 淘汰不重要的 KV | StreamingLLM 支持无限长度 |
| KV-Cache 量化 | FP16 → INT8/INT4 | 显存减少 50-75% |
| KV-Cache 卸载 | 溢出到 CPU 内存/SSD | 突破 GPU 显存限制 |
| MLA | 压缩到潜在向量 | DeepSeek-V2 减少 93.3% |

### 4.3 缓存驱逐策略

长上下文场景下需驱逐部分 KV 以控制显存：

- **StreamingLLM**：保留初始 token（Attention Sink）+ 最近 token 的 KV
  - 初始 token 在注意力中承担"汇聚"作用
  - 移除初始 token 会导致注意力分数坍塌
  - 实现恒定内存下的无限长度流式推理

- **H2O**（Heavy-Hitter Oracle）：基于注意力分数动态淘汰不重要的 KV

- **滑动窗口**：只保留最近 $W$ 个 token 的 KV
  - 简单高效，Mistral 采用 $W=4096$
  - 丢失早期上下文

- **SnapKV**：在 Prefill 阶段预筛选重要 token，压缩 KV-Cache

### 4.4 注意力计算优化

| 方法 | 复杂度 | 原理 | 适用场景 |
|------|--------|------|---------|
| FlashAttention | $O(n^2)$ IO 优化 | 分块计算，减少 HBM 读写 | 通用加速 2-4x |
| Sliding Window | $O(n \cdot w)$ | 仅关注局部窗口 $w$ | Mistral |
| GQA/MQA | 减少 KV 头 | 共享 K/V 头组 | 推理加速 |
| Ring Attention | 分布式 $O(n^2/k)$ | 序列分片到 $k$ 个 GPU | 100K+ 序列 |
| InfLLM | 分块注意力 | 将长序列分块，仅关注重要块 | 400K+ |

### 4.5 分布式长序列处理

**Ring Attention**：
- 将长序列分片到多个 GPU
- 各 GPU 计算本地注意力块
- 通过环形通信传递 KV 块
- 计算与通信重叠，支持 100K+ 序列

**Sequence Parallelism**：
- 将序列维度切分到多 GPU
- 适合 Prefill 阶段的并行化
- 与张量并行互补

### 4.6 Chunked Prefill

将长 prompt 分块预填充，避免单步 OOM：

- 将长 prompt 分为大小为 $C$ 的块
- 逐块 Prefill，增量更新 KV-Cache
- 允许 Prefill 与 Decode 交错调度
- vLLM 等框架已默认支持

### 4.7 推理质量保障

- **Attention Sink**：保留初始 token 的 KV，防止注意力分数坍塌
- **长度泛化微调**：在长序列数据上微调，提升外推质量
- **上下文压缩**：通过摘要/提取压缩长上下文
- **位置编码感知**：确保外推方法与注意力模式的兼容性

## 6. 优势与局限

**优势**：
- 使模型能处理超长文档和多轮对话
- 位置外推方法可大幅扩展训练长度
- KV-Cache 管理有效控制显存增长
- 分布式处理突破单 GPU 限制

**局限**：
- "Lost in the Middle"：长序列中间信息容易被忽略
- 外推质量随长度增加而下降
- 缓存驱逐可能丢失关键上下文
- 分布式处理增加通信开销和系统复杂度
- 长上下文推理成本随长度增长

## 7. 应用场景

- **长文档分析**：学术论文、法律文档、技术报告的全文理解
- **多轮对话**：保持长对话历史的上下文一致性
- **代码理解**：大型代码库的全局分析
- **RAG 增强**：将更多检索结果纳入上下文
- **多文档摘要**：同时处理多份文档进行综合摘要
- **Agent 长期记忆**：Agent 任务中的长程上下文保持

- **无限上下文**：StreamingLLM 等技术追求恒定内存下的无限长度推理

- **层级注意力**：局部精细注意力 + 全局粗粒度注意力的分层设计

- **原生长上下文训练**：从预训练阶段就支持超长序列，而非外推
