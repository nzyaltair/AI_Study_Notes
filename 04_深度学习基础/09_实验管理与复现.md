# 实验管理与复现

## 1. 概述

深度学习实验具有随机性强（初始化、数据shuffle）、超参数多、训练周期长、计算资源消耗大等特点。缺乏系统化的实验管理会导致：结果不可复现、实验对比困难、团队协作混乱、决策缺乏可追溯性。实验管理是将深度学习从"炼丹"推向"科学工程"的关键实践。

- **核心目标**：使每个实验可追踪、可复现、可比较、可审计
- **管理对象**：代码、数据、环境、超参数、指标、模型权重、分析结论

## 2. 发展历史

- **早期（2012-2015）**：手动记录超参数到Excel/文本文件，模型手动保存
- **TensorBoard期（2015-2018）**：TensorFlow自带可视化，本地指标监控
- **平台化期（2018-2021）**：Weights & Biases (W&B)、MLflow、Neptune.ai 等专业实验管理平台兴起
- **MLOps集成期（2021-至今）**：实验管理融入CI/CD流水线，与模型注册表、特征存储、模型监控等组件联动

## 3. 核心概念

### 3.1 实验可复现性的三个层次

| 层次 | 内容 | 难度 |
|:---|:---|:---|
| **代码可复现** | 相同代码+相同数据产生相同结果 | 低（随机种子固定即可） |
| **环境可复现** | 在不同机器上产生相同结果 | 中（需Docker/Conda锁定依赖） |
| **结论可复现** | 在不同数据集/设置下结论依然成立 | 高（统计显著性+多轮重复） |

### 3.2 实验全生命周期

```
实验设计 → 超参数配置 → 训练执行 → 指标追踪
    → 结果分析 → 结论记录 → 模型注册/丢弃 → 下一轮迭代
```

### 3.3 消融实验（Ablation Study）

通过**控制变量法**逐一移除/修改模型的组件，量化每个组件的贡献：

```
Baseline (完整模型):  Accuracy = 90.0%
- 移除数据增强:        Accuracy = 87.2%  (Δ = -2.8%)
- 移除Dropout:          Accuracy = 88.1%  (Δ = -1.9%)
- 替换SwiGLU为ReLU:    Accuracy = 89.3%  (Δ = -0.7%)
→ 数据增强是贡献最大的组件
```

消融实验是学术论文中论证设计合理性的标准方法。

## 4. 技术原理

### 4.1 随机种子的完全固定

```python
import random
import numpy as np
import torch

def set_seed(seed: int):
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)

    # 确保GPU上的确定性（可能影响性能）
    torch.backends.cudnn.deterministic = True
    torch.backends.cudnn.benchmark = False

    # DataLoader worker种子固定
    def seed_worker(worker_id):
        worker_seed = torch.initial_seed() % 2**32
        np.random.seed(worker_seed)
        random.seed(worker_seed)

    g = torch.Generator()
    g.manual_seed(seed)
    return seed_worker, g
```

**注意事项**：
- `cudnn.deterministic=True` 会禁用某些非确定性优化，可能导致训练速度下降5-10%
- 分布式训练中每个进程需要不同的种子（`seed + rank`），确保数据增强不重复

### 4.2 Checkpoint管理

完整的训练checkpoint应包含：

```python
checkpoint = {
    'epoch': epoch,
    'model_state_dict': model.state_dict(),
    'optimizer_state_dict': optimizer.state_dict(),
    'scheduler_state_dict': scheduler.state_dict(),  # 学习率调度器
    'loss': best_loss,
    'config': config,                                  # 超参数配置
    'random_state': random.getstate(),                 # (可选的随机状态)
}

torch.save(checkpoint, f'checkpoint_epoch_{epoch}.pt')
```

- **只保存 `model.state_dict()`**（不含优化器）≈ 节省50%存储，但无法从断点恢复训练
- **保存完整状态**（含优化器+scheduler）≈ 存储翻倍，但可无缝恢复训练

### 4.3 实验配置管理

避免硬编码，使用层次化配置文件：

```yaml
# config.yaml
model:
  name: "resnet50"
  pretrained: true
training:
  batch_size: 256
  epochs: 100
  optimizer:
    name: "adamw"
    lr: 0.001
    weight_decay: 0.05
  scheduler:
    name: "cosine"
    warmup_epochs: 5
data:
  dataset: "imagenet"
  img_size: 224
```

- **Hydra**（Meta）：支持命令行覆盖、配置组合、多任务运行
- **OmegaConf**：轻量级YAML配置库
- **dataclasses / pydantic**：类型安全的配置管理

### 4.4 实验追踪平台对比

| 平台 | 特性 | 部署方式 | 最佳场景 |
|:---|:---|:---|:---|
| **Weights & Biases** | 最强可视化，无缝团队协作，Sweep超参搜索 | 云端SaaS | 研究团队、学术实验 |
| **MLflow** | 开源自托管，Tracking + Registry + Projects | 自托管 | 企业内部、数据合规 |
| **TensorBoard** | 轻量本地使用，PyTorch/TF原生 | 本地 | 个人快速调试 |
| **Neptune.ai** | 元数据管理，大规模实验对比 | 云端SaaS | 大规模对比实验 |
| **DVC** | Git友好的数据版本控制 | 与Git集成 | 数据集+模型版本管理 |

### 4.5 超参数搜索方法

| 方法 | 描述 | 效率 |
|:---|:---|:---|
| **Grid Search** | 枚举所有组合 | 指数级，低效 |
| **Random Search** | 随机采样超参数空间 | 优于Grid（Bergstra & Bengio, 2012） |
| **Bayesian Optimization** | 根据历史结果建模，选择最有前景的下一个点 | **高效**（Optuna, Hyperopt） |
| **Population Based Training (PBT)** | 进化搜索，训练过程中动态调整 | DeepMind用于大模型 |
| **$\mu$Transfer** | 小模型搜索→理论保证→大模型迁移 | 大模型时代的前沿方案 |

## 5. 关键方法/模型

### 代表性工具与论文

- **Weights & Biases** (2017-)：实验追踪的事实标准，已被OpenAI, Meta, NVIDIA等广泛使用
- **MLflow** (Databricks, 2018-)：开源ML生命周期管理，与Spark/Delta Lake生态集成
- **DVC** (2017-)：基于Git的数据和模型版本控制
- **Optuna** (Preferred Networks, 2019-)：高效的贝叶斯超参搜索框架
- **Bergstra & Bengio (2012)**：*Random Search for Hyper-Parameter Optimization*. JMLR. 证明随机搜索优于网格搜索
- **Li et al. (2018)**：*Hyperband: A Novel Bandit-Based Approach to Hyperparameter Optimization*. JMLR. 早停策略加速超参搜索

## 6. 优势与局限

### 实验管理平台的选择权衡

| 维度 | W&B | MLflow | TensorBoard |
|:---|:---|:---|:---|
| 上手速度 | 快 | 中 | 快 |
| 可视化丰富度 | 极高 | 中 | 基础 |
| 团队协作 | 强 | 中 | 弱（本地） |
| 数据隐私 | 云端存储 | 自托管 | 本地 |
| 成本 | 付费（团队版） | 免费 | 免费 |
| 模型注册表 | 有 | 强 | 无 |

### 可复现性的现实挑战

- **GPU非确定性**：`cudnn.benchmark=True` 会自动选择最快算法，不同GPU/驱动版本可能不同
- **分布式训练**：不同数量的GPU导致不同的浮点累加顺序，产生微小数值差异
- **数据版本**：公开数据集也可能有版本更迭（如ImageNet的2012 vs 2017版本）
- **PyTorch版本**：不同小版本的算子实现可能略有差异

**最佳实践**：在论文中报告多次运行（如5次）的均值±标准差，而非单次最佳结果。

## 7. 应用场景

- **学术研究**：W&B 记录实验 → 消融实验探究 → 论文图表生成
- **工业项目**：MLflow追踪 → Model Registry管理 → CI/CD自动部署
- **大模型训练**：分布式训练checkpoint管理、训练曲线实时监控、异常检测
- **模型选型竞赛**：多模型并行对比（W&B Sweep / Optuna Study）
- **数据集版本管理**：DVC追踪数据预处理流水线的每个版本

## 9. 前沿发展

1. **实验管理的AI化**
   - LLM辅助分析实验日志，自动生成实验报告和结论摘要
   - 智能超参推荐：基于历史实验数据，为下一个实验推荐最优超参起点

2. **可复现性的硬性保证**
   - Nvidia GPU确定性模式（deterministic mode）在A100/H100上逐步成熟
   - Docker + Nix/GUix实现比特级可复现环境

3. **MLOps全流程集成**
   - 实验管理 → 模型注册表 → CI/CD（自动测试+部署）→ 线上监控 → 数据漂移检测 → 自动重训练 → 闭环
   - 实验管理不再孤立，而是ML系统工程的核心组件

4. **大模型评测体系**
   - 随着LLM能力评估的标准化（如Open LLM Leaderboard, Chatbot Arena），实验管理需要支持开放式的、多指标的、人类评估参与的评测流水线
