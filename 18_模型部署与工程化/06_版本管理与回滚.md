# 版本管理与回滚

## 1. 概述

大模型版本管理面临超大模型权重（GB/TB 级）、多维度版本依赖（模型、数据、配置、镜像）和快速迭代需求等挑战，与传统软件版本管理有本质差异。

版本管理与回滚解决的核心问题：
- **可追溯性**：每次变更可追溯到具体的模型、数据和配置版本
- **可验证性**：新版本上线前可通过灰度发布逐步验证
- **可回滚性**：异常时可快速恢复到上一稳定版本
- **多维度依赖**：管理模型权重、训练数据、推理配置、服务镜像的版本关系

通过模型注册表统一管理版本元数据，通过灰度发布逐步验证新版本，通过一键回滚在异常时快速恢复服务。

## 3. 核心概念

### 3.1 版本管理维度

| 维度 | 管理内容 | 工具 |
|:---|:---|:---|
| **模型权重** | 基础模型、LoRA 适配器、量化模型 | DVC、模型注册表 |
| **训练数据** | 数据集版本、采样策略 | DVC、LakeFS |
| **配置版本** | 超参数、推理配置、Prompt 模板 | Git |
| **服务镜像** | 推理引擎镜像、依赖版本 | Docker Registry |
| **部署状态** | 副本数、资源分配、路由规则 | Kubernetes、IaC |

### 3.2 模型注册表

提供版本控制、阶段管理（Development → Staging → Production）和元数据存储的中央服务。

### 3.3 语义化版本控制

`<模型>-<功能>-<主版本>.<次版本>.<修订号>`
- 主版本变更：不兼容更新（架构改变、能力退化）
- 次版本变更：兼容性功能新增
- 修订号变更：Bug 修复、小幅优化

### 3.4 灰度发布

- **蓝绿部署**：新旧版本并行运行，通过路由切换流量，回滚即时
- **金丝雀发布**：小流量（如 5%）验证新版本，逐步放量到 100%
- **A/B 测试**：并行运行多版本，基于统计显著性检验对比效果

### 4.2 Kubernetes 金丝雀部署

```yaml
annotations:
  nginx.ingress.kubernetes.io/canary: "true"
  nginx.ingress.kubernetes.io/canary-weight: "10"  # 10% 流量到新版本
```

### 4.3 回滚机制

回滚流程：触发（手动/自动）→ 流量切回旧版本 → 缓存清理（KV-Cache、模型缓存）→ 告警暂停 → 验证恢复。

```bash
# Kubernetes 回滚到上一版本
kubectl rollout undo deployment llm-service
# 回滚到特定版本
kubectl rollout undo deployment llm-service --to-revision=3
```

### 4.4 A/B 测试统计检验

对两组样本均值执行 $t$ 检验：

$$t = \frac{\bar{X}_A - \bar{X}_B}{\sqrt{s_A^2/n_A + s_B^2/n_B}}$$

$p < 0.05$ 时认为存在显著差异。

### 4.5 滚动更新

Kubernetes 逐个替换 Pod，通过 `maxSurge`（超出期望副本数的最大数量）和 `maxUnavailable`（不可用副本的最大数量）控制更新速率，确保零停机部署。

## 5. 关键方法与模型

### 5.1 大模型特有挑战

- **分片管理**：TB 级模型需分片存储和并行加载，需维护分片元数据一致性
- **LoRA 组合**：多个 LoRA 适配器的组合版本需适配器版本矩阵管理
- **在线学习**：实时数据更新下的增量版本控制和版本快照机制

### 5.2 DVC 数据版本控制

```bash
# 添加数据文件到 DVC 追踪
dvc add data/train.jsonl
# 推送到远程存储
dvc push
# 提交 Git 元数据
git add data/train.jsonl.dvc
git commit -m "Update training data v2"
```

### 5.3 不可变模型部署

模型镜像一旦构建不可修改，版本即镜像标签（如 `model:v2.1.3`）。任何变更需构建新镜像，确保部署可追溯和可回滚。

## 6. 优势与局限

### 6.1 优势

- **可追溯**：完整的版本变更历史，满足审计要求
- **低风险**：灰度发布逐步验证，异常快速回滚
- **可复现**：精确还原任意版本的生产环境
- **并行实验**：多版本并行运行对比效果

### 6.2 局限

- **存储成本**：多版本模型权重存储成本高
- **大文件管理**：Git 不适合管理 GB/TB 级文件
- **版本爆炸**：LoRA 组合、多环境部署导致版本矩阵复杂
- **回滚延迟**：大模型回滚需重新加载权重，耗时数十秒
