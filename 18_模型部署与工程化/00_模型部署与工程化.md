# 模型部署与工程化

## 1. 概述

模型部署与工程化是将 AI 模型从离线实验环境迁移到在线生产环境的系统化工程实践，涵盖模型服务化、容器化部署、推理优化、API 治理、监控运维、版本管理、成本控制与合规治理等完整链路。

该领域解决的核心问题：
- **从实验到生产**：训练完成的模型需经过导出、优化、部署、监控才能服务真实用户
- **延迟与吞吐矛盾**：批处理提高吞吐但增加单请求延迟，需动态调度
- **成本与质量矛盾**：大模型精度高但推理成本高，需在性能和预算间平衡
- **效率与安全矛盾**：快速迭代需求与合规审计要求之间存在张力

随着大语言模型（LLM）规模从亿级增长到千亿级，推理引擎、容器编排、API 网关等工程化设施成为保障服务延迟、吞吐与可靠性的关键。MLOps 理念将 DevOps 实践扩展到机器学习系统，形成覆盖模型注册、灰度发布、自动扩缩容、全链路监控的完整体系。

## 3. 核心概念

### 3.1 MLOps

将 DevOps 实践扩展到机器学习系统，覆盖数据管理、模型训练、部署、监控和迭代的全生命周期。核心原则：自动化、可追溯、可复现、可监控。

### 3.2 推理引擎

专门用于执行模型推理的软件系统，负责模型加载、计算调度、内存管理和性能优化。与传统 Web 服务不同，LLM 推理引擎需要处理自回归生成、动态序列长度和 KV-Cache 管理等特有挑战。

### 3.3 分层架构

模型部署的工程化架构包括四层：

- **网关层**：认证授权、限流熔断、请求路由、API 版本管理
- **服务层**：推理引擎调度、动态批处理、流式响应（SSE/gRPC stream）
- **推理层**：推理引擎（vLLM、TensorRT-LLM、TGI）、KV-Cache 管理、量化计算
- **基础设施层**：GPU 集群、Docker 容器、Kubernetes 编排、存储与网络

### 3.4 关键性能指标

| 指标 | 全称 | 含义 |
|:---|:---|:---|
| TTFT | Time To First Token | 首 Token 延迟，用户感知的关键指标 |
| TPOT | Time Per Output Token | 每生成一个 Token 的平均时间 |
| QPS | Queries Per Second | 每秒处理请求数 |
| Throughput | tokens/s | 单位时间 Token 生成速率 |
| GPU Utilization | — | GPU 计算单元利用率 |

### 3.5 推理的计算特性

LLM 自回归推理是**访存密集型**（Memory-bound）而非计算密集型：

$$T_{forward} = \frac{2 \times N_{params}}{\text{Throughput}_{compute}} + \frac{N_{bytes}}{\text{Bandwidth}_{memory}}$$

计算访存比（Arithmetic Intensity）：

$$AI = \frac{\text{FLOPs}}{\text{Bytes}} = \frac{2N_{params}}{N_{bytes}} \approx 1$$

LLM 自回归生成的 $AI \approx 1$，远小于 A100 的平衡点 156，这意味着推理瓶颈在显存带宽而非计算能力。

## 4. 技术原理

### 4.1 PagedAttention

vLLM 的核心创新，将 KV-Cache 划分为固定大小的页（block），按需分配，类似操作系统的虚拟内存分页机制。减少显存碎片化带来的浪费，实测可将吞吐提升 2-4 倍。

### 4.2 连续批处理

与传统静态批处理不同，每个请求完成生成后立即释放资源，新请求动态加入批次，消除队头阻塞。

- 传统静态批处理：$T_{static} = \max(L_i) \times T_{forward}$
- 连续批处理：$T_{cont} = \frac{\sum_i L_i}{N_{max}} \times T_{forward}$

请求长度差异大时 $T_{cont} \ll T_{static}$。

### 4.3 推测解码

小模型（Draft Model）生成候选 Token，大模型（Target Model）并行验证。若候选 Token 被接受，大模型无需逐步自回归生成，可将推理速度提升 2-3 倍。接受概率：

$$p = \min\left(1, \frac{p_{\text{target}}(x)}{p_{\text{draft}}(x)}\right)$$

### 4.4 模型量化

| 量化方法 | 位宽 | 显存减少 | 速度提升 | 精度损失 | 适用场景 |
|:---|:---|:---|:---|:---|:---|
| GPTQ | INT4 | 75% | 2-4x | 1-2% | 后训练量化，广泛使用 |
| AWQ | INT4 | 75% | 2-4x | <1% | 激活感知量化 |
| INT8 | INT8 | 50% | 1.5-2x | <0.5% | 通用量化 |

量化信噪比：$\text{SNR} \approx (2^b)^2$，每增加 1 bit，SNR 提高约 6dB。

### 4.5 多 GPU 并行

- **张量并行（TP）**：将模型单层内权重切分到多 GPU，需高带宽互联（NVLink）
- **流水线并行（PP）**：将模型不同层分配到不同 GPU，通信量小但存在气泡
- **数据并行（DP）**：多副本独立处理不同请求，通过负载均衡分发

## 6. 优势与局限

### 6.1 工程化的价值

- **可靠性**：通过标准化容器镜像、自动化 CI/CD 和多级监控保障服务稳定
- **可扩展性**：水平扩展和弹性伸缩应对流量波动
- **可追溯性**：版本管理和审计日志满足合规要求
- **成本可控**：多级配额和优化技术系统性降低推理成本

### 6.2 工程化的挑战

- **冷启动延迟**：大模型加载到显存需数十秒，影响弹性伸缩
- **GPU 资源稀缺**：高端 GPU 供应紧张，成本高昂
- **模型迭代复杂**：超大模型权重（GB/TB 级）的版本管理困难
- **LLM 特有运维**：自回归生成的不确定性使传统 Web 服务监控不适用
- **合规复杂性**：全球数据保护法规差异大，跨境部署面临法律挑战
