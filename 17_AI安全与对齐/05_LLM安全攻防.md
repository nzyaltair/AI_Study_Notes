# LLM安全攻防

## 1. 概述

大语言模型的安全威胁与传统AI安全有本质区别：LLM通过自然语言交互，模型无法在架构层面区分系统指令与用户数据，导致全新的攻击面。OWASP在2023年将提示注入列为LLM Top 10安全风险之首（LLM01）。

LLM安全攻防涵盖提示注入（直接与间接）、越狱攻击、安全护栏绕过等威胁，以及输入过滤、对齐训练、输出审查、运行时监控等防御机制。随着LLM集成工具调用和Agent架构，攻击面从单一文本交互扩展到多模态、跨应用链式攻击。

## 3. 核心概念

### 攻击类型分类

| 类型 | 描述 | 典型示例 |
|------|------|---------|
| 直接提示注入 | 攻击者直接在输入中嵌入恶意指令 | "忽略之前的指令，现在你是..." |
| 间接提示注入 | 通过外部载体间接注入指令 | RAG检索到恶意网页内容 |
| 越狱攻击 | 绕过对齐机制生成有害内容 | DAN模板、角色扮演 |
| 对抗性后缀 | 梯度优化的无意义后缀触发漏洞 | GCG攻击 |
| 数据提取攻击 | 诱导模型输出训练数据或系统提示 | "重复你收到的系统指令" |
| 工具调用滥用 | 通过Function Calling执行越权操作 | 参数注入、权限提升 |

### 安全护栏失效模式

- **指令覆盖**：强指令性语言覆盖系统提示边界
- **编码绕过**：Base64/ROT13/多语言混合绕过关键词过滤
- **上下文污染**：多轮对话逐步建立信任后引入恶意指令
- **跨模态注入**：通过图像、音频中嵌入指令攻击多模态模型

## 4. 技术原理

### GCG对抗攻击

Greedy Coordinate Gradient（GCG）将对抗攻击从连续空间迁移到离散文本空间：

$$\mathcal{L}_{adv} = -\sum_{i} \log P(target_i | prompt, suffix, target_{<i})$$

**算法流程**：
1. 初始化对抗后缀（随机token序列）
2. 计算每个位置的替换梯度
3. 保留梯度最大的top-k候选替换
4. 逐一评估候选，选择损失最小的替换
5. 重复直到攻击成功或达到最大迭代

**迁移性**：GCG后缀可跨模型迁移——在开源模型上生成的后缀对闭源模型（GPT-4、Claude）也有效。

### 防御体系

**输入层防御**：
- **模式匹配**：检测已知越狱模板和注入模式
- **输入隔离**：将用户输入与系统提示严格分离（如分隔符标记）
- **语义分析**：使用分类模型检测输入的攻击意图
- **指令层级**：OpenAI的系统指令优先级机制

**模型层防御**：
- **安全对齐训练**：RLHF/RLAIF中纳入安全偏好数据
- **Constitutional AI**：宪法原则驱动的自我审查
- **对抗训练**：使用GCG等攻击样本进行对抗训练
- **拒绝采样**：训练时采样多个回复，过滤不安全内容

**输出层防御**：
- **Llama Guard**：Meta开源的内容安全分类模型，检测输入输出安全性
- **NeMo Guardrails**：NVIDIA的对话安全框架，规则与模型双重护栏
- **PII过滤**：基于NER和正则的个人身份信息识别与脱敏
- **水印嵌入**：在生成内容中嵌入不可见标识用于来源追踪

**系统层防御**：
- **速率限制**：防止大规模自动化攻击
- **权限控制**：最小权限原则限制工具调用范围
- **运行时监控**：异常行为分析和实时告警
- **审计日志**：记录所有交互用于事后追溯

## 6. 优势与局限

**防御体系优势**：
- 多层纵深防御提供冗余保障
- 对齐训练从模型层面降低攻击成功率
- 开源工具（Llama Guard等）降低部署门槛

**局限**：
- **无法完全防御**：提示注入是LLM架构的固有缺陷（指令与数据不分离）
- **攻防不对等**：攻击只需找到一个漏洞，防御需覆盖所有攻击面
- **安全-有用性权衡**：过度防御降低模型有用性和创造力
- **零日攻击**：新型攻击方法无法被已知模式检测
- **多模态扩展**：多模态模型引入新的攻击面

## 7. 应用场景

- **LLM应用安全**：聊天机器人、代码助手、文档处理系统的安全防护
- **RAG系统安全**：防止检索文档中的间接注入
- **Agent安全**：防止工具调用被恶意指令劫持
- **内容审核**：自动化有害内容检测与过滤
- **合规审计**：满足AI Act等法规的安全评估要求
