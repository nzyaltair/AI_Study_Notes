# 大模型时代的 NLP

## 1. 概述

大语言模型（LLM）的出现深刻改变了 NLP 的研究范式和实践方式。从"为每个任务训练专门模型"到"一个模型解决所有任务"，从"监督学习为主"到"自监督预训练+提示/微调"，NLP 经历了自 Transformer 诞生以来最重大的范式转移。

本章梳理大模型时代 NLP 的核心变化：任务统一化（In-Context Learning）、知识外部化（RAG）、能力涌现化（推理/规划/工具调用），以及传统 NLP 任务在 LLM 框架下的新形态。

## 3. 核心概念

### 3.1 范式转移

| 维度 | 传统 NLP | LLM 时代 NLP |
|:---|:---|:---|
| 模型 | 每个任务专门训练 | 一个模型多任务 |
| 训练 | 监督学习为主 | 自监督预训练+对齐 |
| 适配 | 微调模型参数 | 提示工程/少样本 |
| 评估 | 任务特定指标 | 通用能力评估 |
| 数据 | 标注数据关键 | 预训练数据+指令数据 |
| 部署 | 单一任务模型 | 统一推理服务 |

### 3.2 In-Context Learning（上下文学习）

LLM 通过在输入中提供少量示例来适应新任务，无需参数更新：

```
零样本（Zero-shot）：
"将以下句子翻译为法语：Hello world →"

少样本（Few-shot）：
"English: Hello → French: Bonjour
English: Goodbye → French: Au revoir
English: Thank you → French:"
```

ICL 的出现使得 NLP 任务适配从"训练模型"变为"设计提示"。

### 3.3 任务统一化

LLM 将多种 NLP 任务统一为文本生成：

| 传统任务 | LLM 统一形式 |
|:---|:---|
| 文本分类 | 生成类别名称 |
| 命名实体识别 | 生成实体列表（JSON） |
| 机器翻译 | 生成翻译文本 |
| 问答 | 生成答案 |
| 摘要 | 生成摘要文本 |
| 情感分析 | 生成情感标签 |
| 句法分析 | 生成句法树文本 |

### 3.4 涌现能力

大模型在规模超过一定阈值后展现出的新能力：

| 能力 | 说明 | 涌现规模 |
|:---|:---|:---|
| In-Context Learning | 从示例中学习新任务 | ~10B 参数 |
| Chain-of-Thought | 逐步推理 | ~60B 参数 |
| 指令遵循 | 按指令执行任务 | 需指令微调 |
| 工具使用 | 调用外部 API/函数 | 需专门训练 |
| 代码生成 | 编写/执行代码 | ~10B+ 参数 |

## 4. 技术原理

### 4.1 RAG（检索增强生成）

RAG 结合检索系统和生成模型，为 LLM 提供外部知识：

```
用户查询 → 检索器（BM25/DPR）→ 相关文档 → LLM（带检索上下文）→ 生成答案
```

**优势**：
- 减少幻觉（答案基于检索文档）
- 提供最新信息（无需重新训练）
- 可追溯来源

**架构演进**：
- **朴素 RAG**：检索→拼接→生成
- **高级 RAG**：查询重写、重排序、上下文压缩
- **模块化 RAG**：检索、路由、融合、过滤等模块灵活组合

参见 [[19_LLM应用工程/03_RAG系统]]。

### 4.2 Chain-of-Thought（思维链）

通过引导 LLM 展示推理过程来提升复杂推理能力：

```
标准提示：
"Q: 餐厅有23个苹果，用了20个做午餐，又买了6个，有多少苹果？"
"A: 9"

思维链提示：
"Q: 餐厅有23个苹果，用了20个做午餐，又买了6个，有多少苹果？"
"A: 原来有23个苹果。用了20个做午餐，所以剩下 23-20=3 个。
   又买了6个，所以现在有 3+6=9 个。答案是9。"
```

CoT 显著提升数学推理、逻辑推理和多步推理任务的性能。

### 4.3 Tool Use / Function Calling

LLM 调用外部工具扩展能力：

```
用户："今天北京天气怎么样？"
LLM → 调用 weather_api(city="北京") → 获取结果 → 生成回答
```

OpenAI Function Calling、Anthropic Tool Use 等机制使 LLM 能够：
- 调用搜索引擎获取实时信息
- 执行代码进行计算
- 查询数据库
- 操作外部系统

### 4.4 推理时扩展（Inference-Time Scaling）

OpenAI o1 / DeepSeek-R1 展示了通过在推理时增加计算来提升性能的新范式：

- **训练阶段**：用 RL 训练模型生成长思维链
- **推理阶段**：模型自主进行多步推理、自我验证、回溯

这与传统的"扩大模型规模"不同——通过增加推理时的计算量来换取更好的推理质量。

## 5. 传统 NLP 任务在 LLM 时代的新形态

| 传统任务 | 传统方法 | LLM 方法 | 变化 |
|:---|:---|:---|:---|
| 分词 | BPE/SentencePiece | 仍需分词器 | 基础组件不变 |
| 词性标注 | CRF/BiLSTM-CRF | 提示工程/不再独立 | 隐式化 |
| NER | BERT-CRF | Few-shot + 提示 | 零样本可用 |
| 句法分析 | Transition-based | 直接生成/隐式 | 弱化 |
| 文本分类 | SVM/BERT 微调 | Zero/Few-shot | 无需训练 |
| 机器翻译 | NMT 专用模型 | LLM 直接翻译 | 通用化 |
| 信息抽取 | 专门 IE 模型 | LLM 生成式抽取 | 统一化 |
| 摘要 | BART/T5 微调 | LLM 零样本摘要 | 通用化 |
| 问答 | MRC 专用模型 | RAG + LLM | 增强化 |
| 对话 | 模块化系统 | LLM 端到端 | 革命性变化 |

### 什么变了？

1. **任务边界模糊**：一个 LLM 可同时处理分类、翻译、摘要、问答等
2. **数据需求降低**：零样本/少样本减少标注需求
3. **开发模式变化**：从"训练模型"到"设计提示+调用 API"
4. **评估更复杂**：需要评估通用能力而非单一任务

### 什么没变？

1. **基础组件**：分词、嵌入、注意力仍是 LLM 的基础
2. **理论根基**：语言模型、概率论、信息论仍是 NLP 的数学基础
3. **评估需求**：仍需要准确、忠实、流畅的评估标准
4. **领域知识**：专业领域仍需要领域适配

## 6. 优势与局限

### 优势

1. **通用性**：一个模型解决多种任务
2. **零样本能力**：无需标注数据即可使用
3. **能力涌现**：规模带来质变
4. **开发效率**：提示工程替代模型训练

### 局限

1. **幻觉**：生成看似合理但错误的内容
2. **成本高**：大模型推理成本远高于专用小模型
3. **延迟高**：自回归生成延迟高于分类模型
4. **可控性差**：输出难以精确控制
5. **隐私安全**：数据需发送到模型服务
6. **领域深度**：通用模型在专业领域可能不如专用模型

## 7. 应用场景

- **通用助手**：ChatGPT、Claude、Gemini——覆盖几乎所有 NLP 任务
- **代码助手**：Copilot、Cursor——编程+NLP 的融合
- **企业知识库**：RAG 系统——检索+生成
- **Agent 系统**：LLM 驱动的自主任务执行
- **多模态交互**：GPT-4o 等统一文本、图像、语音
- **推理增强**：o1/R1 等推理模型解决复杂问题

## 9. 前沿发展

- **推理模型**：o1/R1 开创 RL 驱动的长链推理范式，NLP 从"模式匹配"走向"深度推理"
- **Agentic NLP**：LLM Agent 自主规划、检索、执行——NLP 从"被动响应"走向"主动行动"
- **多模态统一**：文本、图像、音频、视频的统一处理——NLP 成为多模态 AI 的语言模块
- **个性化与记忆**：长期记忆和个性化适配——LLM 成为"个人 NLP 助手"
- **端侧 NLP**：小模型在终端设备运行——隐私、延迟、成本的最优平衡
- **NLP 科学发现**：LLM 辅助科研——文献挖掘、假设生成、科学写作
- **自我改进**：LLM 通过自我生成数据和反馈持续改进 NLP 能力
