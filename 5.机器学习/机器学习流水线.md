# 机器学习流水线

## 背景

机器学习流水线（ML Pipeline）是将ML项目的各个阶段——从数据收集到模型部署和监控——自动化和标准化的流程。它解决了传统ML开发中手动操作多、实验难复现、部署不规范的问题，是[[软件工程基础|工程化]]思维在ML领域的应用。

## 核心思想

将数据处理、[[特征工程|特征工程]]、模型训练、评估和部署等步骤串联为模块化的工作流，确保整个流程可复现、可测试、可维护。

## 技术原理

### 流水线典型架构

```
数据收集 → 数据预处理 → 特征工程 → 模型训练与评估 → 模型部署 → 模型监控
```

### 各阶段职责

| 阶段 | 职责 | 关键工具 |
|------|------|---------|
| **数据收集** | 数据获取、质量检查、版本控制 | DVC、Git LFS |
| **数据预处理** | 清洗缺失值/异常值、类型转换、数据分割 | Pandas、Scikit-learn |
| **特征工程** | 特征提取、选择、构造、转换 | Featuretools、Scikit-learn |
| **模型训练** | 超参调优、交叉验证、模型选择 | Optuna、MLflow |
| **模型评估** | 指标计算、模型解释 | SHAP、LIME |
| **模型部署** | API服务、容器化、灰度发布 | Docker、FastAPI、KServe |
| **模型监控** | 数据漂移检测、性能监控 | Prometheus、Evidently AI |

### 数据漂移与概念漂移

- **数据漂移（Data Drift）**：输入数据分布变化，导致模型性能下降
  - 检测：KS检验、AD检验
  - 解决：重新训练模型
- **概念漂移（Concept Drift）**：特征与目标关系变化
  - 检测：监控性能指标变化
  - 解决：更新训练数据、重新定义问题

### 流水线自动化框架

| 框架 | 特点 | 适用场景 |
|------|------|---------|
| **Airflow** | DAG定义工作流，生态丰富 | ETL、通用流水线 |
| **Kubeflow** | 基于K8s，端到端ML | 大规模分布式训练 |
| **MLflow** | 实验跟踪+模型管理+部署 | 实验管理、模型版本控制 |
| **Metaflow** | Netflix开源，原型到生产 | 数据科学原型开发 |
| **Prefect** | 现代工作流管理，动态工作流 | 数据工程、ML流水线 |

### MLOps核心原则

- **自动化**：数据处理→训练→评估→部署全流程自动化
- **可重复性**：确保实验和模型训练可复现
- **版本控制**：代码、数据、模型三重版本管理
- **持续监控**：性能指标、数据分布、业务指标全面监控
- **CI/CD**：模型持续集成与持续部署

## 发展演进

手动执行 → 脚本化 → 框架化（Airflow/Kubeflow）→ 平台化 → AutoML自动化

从[[机器学习|传统ML]]的手动流程，发展到MLOps体系化的工程实践，再到LLMOps针对[[大语言模型|大模型]]的特殊运维需求。

## 应用领域

- **企业ML系统**：金融风控、推荐系统、广告投放
- **[[深度学习|深度学习]]项目**：模型训练→部署→监控全生命周期
- **[[大语言模型|LLM]]应用**：LLMOps——模型压缩、量化、安全对齐、推理优化

## 与其他技术关系

- 流水线依赖[[软件工程基础|软件工程]]的模块化设计和CI/CD实践
- [[特征工程]]是流水线的核心环节
- [[模型选择与调参|模型选择]]与评估嵌入流水线的训练阶段
- LLMOps是MLOps在[[大语言模型|大模型]]时代的延伸
- 部署与监控详见[[容器化与Docker]]和系统工程化模块
