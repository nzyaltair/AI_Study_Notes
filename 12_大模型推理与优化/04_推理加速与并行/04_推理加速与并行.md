# 推理加速与并行

## 1. 概述

推理加速与并行技术旨在通过计算优化、内存优化、并行策略和调度创新，在保持模型精度的前提下最大化推理吞吐量、降低延迟。大语言模型推理面临三大瓶颈：计算瓶颈（自注意力 $O(n^2)$ 复杂度）、内存瓶颈（模型参数和 [[KV-Cache]] 占用大量显存）、带宽瓶颈（显存带宽限制参数访问速度）。

推理加速技术从早期的算子融合和模型并行，发展到 PagedAttention、连续批处理、Speculative Decoding 等系统级创新，使推理吞吐量提升了数十倍。

## 3. 核心概念

### 3.1 Prefill 与 Decode 两阶段

LLM 推理分为两个截然不同的阶段：

| 阶段 | 特征 | 瓶颈类型 | 优化方向 |
|------|------|---------|---------|
| Prefill | 并行处理输入 prompt | 计算密集（compute-bound）| 算子优化、并行 |
| Decode | 逐 token 生成 | 带宽密集（memory-bound）| 批处理、投机解码 |

### 3.2 核心优化方向

- **计算优化**：算子融合、FlashAttention、Flash-Decoding
- **内存优化**：[[KV-Cache]] 管理、PagedAttention
- **并行策略**：张量并行、流水线并行、序列并行
- **调度优化**：连续批处理、Prefill-Decode 分离
- **解码加速**：Speculative Decoding、Medusa、EAGLE

## 4. 技术原理

### 4.1 算子融合

将多个独立算子合并为复合算子，减少内核启动和中间结果的内存访问开销。典型应用：

- **FlashAttention**：融合 QK^T、softmax、AV 计算
- **TensorRT-LLM 内核融合**：融合 LayerNorm、GEMM、激活函数
- ** fused MLP**：融合 Gate、Up projection 和 SiLU 激活

### 4.2 FlashAttention

重新设计注意力计算的内存访问模式：

**核心思想**：将注意力计算分块在 GPU SRAM（高速缓存）中执行，避免在 HBM（显存）中实例化完整的 $N \times N$ 注意力矩阵。

**工作流程**：
1. 将 Q, K, V 分为大小为 $B$ 的块
2. 逐块加载到 SRAM，计算部分注意力
3. 使用在线 softmax 技术累加结果
4. 只将最终结果写回 HBM

**效果**：
- 速度提升 2-4x，内存使用降低 50-70%
- 支持更长序列
- 计算复杂度仍为 $O(n^2)$ 但 IO 大幅优化
- FlashAttention-2 进一步优化并行度和分块策略
- FlashAttention-3 针对 Hopper 架构 FP8 优化

### 4.3 Flash-Decoding

标准注意力在 Decode 阶段每步只处理一个 token（Q 维度 1），无法利用 GPU 并行度。Flash-Decoding 在 K/V 序列维度上并行化：

- 将 KV-Cache 分成多个块
- 每个块独立计算部分注意力
- 通过 reduction 合并结果

适合长序列（KV-Cache 大）的 Decode 阶段加速。

### 4.4 PagedAttention

借鉴 OS 虚拟内存管理，将 [[KV-Cache]] 划分为固定大小的块：

- 每个序列的 KV-Cache 由逻辑块组成，映射到物理块
- **消除内存碎片**：利用率提升 60-70%
- **支持连续批处理**：新请求动态加入，已完成请求的页立即回收
- **前缀共享**：相同系统 prompt 的请求共享 KV-Cache 前缀
- 减少 OOM 风险

### 4.5 连续批处理（Continuous Batching）

传统批处理需等待同 batch 所有请求完成（同步等待），短请求被长请求拖累。

**连续批处理**（又称 In-flight Batching / Dynamic Batching）：
- 在每个 Decode 步骤的边界动态调度
- 新请求可随时加入（立即开始 Prefill）
- 已完成请求立即移除，释放资源
- GPU 利用率从 30-40% 提升到 80-90%
- 吞吐量提升 3-4x，延迟降低 30-50%

**实现挑战**：
- 需要管理变长序列的 KV-Cache（PagedAttention 解决）
- Prefill 和 Decode 混合调度策略
- 请求优先级和公平性管理

### 4.6 张量并行（Tensor Parallelism）

将模型权重矩阵在多个 GPU 上分割：

- **列并行**：$Y = XW$，$W$ 按列切分，各 GPU 计算部分输出，通过 All-Reduce 聚合
- **行并行**：$W$ 按行切分，各 GPU 计算部分乘积，通过 All-Reduce 求和

适合超大模型（>30B），计算速度与 GPU 数量近似线性增长，但通信开销随 GPU 数量增加。

### 4.7 流水线并行（Pipeline Parallelism）

将模型层在多个 GPU 上分割，形成流水线：

- 适合层数较多的模型
- 存在流水线气泡（bubble），GPU 利用率有损失
- GPipe 和 1F1B 策略减少气泡

### 4.8 Speculative Decoding

使用小模型（Draft Model）提前生成多个候选 token，大模型（Target Model）一次性验证：

1. 小模型用采样策略生成 $k$ 个候选 token
2. 大模型并行计算这 $k$ 个 token 的概率
3. 接受匹配的 token，在首个拒绝处重新生成

**接受概率**：$\min\left(1, \frac{\pi_{\text{target}}(y_i)}{\pi_{\text{draft}}(y_i)}\right)$

**关键特性**：
- 生成质量与大模型**完全一致**（通过拒绝采样保证）
- 延迟降低 30-50%，吞吐量提升 2-3x
- 加速比取决于 Draft 模型的接受率

### 4.9 多 token 预测变体

| 方法 | 原理 | 特点 |
|------|------|------|
| **Speculative Decoding** | 独立小模型草拟 | 需要额外草稿模型 |
| **Medusa** | 大模型添加多个预测头，每头预测未来 token | 无需独立草稿模型 |
| **EAGLE** | 基于大模型隐藏状态的自回归草拟 | 接受率高，加速比大 |
| **Lookahead Decoding** | Jacobi 迭代并行解码 | 无需草稿模型，无需训练 |
| **Self-Speculative** | 跳过部分层作为"草稿模型" | 无需额外模型 |

### 4.10 Offloading

将部分模型参数或 KV-Cache 从 GPU 转移到 CPU 内存或 NVMe 存储：

- 支持比 GPU 显存更大的模型
- 代价是增加延迟、降低吞吐量
- 典型实现：llama.cpp GPU-CPU offloading、DeepSpeed ZeRO-Inference
- 适合资源受限但延迟要求不高的场景

### 4.11 Prefill-Decode 分离推理

将 Prefill 和 Decode 部署到不同 GPU 集群：

- **Prefill 集群**：计算密集，配置高算力 GPU，优化吞吐
- **Decode 集群**：带宽密集，配置高带宽 GPU，优化并发
- **KV-Cache 传输**：Prefill 完成后将 KV-Cache 传输到 Decode 集群
- 避免两阶段相互干扰，各自独立优化
- 代表系统：Splitwise、DistServe

## 6. 优势与局限

**优势**：
- 吞吐量大幅提升：vLLM 比原生 HF 高 10-20x
- 延迟显著降低：TensorRT-LLM + FP8 实现最低延迟
- 生成质量保持：投机解码等技术不改变输出分布
- 灵活部署：从边缘到云端的全覆盖方案

**局限**：
- 系统工程复杂度高：PagedAttention、连续批处理实现复杂
- 并行通信开销：张量并行的 All-Reduce 限制扩展性
- 投机解码加速比不稳定：依赖草稿模型质量
- Prefill-Decode 分离增加系统复杂度和 KV-Cache 传输开销

## 7. 应用场景

- **高并发在线服务**：vLLM + 连续批处理 + PagedAttention
- **低延迟实时推理**：TensorRT-LLM + FP8 + Speculative Decoding
- **超大模型部署**：张量并行 + 流水线并行 + Offloading
- **边缘/本地部署**：llama.cpp + CPU/GPU 混合推理
- **Agent 应用**：SGLang + RadixAttention + 结构化生成
- **批量离线处理**：最大化吞吐量，优化单位成本

- **Prefill-Decode 分离架构**：DistServe、Splitwise 等将两阶段解耦到不同集群
