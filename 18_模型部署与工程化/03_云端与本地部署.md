# 云端与本地部署

## 1. 概述

部署环境选型是模型部署的第一步，直接影响成本、延迟、隐私和运维复杂度。云端部署利用弹性资源应对流量波动，本地部署通过固定硬件投资换取数据主权和可控延迟。

部署环境选型解决的核心问题：
- **成本优化**：在性能需求和预算约束间找到最优平衡
- **数据主权**：满足数据不出域的合规要求
- **延迟控制**：对延迟敏感的场景需要本地或边缘部署
- **弹性扩展**：应对流量波动的弹性计算能力

生产环境通常采用混合策略：核心模型本地部署保障隐私和延迟，弹性容量由云端承接。

## 3. 核心概念

### 3.1 云端托管服务

云厂商提供全托管的模型部署服务，用户无需管理底层基础设施，按需付费。

### 3.2 本地推理引擎

部署在本地硬件上的推理软件，数据不出域，延迟可控。

### 3.3 混合部署

核心推理本地化保障隐私和低延迟，峰值流量溢出到云端弹性处理。

### 3.4 硬件加速器

| 硬件 | 典型产品 | 支持模型规模 | 特点 |
|:---|:---|:---|:---|
| GPU | A100/H100/L4 | 7B-405B | 通用并行计算，Tensor Core 加速 |
| TPU | TPU v4/v5 | 大规模分布式推理 | 专为矩阵运算优化 |
| NPU | 华为昇腾 910/310 | 7B-70B | 国产硬件，能效比高 |
| CPU | AMD EPYC / Intel Xeon | 1B-7B（量化后） | 通用性强，成本最低 |

## 4. 技术原理

### 4.1 云端平台对比

| 平台 | 特点 | 适用场景 |
|:---|:---|:---|
| **AWS SageMaker** | 托管部署，集成 S3/CloudWatch | 全托管 LLM 服务 |
| **Azure ML / OpenAI** | Azure OpenAI 托管 GPT 系列 | 企业级 GPT 部署 |
| **Google Vertex AI** | TPU 加速，AutoML 集成 | TPU 推理场景 |
| **阿里云 PAI** | 国内合规，预训练模型丰富 | 国内企业部署 |

### 4.2 本地推理引擎

| 引擎 | 核心特点 | 硬件支持 |
|:---|:---|:---|
| **llama.cpp** | 轻量 C++ 实现，GGUF 量化格式 | CPU/GPU/Metal |
| **Ollama** | 一键部署，CLI 友好 | CPU/GPU/Metal |
| **ONNX Runtime** | 跨平台，硬件后端丰富 | CPU/GPU/NPU |
| **mlx** | Apple Silicon 优化，统一内存 | M 系列芯片 |
| **TensorRT-LLM** | NVIDIA GPU 内核级优化 | NVIDIA GPU |

### 4.3 GGUF 量化格式

llama.cpp 定义的模型存储格式，支持 INT2-INT8 量化，通过 k-quants 方法减少精度损失。是本地部署大模型的主流格式。

### 4.4 并行部署策略

- **张量并行（TP）**：将模型单层内权重切分到多 GPU，需高带宽互联（NVLink）
- **流水线并行（PP）**：将模型不同层分配到不同 GPU，通信量小但存在气泡
- **数据并行（DP）**：多副本独立处理不同请求，通过负载均衡分发

### 4.5 弹性伸缩

基于 Kubernetes HPA 实现：监控 GPU 利用率或请求队列长度，触发 Pod 副本增减。冷启动优化通过模型预热和镜像缓存减少扩容延迟。

### 4.6 MIG（Multi-Instance GPU）

A100/H100 支持将单 GPU 切分为多个隔离实例，提高资源利用率，适合多租户场景。

## 5. 关键方法与模型

### 5.1 模型分片加载

大模型按层切分存储，按需加载到显存，支持超过单卡显存的模型推理。

### 5.2 混合部署架构

```
                    ┌─────────────┐
 用户请求 ─────────▶│  API 网关   │
                    └──────┬──────┘
                           │
                ┌──────────┴──────────┐
                ▼                      ▼
     ┌──────────────────┐   ┌──────────────────┐
     │  本地推理集群    │   │  云端弹性实例    │
     │  (核心模型)      │   │  (溢出流量)      │
     │  数据不出域      │   │  Spot 实例       │
     └──────────────────┘   └──────────────────┘
```

### 5.3 边缘推理优化

- INT8/INT4 量化减少模型大小
- 模型剪枝降低计算量
- 知识蒸馏训练小模型
- NNAPI（Android）/ CoreML（iOS）硬件加速

## 6. 优势与局限

### 6.1 云端部署优势

- **弹性伸缩**：按需扩缩容，应对流量波动
- **低初始投入**：按需付费，无需采购硬件
- **运维简化**：云厂商管理基础设施
- **全球部署**：利用云厂商的全球网络

### 6.2 本地部署优势

- **数据主权**：数据不出域，满足合规要求
- **可控延迟**：无网络往返，延迟稳定
- **长期成本**：高利用率场景下成本低于云端
- **定制化**：可深度优化硬件和软件栈

### 6.3 各自局限

| 维度 | 云端局限 | 本地局限 |
|:---|:---|:---|
| 成本 | 高利用率时成本高 | 初始投入大 |
| 弹性 | 峰值时可能受限 | 无法弹性扩容 |
| 运维 | 依赖云厂商 | 需专业运维团队 |
| 数据 | 数据传输合规风险 | 数据本地化要求 |

## 7. 应用场景

| 场景 | 推荐方案 | 原因 |
|:---|:---|:---|
| 公共 LLM API | 云端 + K8s | 弹性扩缩容，按需付费 |
| 金融/医疗私有化 | 本地 GPU + Docker | 数据敏感，合规要求 |
| 移动端/IoT | 边缘量化推理 | 离线运行，低延迟 |
| 混合部署 | 本地核心 + 云端弹性 | 平衡隐私和弹性 |
| 开发测试 | Ollama / 本地 CPU | 快速迭代，零成本 |
