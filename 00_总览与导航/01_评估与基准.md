# 评估与基准

## 1. 概述

评估与基准是对 AI 模型进行客观评测、对比和追踪的标准体系。它涵盖评估指标的定义、标准数据集的构建、评测方法论的设计以及性能基准的建立。没有可靠的评估体系，模型改进就无法被量化，研究进展也无法被客观衡量。

评估与基准解决的核心问题：
- **量化模型能力**：用标准化的指标衡量模型在不同任务上的表现
- **公平对比**：在相同数据集和评估协议下横向比较不同模型
- **追踪进展**：通过持续更新的排行榜追踪领域技术进步
- **发现短板**：通过细粒度评估发现模型的能力边界和失败模式
- **指导研发**：评估结果指导模型改进方向和资源分配

LLM 时代评估面临新挑战：模型能力从单一任务扩展到通用智能，开放式生成难以用固定标准评判，且存在数据污染（训练集包含测试集）问题。

## 3. 核心概念

### 3.1 评估方法论

| 方法 | 描述 | 适用场景 |
|:---|:---|:---|
| 固定测试集 | 在标准数据集上评估准确率 | 分类、检测等封闭任务 |
| 人类评估 | 人工评分模型输出质量 | 翻译、摘要、对话等开放任务 |
| 成对比较 | 两个模型输出两两对比，人类选择更好者 | LLM 对话质量评估 |
| Elo 评分 | 基于成对比较结果的竞技场排名 | Chatbot Arena |
| LLM-as-Judge | 用强模型（如 GPT-4）评分弱模型 | 大规模自动化评估 |
| 基准测试套件 | 多任务综合评估 | MMLU, BIG-bench |
| 在线 A/B 测试 | 真实用户环境中的对比 | 生产系统 |

### 3.2 评估指标体系

#### 分类指标

基于混淆矩阵：

| | 预测正 | 预测负 |
|:---|:---|:---|
| **实际正** | TP | FN |
| **实际负** | FP | TN |

- **准确率**（Accuracy）：$\frac{TP + TN}{TP + TN + FP + FN}$
- **精确率**（Precision）：$\frac{TP}{TP + FP}$
- **召回率**（Recall）：$\frac{TP}{TP + FN}$
- **F1 分数**：$\frac{2 \cdot P \cdot R}{P + R}$
- **AUC-ROC**：ROC 曲线下面积，衡量分类器区分能力

#### 生成指标

| 指标 | 公式/方法 | 适用场景 |
|:---|:---|:---|
| BLEU | n-gram 精确率的几何平均 | 机器翻译 |
| ROUGE | n-gram 召回率 | 文本摘要 |
| METEOR | 对齐 + 词序惩罚 | 翻译 |
| BERTScore | 嵌入空间相似度 | 生成质量 |
| Perplexity | $2^{-\frac{1}{N}\sum \log_2 p(x_i)}$ | 语言模型 |
| CER/WER | 编辑距离 / 参考长度 | 语音识别 |

#### LLM 评估指标

| 指标 | 评估维度 | 方法 |
|:---|:---|:---|
| MMLU | 多领域知识 | 多选题准确率 |
| GSM8K | 数学推理 | 答案正确率 |
| HumanEval | 代码生成 | Pass@k |
| TruthfulQA | 事实性 | 真实性评分 |
| MT-Bench | 多轮对话 | GPT-4 评分 |
| AlpacaEval | 指令遵循 | 胜率对比 |

### 3.3 数据污染问题

训练数据包含测试集导致评估结果虚高：

- **直接污染**：测试集原文出现在训练数据中
- **间接污染**：测试集的变形/翻译版本出现在训练数据中
- **数据泄露**：模型在测试集上进行过验证或超参搜索

**检测方法**：
- n-gram 重叠检测（检查测试集是否出现在训练语料中）
- 成员推理攻击（判断样本是否在训练集中）
- 时序分割（用训练截止日期之后的新数据测试）

## 5. 评估最佳实践

### 5.1 评估设计原则

1. **多维度评估**：不依赖单一指标，从准确率、鲁棒性、公平性、效率等多角度评估
2. **统计显著性**：报告置信区间和多次运行的标准差
3. **数据集分割**：严格区分训练/验证/测试集，避免信息泄露
4. **对抗样本**：在对抗扰动和分布偏移下评估鲁棒性
5. **错误分析**：不只是看总分数，分析失败案例的分布和模式
6. **人工验证**：自动指标无法完全替代人工评估

### 5.2 LLM 评估的特殊挑战

- **开放式生成**：没有唯一正确答案，需人类判断或 LLM-as-Judge
- **指令遵循**：模型是否真正理解并执行了指令
- **安全性**：是否产生有害、偏见或虚假内容
- **长文本能力**：长上下文理解和长文本生成质量
- **多轮对话**：上下文保持和对话连贯性
- **工具使用**：函数调用和 Agent 能力
- **数据污染**：测试集可能被包含在训练数据中

## 6. 优势与局限

### 6.1 评估体系的价值

- **客观追踪进展**：量化技术进步，识别突破性工作
- **公平竞争**：统一标准下的模型对比
- **指导研发**：评估短板指导改进方向
- **建立信任**：标准化评估增强模型部署的可信度

### 6.2 评估体系的局限

- **Goodhart 定律**：当指标成为目标时，它就不再是好指标——模型可能过拟合基准
- **数据污染**：训练数据包含测试集导致结果虚高
- **覆盖不全**：固定基准无法覆盖模型的所有能力维度
- **静态基准**：模型快速进步使基准迅速过时
- **文化偏见**：英文为主的基准对其他语言不公平
- **评估成本**：人工评估昂贵且难以规模化
- **开放式生成**：创造性任务难以量化评估

## 8. 前沿发展

- **动态基准**：定期更新测试集防止数据污染（如 LiveBench）
- **Agent 评估**：评估 LLM Agent 的工具使用、规划和多步推理能力
- **多模态评估**：统一评估视觉、语音、文本的多模态理解能力
- **人类对齐评估**：评估模型与人类价值观的对齐程度
- **效率评估**：在性能之外加入推理成本、能耗等效率维度
- **鲁棒性基准**：在分布偏移、对抗攻击下的性能评估
- **个性化评估**：针对不同用户群体和场景的定制化评估
- **元评估**：评估评估方法本身的有效性和偏差
- **自动化评估**：LLM-as-Judge 的可靠性和偏差研究
- **竞赛平台演化**：从 Kaggle 到 LLM 竞技场的评估范式转变
